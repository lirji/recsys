# 08 · 本地运行(整栈一键起 · 全链路容器化)

面向“把整套搜推广系统在本机跑起来看页面”的最短路径。**所有容器编排统一在 `docker/` 目录**(`docker/docker-compose.yml` 是单一入口);`scripts/dev-local.sh` 是它的一键封装,把「基础设施 + 8 个 Java 服务 + 前端」**全部以容器方式**拉起并做健康检查。

## TL;DR

```bash
# 1) 首次:如本机默认端口(网关 8080 / pg 5432 / redis 6379 / nacos 8848)被占,复制并按需改端口覆盖
cp scripts/dev-local.env.example scripts/dev-local.env   # 本仓库开发机已内置一份(见文末端口表)

# 2) 一键起全栈(容器化:基础设施 + 8 app + 前端 nginx),缺镜像自动构建 + 健康检查
scripts/dev-local.sh up
#    首次会构建 8 个 fat-jar 镜像,较慢(有 BuildKit 层缓存,后续快);Nacos 首启约 30–60s。

# 3) 打开前端(容器,同源经网关)
#    → http://localhost:${CONSOLE_WEB_PORT}   (本机默认 9095;用演示身份 admin 登录)

# 其它
scripts/dev-local.sh status              # compose ps + 网关健康
scripts/dev-local.sh logs rec-engine     # tail 某服务(compose 服务名)
scripts/dev-local.sh rebuild advertiser  # 改了后端代码 → 重建该服务镜像并重启
scripts/dev-local.sh restart             # 全栈重启
scripts/dev-local.sh down                # 停并删容器(保留数据卷 pgdata)
scripts/dev-local.sh up rec-engine gateway   # 只起指定服务
scripts/dev-local.sh infra               # 只起 pg/redis/nacos
scripts/dev-local.sh obs                 # 追加观测栈(prometheus/grafana/tempo/alertmanager)
scripts/dev-local.sh authz               # 追加判权栈(recsys 专属 SpiceDB;判权服务在宿主机另起,见 docs/09)
scripts/dev-local.sh frontend            # 前端热开发:宿主机跑 Vite(改前端即时热更)→ http://localhost:5173
```

> 用的是 **bash**(脚本 shebang 已指定)。

## docker/ 目录布局

```
docker/
  docker-compose.yml         # 单一编排入口(profiles: 默认=pg+redis / full / apps / console / obs / authz)
  docker-compose.local.yml   # 本机观测栈覆盖(gitignore)
  Dockerfile                 # 参数化多模块镜像(MODULE/PORT),8 个 Spring Boot app 共用;构建上下文=仓库根
  monitoring/                # prometheus / grafana / tempo / alertmanager 配置
```
> postgres 首启的建库脚本在 `recsys-offline/sql/`(整目录挂进 initdb);其中 `04b_ds1_bootstrap.sql`
> 自动建分库 ds_1(容器化必需)。不放 docker/ 是因为 Docker Desktop 不支持把单文件嵌套挂进已挂载目录。

手动用 docker compose(不经脚本)时,从仓库根用 `-f` 指向,或 `cd docker` 后直接跑:

```bash
cd docker && docker compose --profile apps --profile console up -d --build
# 或从仓库根:
docker compose -f docker/docker-compose.yml --profile apps --profile console up -d --build
```

## 服务与端口

`scripts/dev-local.sh` 默认走“标准端口”;本机冲突时用 `scripts/dev-local.env` 覆盖(gitignore)。

**关键:这些覆盖只影响「容器对宿主暴露的端口」。** 容器**之间**固定用「服务名 + 标准内部端口」通信(`postgres:5432` / `redis:6379` / `nacos:8848` / `gateway:8080` …),不受宿主端口重映射影响 —— 全容器化下后端 app 根本不需要从宿主访问 pg/redis。

| 服务(compose 名) | 模块 | 容器内端口 | 宿主暴露(标准 / 本开发机) |
|---|---|---|---|
| gateway | recsys-gateway | 8080 | **8080 / 9080**(前端入口) |
| rec-engine | recsys-rec-engine | 8081 | 不暴露(内部) |
| behavior | recsys-behavior | 8082 | 不暴露 |
| advertiser | recsys-advertiser | 8083 | 不暴露 |
| ad-serving | recsys-ad-serving | 8085 / gRPC 9095 | 不暴露 |
| content | recsys-content-service | 8086 / gRPC 9096 | 不暴露 |
| user | recsys-user-service | 8087 / gRPC 9097 | 不暴露 |
| console-api | recsys-console | 8090 | 不暴露 |
| console(前端 nginx) | console/ | 80 | **9095** |
| postgres | pgvector/pg16 | 5432 | 5432 / 5456 |
| redis | redis:7 | 6379 | 6379 / 6381 |
| nacos | nacos v2.4.3 | 8848 / 9848 | 8848 / 8858(+9858) |
| spicedb(可选,authz) | authzed/spicedb | 8443 / gRPC 50051 | 8544 / 50052 |

请求链路:浏览器 → 前端 nginx(:9095,同源)→(`/api` 反代)→ 容器网关 gateway:8080 → 各服务(容器 DNS + Nacos lb://)。仅网关(与前端 nginx、基础设施)对宿主暴露端口。

## 曾经的坑,现已在 docker/docker-compose.yml 固化(无需手敲)

1. **网关/各服务 `RECSYS_SECURITY_ENABLED=false`**:否则 `/api/advertiser`、`/api/search-ads` 等经网关 401。compose 里各 app 已注入(可 `RECSYS_SECURITY_ENABLED=true` 覆盖,生产用)。
2. **rec-engine / advertiser / ad-serving 的 `PG_JDBC_URL` + `PG_DS1_JDBC_URL`**:它们经 ShardingSphere(`sharding.yaml`,整条 URL 单占位符)连主/次数据源。compose 里已指向 `postgres:5432` 的 `recsys` / `recsys_ds1`。
3. **分库 ds_1 自动建库**:`sql/04_ad_sharding.sql` 里的 `CREATE DATABASE recsys_ds1` 是手动步骤;容器化由 `recsys-offline/sql/04b_ds1_bootstrap.sql`(随 schema 一起挂进 initdb)在 postgres 首启自动建库 + 分片骨架,否则上面三个服务的 ds_1 连接池初始化失败、启动即挂。
4. **启动顺序**:compose `depends_on` + healthcheck 保证 pg/redis/Nacos 就绪后再起 app(取代旧脚本的手工 staggered)，避免 Nacos 首启较慢时应用注册失败退出。

## 验证

```bash
scripts/dev-local.sh status                            # 期望容器全 Up、gateway 健康 UP
GW=9080   # = 本机 GATEWAY_PORT
curl "localhost:$GW/api/recommend?userId=1&size=3&explain=true"  # 含 explain(逐阶段+每路召回+打分分解)
curl "localhost:$GW/api/console/items?ids=1,2,3"        # 真实标题
curl "localhost:$GW/api/search-ads?userId=1&q=action&size=3"    # 广告(需先跑离线 seed-ads 灌数据)
```

> 容器 postgres 是**全新库**(schema 由 `sql/` 首启自动建,但没有 item/behavior/广告数据)。要看到推荐/广告结果,需跑离线作业向它灌数据:把离线作业的 `PG_PORT` 指向容器 postgres 的宿主暴露端口(本机 5456),跑 `import-items`/`import-behavior`/`seed-ads` 等(见 CLAUDE.md「离线任务」)。空库时在线链路优雅降级(hot/空召回),页面能开但结果稀疏。

页面:`http://localhost:9095`(容器前端)或 `http://localhost:5173`(`frontend` 热开发)登录后,在线调试台、广告主后台、用户360/诊断/告警、离线报表全部可用。

## 常见问题

- **`docker compose up` 端口占用报错**:某个宿主暴露端口被占 → 在 `scripts/dev-local.env` 改对应 `*_PORT` 后 `scripts/dev-local.sh restart`。容器**内部**端口无需改。
- **改了后端代码没生效**:全容器化下改代码需**重建镜像** → `scripts/dev-local.sh rebuild <svc>`(如 `rec-engine`)。想要即改即跑的后端热开发,用下方「宿主机 mvn 模式」。
- **首次 up 很慢**:在构建 8 个 fat-jar 镜像(BuildKit 并行 + 层缓存);后续无代码变更时 `up` 直接复用镜像、很快。
- **某服务不健康**:`scripts/dev-local.sh logs <svc>` 看日志;`rebuild <svc>` 重建。Docker Desktop 资源不足会整体变慢/退出 → 给它多分配 CPU/内存。
- **前端 500 / 401**:多半后端没起齐或网关安全没关。确认 `status` 全 Up、`RECSYS_SECURITY_ENABLED=false`。

## 附:开启判权 / 统一登录(可选,默认全关;详见 docs/09)

compose 已内置三套开关(默认值零行为变化),对应变量可写进 `scripts/dev-local.env`(脚本 `set -a` 自动 export 给 compose):

```bash
# 1) 细粒度归属判权(advertiser 接 auth-platform/SpiceDB)
scripts/dev-local.sh authz          # 起 recsys 专属 SpiceDB(:8544;先 migrate 再 serve,postgres datastore 持久化)
#   宿主机另起判权服务 auth-platform-server(:8210,从 auth-platform 仓库构建,见 docs/09 §3)
#   dev-local.env 里设 RECSYS_AUTHZ_MODE=shadow(只记 would-deny)或 enforce(真拦 403)
scripts/dev-local.sh rebuild advertiser   # advertiser 容器经 host.docker.internal:8210 访问判权服务

# 2) Casdoor 统一登录(成对开关:网关 + 前端,缺一不可)
#   dev-local.env: RECSYS_EDGE_CASDOOR=true(网关 RS256 JWKS 边缘认证)
#                  CONSOLE_AUTH_MODE=oidc(前端构建期烘焙,登录页切统一登录按钮)
#   Casdoor 跑在宿主机(:8000);issuer 须浏览器可达(勿用容器名)
scripts/dev-local.sh rebuild gateway console
```

默认(全关)= 演示自签 JWT 登录 + 无归属判权,与本文档其余部分行为一致。

## 附:宿主机 mvn 模式(后端热开发)

全容器化适合“一键看整套”;若要**后端即改即跑**(不必每次重建镜像),可只容器化基础设施、后端用 mvn 跑在宿主机:

```bash
scripts/dev-local.sh infra     # 只起 pg/redis/nacos(容器)
# 然后针对要热改的服务,用 mvn 跑在宿主机,连容器基础设施的宿主暴露端口(本机 pg 5456 / redis 6381):
PG_HOST=127.0.0.1 PG_PORT=5456 PG_USER=recsys PG_PASSWORD=recsys \
REDIS_HOST=127.0.0.1 REDIS_PORT=6381 RECSYS_SECURITY_ENABLED=false \
NACOS_DISCOVERY=false SERVER_PORT=9081 \
mvn -pl recsys-rec-engine spring-boot:run
```

要点:分库服务(rec-engine/advertiser/ad-serving)还要带 `PG_JDBC_URL=jdbc:postgresql://127.0.0.1:5456/recsys` + `PG_DS1_JDBC_URL=...recsys_ds1`;`NACOS_DISCOVERY=false` 时网关加 `--spring.profiles.active=static` 走静态路由。这是老一套手工 env,容器化编排已把它们固化 —— 除非需要热改后端,否则直接用一键容器化即可。
