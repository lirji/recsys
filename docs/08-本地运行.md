# 08 · 本地运行(整栈一键起)

面向“把整套搜推广系统在本机跑起来看页面”的最短路径。核心是 `scripts/dev-local.sh` —— 它把一堆容易踩的 env / 端口 / 启动顺序坑固化成一条命令。

## TL;DR

```bash
# 1) 首次:如本机默认端口(8080-8090 / pg 5432 / redis 6379)被占,复制并按需改端口覆盖
cp scripts/dev-local.env.example scripts/dev-local.env   # 本仓库开发机已内置一份

# 2) 起后端全栈(基础设施检查 → 8 个 app,逐个 staggered + 健康检查)
scripts/dev-local.sh up

# 3) 起前端(Vite,自动指向本机网关端口)
scripts/dev-local.sh frontend
# 打开 http://localhost:5173,用演示身份(admin)登录

# 其它
scripts/dev-local.sh status         # 各服务健康表
scripts/dev-local.sh logs rec-engine
scripts/dev-local.sh restart console
scripts/dev-local.sh down            # 全停
scripts/dev-local.sh up console gateway   # 只起指定服务
```

> 用的是 **bash**(脚本 shebang 已指定);别用 zsh 直接 `source`。

## 服务与端口

`scripts/dev-local.sh` 默认走“标准端口”;本机冲突时用 `scripts/dev-local.env` 覆盖(已 gitignore)。本开发机的映射:

| 服务 | 模块 | HTTP | gRPC | 标准端口 |
|---|---|---|---|---|
| 网关(前端入口) | recsys-gateway | 9080 | — | 8080 |
| 推荐编排 | recsys-rec-engine | 9081 | — | 8081 |
| 行为采集 | recsys-behavior | 9082 | — | 8082 |
| 广告主后台 | recsys-advertiser | 9083 | — | 8083 |
| 广告投放(gRPC) | recsys-ad-serving | 9085 | 9095 | 8085 |
| 内容服务(gRPC) | recsys-content-service | 9086 | 9096 | 8086 |
| 用户服务(gRPC) | recsys-user-service | 9087 | 9097 | 8087 |
| 控制台 BFF | recsys-console | 9090 | — | 8090 |
| 前端 Vite(dev) | console/ | 5173 | — | 5173 |

基础设施:postgres `recsys-pg`(本机 **5455**,库 `recsys` + 分库 `recsys_ds1`)、redis `recsys-redis`(本机 **6380**)。`up`/`infra` 会 `docker start` 兜底拉起这两个容器;redis 缺失会用持久 volume 新建。

请求链路:浏览器 → Vite:5173 →(`/api` 反代)→ 网关:9080 → 各服务。**全站同源经网关**,前端只用相对 `/api`。

## 为什么需要这个脚本(四个每次都踩的坑,已内建规避)

1. **必须 bash,不能 zsh**:zsh 不对无引号变量分词,`env $VARSTR mvn` 会把整串 env 塞进第一个变量的值(典型报错 `UnknownHostException: 127.0.0.1 PG_PORT=...`)。脚本用 bash 数组传 env。
2. **网关也要 `RECSYS_SECURITY_ENABLED=false`**:否则 `/api/advertiser`、`/api/search-ads`、`/api/ad`、`/api/experiment`、`/api/user` 经网关 **401**(直连服务是 200,是网关这层拦的)。
3. **rec-engine / advertiser / ad-serving 都要 `PG_JDBC_URL` + `PG_DS1_JDBC_URL`**:它们的广告读写侧走 ShardingSphere(`sharding.yaml`,整条 URL 用单占位符 `$${PG_JDBC_URL::...}`)。不给就默认连 5432 → 健康聚合 **DOWN**、搜索广告有广告但**标题为空**。脚本对所有服务统一注入这两条(其余服务无害忽略)。
4. **staggered 启动**:同时初始化多个数据源会打爆 pg 连接池 / docker 端口转发风暴(Hikari 池被“毒化”),脚本逐个等就绪再起下一个。

其它已固化的 env:`RECSYS_SECURITY_ENABLED=false`(本地免登录直连)、`NACOS_DISCOVERY=false` + `SPRING_CLOUD_NACOS_CONFIG_ENABLED=false`(不连外部 Nacos,否则健康被拖 DOWN)、console 的 `RECSYS_*_URL` 健康探针指向重映射端口、`PROMETHEUS_URL=http://localhost:19090`(本机 Prometheus 挪了口)。

## 验证

```bash
scripts/dev-local.sh status                         # 期望 8 个全 UP
curl 'localhost:9080/api/recommend?userId=1&size=3&explain=true'  # 含 explain(真实逐阶段+每路召回+打分分解)
curl 'localhost:9080/api/console/items?ids=1,2,3'    # 真实标题
curl 'localhost:9080/api/console/user/1'             # 用户360(DB 聚合 + Redis 实时态)
curl 'localhost:9080/api/console/diagnosis'          # 一键诊断(overall + 逐项)
curl 'localhost:9080/api/search-ads?userId=1&q=action&size=3'   # 广告带真实标题
```

页面:`http://localhost:5173` 登录后,在线调试台(推荐/搜索/搜索广告/混排Feed/Query/实验/冷启动)、广告主后台、用户360/一键诊断/告警、离线报表全部可用。

## 常见问题

- **页面全 500 / 401**:多半是①后端没起,或②Vite 代理指到了别的进程。用 `scripts/dev-local.sh frontend` 起前端(它会把 `/api` 指到本机网关端口),别用裸 `npm run dev`(默认代理 8080,本机被占)。
- **某服务 DOWN**:`scripts/dev-local.sh logs <svc>` 看日志;`restart <svc>` 重启单个。
- **pg `too many clients`**:`recsys-pg` 的 `max_connections` 已调到 500;若仍不够,`docker exec recsys-pg` 改 `postgresql.conf` 后 `docker restart recsys-pg`。
- **重打了模型/资源**:改了 `recsys-rank`/`recsys-recall` 的 ONNX 要 `mvn -pl <module> clean install` 重打进 jar,再 `restart` 对应服务。

## 与 docker compose 的关系

`docker-compose.yml` 起基础设施(pg+redis)与可选组件(kafka/nacos/obs);`docker-compose.local.yml` 是本机 obs 栈端口调整版。app 本身本地用 `mvn spring-boot:run`(经本脚本),便于热改热跑;容器化部署见 `Dockerfile` 与 `deploy/k8s/`。
