## 1. 这是什么

一个**面向生产的搜推广一体化脚手架**：Java 21 + Spring Boot 3.2，Maven 多模块单仓。核心是经典的 **召回 → 排序 → 重排** 漏斗，离线/近线链路（embedding、协同过滤、模型训练）向在线存储（pgvector、Redis）灌数据；在此之上叠加 **Query 理解 / 搜索** 与 **搜索广告（竞价 + 计费 + 智能出价 + 分库分表）** 两条扩展链路。

设计为 `docker compose` 一键本地跑通，但接口、可观测、样本回流、A/B 实验、分库分表都按「将来上生产」的标准预留。**单体优先、微服务就绪**：`rec-engine` 用 `scanBasePackages="com.recsys"` 把所有 `[lib]` 模块装配进一个进程；模块边界已划好，跨模块耦合限制在 `recsys-common` 契约上，后续拆微服务成本很低。

一句话定位：**搜（query 理解 + 混合检索）、推（12 路召回 + 9 排序策略 + 4 重排 + 分层 A/B）、广（召回→相关性→竞价→GSP/VCG 计费→oCPC→混排，全链路 ShardingSphere 分库分表）三位一体的可服务学习/脚手架系统。** 已从"单体优先"演进出可回滚的 **DDD 微服务拆分**（ad-serving / content / user gRPC）+ **两级零信任安全** + **DB-per-service**。

> **技术深度拆解见 [`docs/skills/`](skills/README.md)**：按主题(召回/排序/Embedding/特征一致性/融合/广告竞价/微服务…)逐篇讲清「原理 → 本项目实现 → 在线离线契约 → 踩坑 → 面试要点」,共 19 篇 + 索引。

### 三块的关系

```
                   ┌─────────────── 用户请求 ───────────────┐
                   │                                        │
             无 query（推荐）                          有 query（搜索 / 搜索广告）
                   │                                        │
                   ▼                                        ▼
          /api/recommend                    recsys-query 理解(归一/意图/纠错/向量化)
          /api/feed(混排)                                   │
                   │                          ┌─────────────┴─────────────┐
                   ▼                          ▼                           ▼
        推荐漏斗:召回→排序→重排        /api/search 自然搜索         /api/search-ads 广告
                   │                (召回权重↑ + RRF 混合检索)   (广告召回→相关性门槛→
                   │                          │                  复用排序 pCTR/pCVR→校准→
                   └────────────┬─────────────┘                  oCPC 出价→eCPM 竞价→GSP)
                                ▼                                          │
                       同一套 recall / rank / feature / embedding ────────┘
                       （广告只净增「query / 钱 / 校准 / 拍卖 / 出价」几块）
```

**关键设计思想**：广告 pCTR/pCVR **复用推荐的排序模型**，搜索/广告与推荐**共用** recall/rank/feature/embedding；广告链路只补相对推荐的「净增量」。三块共享同一数据底座与在线/离线契约。

## 3. 整体架构

### 3.1 在线 / 离线双链路

```
        搜推广控制台 (console/ React SPA)  ──静态+/api 反代──┐
              用户 / 前端                                    │ HTTP
              API Gateway (recsys-gateway :8080, 边缘认证(自签 JWT 默认/Casdoor OIDC 可选) + 限流)
                        │
   ┌────────────────────┴──────────────────────────────────────────┐
   │            在线链路（低延迟、同步：只做「查表 + 轻打分」）          │
   │                                                                │
   │  rec-engine :8081 ──┬── recall(多路) ── pgvector / Redis        │
   │  (编排/兜底/混排)    ├── rank(特征+ONNX/规则)                    │
   │                     ├── rerank(多样性/MMR)                      │
   │                     ├── query 理解(搜索/广告入口)                │
   │                     └── ad(召回→相关性→竞价→GSP→oCPC→混排)       │
   │  advertiser :8083 ── 广告主写侧/管理面(CRUD + 报表 + 派生存储同步)│
   │  behavior :8082 ──── 行为接入 → Kafka / DB                      │
   └────────────────────┬──────────────────────────────────────────┘
                        │  提前把重活算好，把结果「喂」进在线存储
   ┌────────────────────┴──────────────────────────────────────────┐
   │      离线 / 近线链路（异步、不影响在线延迟）                       │
   │                                                                │
   │  recsys-offline(25 个 Job):导入 → embedding → CF/Swing/Hot     │
   │      → 样本生成 → 训练(train/*.py→ONNX) → 灌 pgvector/Redis      │
   │      → 评估(eval/ab-report) → 广告(seed/校准/oCPC/质量度/报表)   │
   │  recsys-streaming(Flink):Kafka 行为流 → 实时热门 + 实时类目偏好  │
   └────────────────────────────────────────────────────────────────┘
```

**核心区别**：在线链路毫秒级、只「查 + 轻打分」；离线/近线预计算 embedding、CF 倒排、训练模型，把结果写入在线存储。**在线与离线的特征计算必须一致**（同名、同逻辑）——特征不一致 / 数据泄漏是部署后推荐质量崩溃的首因。每一层都可优雅降级（hot 兜底、embedding 回退本地 BGE、排序 ONNX 失败回退规则）。

### 3.2 一次推荐请求时序（`/api/recommend`）

1. 查 `cache:rec:{userId}` → 命中直接返回
2. 冷启动检测 + 分层 A/B 分桶（recall × rank × rerank 三层，确定性分桶）
3. **召回**：并行多路（按实验/冷启动 `enabledChannels` 启停）→ **各路先按路归一化到 [0,1] 再跨路取 max**（否则热度计数压垮语义/向量余弦分）→ 合并去重
4. **排序**：批量取特征 → 按 rank 变体选 v1/onnx/deepfm/mmoe/din 打分
5. **融合**：`(recall-weight·归一化召回分 + rank-weight·排序分) · channel-boost`（TAG 默认 ×1.5，抬升兴趣信号不被热度淹没）
6. **重排**：按 rerank 变体选 diversity/mmr/none（冷启动强制强多样性）
7. 截断 top-N → 构建推荐理由 → `ExposureLogger` 带 bucket 异步写曝光 → 写缓存返回
8. 用户点击 → `POST /api/behavior` → 反馈闭环（服务端按 `expo:{user}:{item}` 回填 bucket 归因）

### 3.3 一次搜索广告请求时序（`/api/search-ads`）

`query 理解 → 广告召回（关键词倒排/语义/U2A 用户向量/兜底）→ 相关性门槛 → 预算/pacing 过滤 → 复用排序模型出 pCTR/pCVR → 保序回归校准 → oCPC 自动出价 → eCPM 竞价排序 → GSP 次价计费 →（可选 List-wise 外部性 / VCG 位置拍卖）→ DCO 创意择优 → 曝光埋点 ad_event`。详见 [05 §5](05-搜索广告扩展设计.md)。

## 5. 能力集

### 5.1 召回能力（12 通道，`recsys-recall` + `recsys-ad`）

各路先**按路归一化再跨路取 max**（搜索场景改 **RRF** 名次融合）；每路优雅降级（缺数据返回空，其他路兜底）。

| 通道 | 类型 | 原理 |
|---|---|---|
| `VECTOR` | 内容向量 | `user_embedding` → pgvector 余弦最近邻 |
| `I2I` | 协同 | 用户近期物品 → `i2i:{itemId}`（ItemCF 共现） |
| `HOT` | 兜底 | `recall:rt_hot`（实时）→ `recall:hot`（离线）；始终在线兜底 |
| `TAG` | 画像 | 用户偏好类目 + **实时类目偏好** `rt:user:{id}` 融合加权 |
| `U2U` | 协同 | 相似用户正反馈（离线物化 `u2u:{userId}`） |
| `SWING` | 协同 | Swing i2i（抗热门的物品相似 `swing:{itemId}`） |
| `SEMANTIC` | 语义 | query/伪 query → embedding → pgvector；搜索场景与 LEXICAL RRF 混合 |
| `COLD` | 冷启动 | 跨类目铺开，最大化覆盖 |
| `TWO_TOWER` | 学习型 | DSSM 双塔：item 塔离线烘焙进 `item_tower_embedding`，user 塔在线 ONNX 算 → ANN |
| `GENERATIVE` | 生成式 | RQ-VAE 语义 ID：离线量化成 `(c0,c1,c2)`，在线按用户近期 item 的语义 ID **前缀**检索同簇 |
| `LEXICAL` | 词法 | Postgres 全文检索 `ts_rank_cd`（BM25）over `item.title_tsv` |
| `TIGER` | 生成式 | **完整 TIGER**：decoder-only Transformer 自回归 beam search 生成下一个 item 的语义 ID → 映射回 item |

> 广告召回（`recsys-ad`）另有：关键词倒排 `bidword:inv:{keyword}`、SEMANTIC_AD（query↔ad 向量）、**U2A**（`user_embedding`→`ad_embedding` 余弦，query 无关的个性化定向）、兜底多路。

### 5.2 排序能力（9 策略 + 双塔粗排，`recsys-rank`，`RANK_STRATEGY` 切换）

| 策略 | 模型 | 输入 / 特点 |
|---|---|---|
| `v1` | 规则加权 | 纯 Java 兜底，零模型 |
| `onnx` | LightGBM | 单输入 `dense[N,5]`（FeatureAssembler 5 特征） |
| `deepfm` | PyTorch DeepFM | 双输入 `dense[N,5\|8]` + `sparse[N,3\|6]`（userId/itemId/category，可选 +语义 ID） |
| `dcn` | DCN v2 | 与 DeepFM 同契约；Cross Network 显式可控阶数高阶交叉 ∥ Deep |
| `mmoe` | MMoE 多任务 + ESMM | 双输出 `ctr`+`cvr`；ESMM 全空间建模消除 CVR 样本选择偏差 |
| `ple` | PLE/CGC 多任务 + ESMM | 与 MMoE 同契约；共享+专属专家,缓解跷跷板负迁移 |
| `din` | DIN 序列 + MMoE head | 四输入（+ 行为序列 target-attention）；双输出 ctr/cvr |
| `dien` | DIEN 兴趣演化 | 与 DIN 同契约；GRU 兴趣抽取 + AUGRU 兴趣演化 |
| `sim` | SIM 长序列 | GSU 类目硬检索(500 长历史) + ESU 注意力；逐候选子序列 |
| 粗排 | 双塔粗排(R6) | `PreRankService` 精排前截到 50，`mode=two-tower` 叠双塔学习分 |

**共性**：模型缺失/失败一律回退规则打分；在线/离线契约（`FeatureAssembler` 稠密 5/8、`SparseFeatureEncoder` 稀疏 3/6、`SequenceEncoder` 序列）必须与训练器一致。
**位置偏差去偏（PAL）**：mmoe/ple/din 的 CTR head 叠加位置偏置塔 `b(position)`，导出 ONNX 时该塔不入图 → 线上零改动；真实位次走曝光日志闭环（`gen-samples-impr`）。
> 详见 [skills/02 排序模型全家桶](skills/02-排序-CTR与多任务模型全家桶.md)。

### 5.3 重排能力（`recsys-rec-engine` 内 `rerank/`，可插拔）

`RerankRouter` 按 rerank 变体路由到 `diversity`（类目打散）/ `mmr`（相关性-多样性平衡）/ `dpp`（行列式点过程,Chen 2018 fast-greedy MAP）/ `none`（NoopReranker）；冷启动强制强多样性。重排前有 `SeenFilter` 过滤已看。详见 [skills/05 重排](skills/05-重排-多样性与DPP.md)。

### 5.4 实验与冷启动

- **分层 A/B**：recall × rank × rerank 三层独立确定性分桶，曝光记 `user_behavior.bucket`；另有 `ad` 层用于**广告分层 A/B**（`ad_event.ad_bucket`，变体可覆盖 reserve-price 等）。
- **冷启动**：检测器 + 兴趣引导（`GET/POST /api/user/{id}/interests`）+ 强多样性重排。
- **已看过滤**：过滤用户已曝光/交互物品。

### 5.5 搜索广告能力（`recsys-ad` + `recsys-advertiser`，M4–M8）

| 环节 | 组件 | 说明 |
|---|---|---|
| 广告召回 | `AdRecallService` | 关键词倒排 / 语义 / U2A 用户向量定向 / 兜底 |
| 相关性门槛 | `RelevanceGate` | query↔ad 相关性过滤（守广告体验红线） |
| pCTR/pCVR | 复用 `recsys-rank` | 广告不训练独立模型，复用推荐排序模型 |
| 校准 | `IsotonicCalibrator` | pCTR 保序回归（PAVA）校准，读 `ad:calib:{model}` |
| 智能出价 | `OcpcBidder` | oCPC：`bid=targetCpa·pCVR·k`，k 读 `ad:ocpc:{adv}`（反馈控制 + 延迟纠偏） |
| 竞价排序 | `BiddingService` | eCPM 排序 + **GSP 次价计费**；可选 List-wise 外部性 / 完整 **VCG 位置拍卖** |
| 博弈分析 | `AuctionGame` | 离线诊断激励相容（VCG 真实 / GSP 纳什均衡 / 合谋套利检测） |
| 质量度 | `QualityScoreService` | 相关性/CTR/CVR 三因子贝叶斯收缩融合，替代随机基线 |
| DCO | `CreativeSelector`+`CreativeBandit` | 竞价后 UCB 多臂老虎机按创意级 CTR 择优展示 |
| 预算/pacing | `PacingService` | 实时预算熔断 + pacing 平滑（Redis `ad:budget:{adv}:{date}`） |
| 探索 EE | `ExplorationService` | 新广告 UCB 加成抬曝光（只进排序不进计费，守红线） |
| 反作弊 | `AntiFraudService` | 点击去重/频次/归因校验，无效点击落 `INVALID_CLICK` 不计费 |
| 混排 Ad Load | `AdMixer`+`FrequencyCapService` | 广告按位次/密度混入自然结果 + 频控（`/api/feed`） |
| 延迟反馈 | `ad-delay` 作业 | 转化延迟指数分布 MLE，Horvitz–Thompson 加权补回在途转化 |
| 广告主管理 | `recsys-advertiser` | 写侧 CRUD + 报表 + `AdIndexSyncService` 增量维护倒排/向量/预算 |

**分库分表（ShardingSphere-JDBC，全链路已接入）**：`advertiser`/`ad`/`bidword`/`ad_creative` 按 key 取模分到 `ds_0`(recsys)/`ds_1`(recsys_ds1)；Snowflake 主键；`ad`（按 `advertiser_id` 而非 Snowflake 低位）；`ad_embedding` **刻意不分片**（向量 ANN 须看全量）。写侧（advertiser）+ 在线读侧（rec-engine/ad 双数据源）+ 离线侧（offline 双数据源）均已实测。详见 [05 §8](05-搜索广告扩展设计.md)。

### 5.6 实时特征（`recsys-streaming`，Flink MiniCluster）

消费 Kafka `behavior-events` → 实时热门 ZSet `recall:rt_hot`（`HotRecaller` 优先读，回落离线 `recall:hot`）+ 每用户实时类目偏好 `rt:user:{id}`（`TagRecaller` 融合进 TAG 通道）。

### 5.7 评估与观测

- **离线评估**（`eval` 作业）：按时间切分留出正样本 → 复用在线 recall→rank 流水线 → Precision/Recall/NDCG/MAP/MRR/HitRate/Coverage/Diversity/Novelty @K；`--recall-only` 逐通道；`run_eval_compare.sh` 同 ground-truth 比 5 策略。
- **严格 eval**（消除乐观偏置）：`--max-ts` 只用切分点前行为重建存储，`run_strict_eval.sh` 一键完成（实测偏置仅 ±1~2%）。
- **在线观测**（Prometheus + Grafana，profile=obs）：`recsys_recommend_duration_seconds`（P50/P95/P99）、缓存命中率、分桶曝光/点击 → 在线 CTR、模型回退率。
- **广告报表**：`ad-report` 按位次聚合曝光/点击/转化/eCPM/CTR/CVR；`ab-report` 逐桶 CTR。

## 8. 离线作业（`recsys-offline`，`--job=<name>` 分发，共 40+ 个）

`JobRunner`(ApplicationRunner)按 `name()` 注入 `Map<String,OfflineJob>`,`--job=<name>` 分发(可多个);无 `--job` 打印可用集。**引导初始化**（大致按序）：`import-items` → `import-behavior` → `backfill-embedding` → `user-embedding` → `item-cf`/`user-cf`/`swing`/`hot`/`idf`/`cold-bandit-stats` → `build-features` → `gen-samples`/`gen-samples-mt`/`gen-samples-impr` → `import-tower`/`import-semantic-id` → `sim-rec-events`(教学)。

| 类别 | 作业 |
|---|---|
| 数据导入 | `import-items`、`import-behavior` |
| 向量 / CF | `backfill-embedding`、`backfill-multimodal`、`user-embedding`、`item-cf`、`user-cf`、`swing`、`hot`、`idf`、`cold-bandit-stats` |
| 特征 / 样本 | `build-features`、`gen-samples`、`gen-samples-mt`（多任务+序列）、`gen-samples-impr`（曝光日志闭环，真负样本+真位次）、`gen-ad-cvr-samples`（延迟反馈 CVR 样本） |
| 学习型召回灌数 | `import-tower`（双塔）、`import-semantic-id`（RQ-VAE 语义 ID） |
| 近线学习 | `bandit-stats`（LinUCB 充分统计）、`train-ftrl`（FTRL-Proximal 增量）、`rec-calibrate`（保序回归校准）、`tune-fusion`（多任务权重网格） |
| 评估 / 质量 | `eval`（推荐质量 @K）、`ab-report`（逐桶 CTR + 显著性）、`data-quality`（覆盖率/ECE/PSI 漂移）、`publish-reports`（灌 `eval_report` 供 console） |
| 调度 | `run-dag`（声明式依赖拓扑串行 + Redis 锁 + 断环即止） |
| DB-per-service 同步 | `sync-behavior-log`、`sync-ad-event-log`、`sync-item-catalog`（CDC 替身,watermark 增量） |
| 教学模拟 | `sim-rec-events`（带位置偏置曝光/点击）、`sim-ad-events`（广告曝光/点击，含延迟反馈） |
| 广告 | `seed-ads`、`ad-calibrate`、`ad-delay`、`ad-ocpc`、`ad-explore-stats`、`ad-quality`、`ad-report`、`ad-attribution`（MTA）、`lookalike`（Look-alike 人群包） |
