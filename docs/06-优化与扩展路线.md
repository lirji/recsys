## 0. 一句话诊断

**能力密度极高(搜推广全链路:12 召回 + 5 排序 + 广告竞价计费 + 分库分表)、但工程护栏偏薄。** 真正的瓶颈不是模型不够,而是:
1. **召回质量被 HNSW 参数静默腰斩**(`ef_search=40 < LIMIT 200`);
2. **在线无隔离 / 超时 / 熔断**(多路召回串行、外部调用无超时);
3. **排序特征太薄(仅 5 维)+ 无粗排层**——所有模型的共同天花板;
4. **核心链路零测试**,尤其在线/离线特征一致性契约(项目自认"部署即崩溃"首因)。

推进节奏:**先做第一批 5 个快赢 → 再补粗排 + 弹性 + 特征 → 后上测试/CI/去偏**。

### 落地备注(第一批)

- **Q1 对已有库需手动重建索引**:`CREATE INDEX IF NOT EXISTS` 不会改动既存索引;老 `pgdata` 卷需 `DROP INDEX idx_item_embedding_hnsw;` 后重跑 DDL 才吃到新 `m/ef_construction`。而 `SET hnsw.ef_search=200`(rec-engine Hikari `connection-init-sql`)对新旧库**立即生效**,是召回质量的关键项。
- **Q2 可回退**:`recsys.recall.parallel.enabled=false` 退回串行;`timeout-ms`(默认 150)、`pool-size`(默认 12)可调。合并结果顺序与串行完全一致(确定性)。
- **Q3 契约兼容**:`FeatureService.itemFeatures(Collection)` 是 default 方法(旧实现自动逐个兜底),仅 `RedisFeatureService` 覆盖为 pipeline;pipeline 异常自动回退逐个读。
- **Q5 pool**:顺带给 rec-engine 显式配 Hikari(`max=20/min=5/connectionTimeout=3s`),消除默认 10 连接在多路并发下的连接饥饿。

## 3. 第三批 · 扩展新能力(算法 + 工程,中长期)

**算法**(✅ 已落地 4 项)
- ✅ **流行度去偏**:融合分 × `1/(1+item_pop_norm)^β`(`recsys.fusion.pop-debias`),从机制上替代 channel-boost 补丁。
- ✅ **`user_embedding` 时间衰减**:`UserEmbeddingJob` 正反馈向量按半衰期 `0.5^(age/halfLife)` 加权(`--half-life-days`,默认180),让用户向量偏向近期兴趣,修 VECTOR 召回"纯长期、无新鲜度"。
- ✅ **推荐精排分数校准**:离线 `rec-calibrate` 重跑 recall→rank 于时间切分留出集,拟合 rankScore→真实正反馈率的保序回归 → `rec:calib:{model}`;在线 `RecScoreCalibrator` 在融合前把 rank 分校准成可比概率(`recsys.fusion.calibration`),让 recall+rank 融合量纲一致(isotonic 单调,不改单策略内排序)。
- ✅ **多目标融合权重自动搜参**:离线 `tune-fusion` 重跑一次 recall→rank(mmoe/din 已把 pCTR/pCVR 写进快照),再对权重网格做纯内存重打分 + NDCG@K,搜出最优 `cvrBias/cvrWeight` 并写 CSV(把拍脑袋改成数据驱动;`RANK_STRATEGY=mmoe` 启动)。
- ✅ **PLE 替 MMoE**:`train_ple.py`(单层 CGC:共享专家 + 每任务专属专家 + 定制门控治多任务跷跷板,与 MMoE 同 ESMM/PAL/ONNX 契约)→ `model_ple.onnx`;在线 `PleRankService`(`strategy=ple`,serving 与 MMoE 同形)+ `RankRouter` ple 路由,可 `--rank-strategy=ple` 与 MMoE 对比。
- ✅ **冷启动 contextual bandit**:COLD 召回从"跨类目均匀铺开"升级为类目 **UCB** 驱动(`ColdStartBandit`:经验正反馈率 exploit + 欠曝光 explore,与广告 `ExplorationService` 同 UCB 家族);统计由离线 `cold-bandit-stats` 物化 `cold:cat:{category}`;关闭/无统计退回旧行为,下游强多样性重排仍保证覆盖(`recsys.recall.cold-bandit`)。
- ✅ **近线增量学习(FTRL)**:`FtrlProximal`(FTRL-Proximal LR,稀疏状态 + L1/L2)+ 离线 `train-ftrl`(可 `--incremental` 从 `ftrl:state` warm-start,负采样)近线刷新协同 CTR 模型 → `ftrl:weights`;在线 `FtrlScorer` 在融合加 `weight·pFtrl(user,item)`;哈希契约 `FtrlHashing`(在线/离线一致)。**已验证学习有效**(p(u,i) 训练后 0.50→0.72,负例 0.45)。`recsys.fusion.ftrl`。
- ✅ **多模态海报向量**:`GeminiEmbeddingClient.embedImage`(多模态 inlineData,同 768 维/空间)+ 离线 `backfill-multimodal`(海报图向量与文本向量 `α·text+(1-α)·image` 融合写回 `item_embedding`,pgvector 召回零改动);无海报/不支持多模态则优雅跳过。需海报图数据(MovieLens 用 tmdbId 拉 TMDB)。
- ✅ **语义 ID 当精排稀疏特征**:`SparseFeatureEncoder` schema 驱动、向后兼容地在 (user,item,cat) 后追加 RQ-VAE 语义 ID (c0,c1,c2)(**已验证**:无声明 width=3 不变、启用 width=6 追加、缺失补 0);`train_deepfm.py --semantic-id` 读 `item_semantic_id.csv` 训练并导出 `semantic_id_map.json`+schema 标志。DeepFM 按 `n_fields` 泛化,rank 服务 `sparse[i]=encode(...)` 透传变宽,**在线零改动**;需重训激活。

**工程**
- **测试体系 0→1**:优先在线/离线特征一致性 golden-file 契约测试(`FeatureAssembler`/`SparseFeatureEncoder`/`SequenceEncoder`/`BgeTokenizer`/`BidwordInvCodec` 全是纯函数极易测)+ 核心纯函数单测 + Testcontainers 集成测试 + jacoco 门槛。
- **最小 CI**(GitHub Actions:build + test gate + ONNX 单文件自包含校验,卡"陈旧 target/classes"事故)。
- **告警规则 + SLO**(有 Grafana 看板但零告警;metrics 已现成,补 alert-rules + burn-rate)。
- **结构化日志 + traceId + 分布式追踪**(重 fan-out 却无法定位延迟花在哪路;`micrometer-tracing-bridge-otel` + Tempo/Jaeger)。
- **离线作业调度编排 DAG**(25 个 Job 全靠手动 `--job`,断一环下游静默用陈旧数据;Airflow/Argo 或轻量 `@Scheduled`+分布式锁)。

- **数据质量/漂移监控**(embedding 覆盖率/PSI、特征分布偏移、pCTR vs 实际 CTR 校准偏差)。

## 4. 微服务拆分:真实阻碍与路径(不要 16 模块拆 16 服务)

**四个硬阻碍**:① 共享单库 + 共享 DataSource;② 进程内方法调用无网络契约(RecallService/RankService 是 Java 接口);③ ONNX 模型打进 jar(部署与构建强耦合);④ `recsys-common` 上帝模块全局波及。

**渐进路径**:第 0 步先做治理(Q2/Q3/S3/S4,无论拆不拆都需要)→ 第 1 步剥离 `recsys-embedding`(无状态重算力,依赖最少)→ 第 2 步 `recsys-advertiser`(写侧已天然独立)→ 第 3 步 recall/rank 独立推理服务(需先解决阻碍②③,成本最高,仅在独立扩缩诉求出现时做)。**当前负载下,单体优先仍是正确选择。**
