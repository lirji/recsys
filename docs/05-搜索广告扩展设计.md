## 0. 一句话目标

> 在现有 `recall→rank→rerank` 漏斗之上,接入 **query 驱动的广告检索 + 竞价排序(eCPM)+ 拍卖计费(GSP)+ 预算控制(pacing)+ 概率校准(calibration)**,把"给用户推他喜欢的内容"升级为"在用户搜索意图下,选出对用户相关、且对平台变现最优的广告并正确计费"。

## 1. 搜索广告 vs 内容推荐:三个本质差异(为什么不能直接套用)

| 维度 | 内容推荐(现状) | 搜索广告(目标) | 影响 |
|---|---|---|---|
| **触发** | `userId` 驱动,系统主动选 | `query + userId + context` 驱动,用户主动表达意图 | 必须新增 **Query 理解层**;召回从 user→item 变成 query→ad |
| **排序目标** | 相关性 / CTR 最大化 | **eCPM = bid × pCTR × quality** 最大化(平台收入)+ 相关性约束 | rank 之后要插 **竞价排序层**,且引入广告主出价 |
| **概率用途** | pCTR 仅用于排序,偏一点无所谓 | pCTR **进计费公式**,偏差=多收/少收钱 | 必须 **校准(calibration)** |
| **约束** | 多样性、已看过滤 | 预算、相关性门槛、广告负载、频控、计费公平 | 新增 **pacing / relevance-gate / 拍卖** |
| **反馈标签** | 即时点击/评分 | 点击(即时)+ 转化(长延迟、advertiser 回传) | CVR 需 **延迟反馈建模** |

**一句话**:推荐是 `argmax 相关性`,搜索广告是 `argmax (平台收入) s.t. 用户体验/相关性约束`。多了「钱」和「query」两个轴。

## 3. 数据模型扩展(在 `01_schema.sql` 之上新增)

```sql
-- 广告主
CREATE TABLE advertiser (
  advertiser_id BIGINT PRIMARY KEY,
  name          TEXT,
  daily_budget  DOUBLE PRECISION,        -- 日预算(分/元)
  status        TEXT DEFAULT 'active'     -- active/paused/over_budget
);

-- 广告(创意 + 落地页 + 关联物品)
CREATE TABLE ad (
  ad_id         BIGINT PRIMARY KEY,
  advertiser_id BIGINT REFERENCES advertiser(advertiser_id),
  item_id       BIGINT,                   -- 关联现有 item(复用其 embedding/特征)
  title         TEXT,                     -- 创意标题
  landing_url   TEXT,
  quality_score DOUBLE PRECISION DEFAULT 1.0,  -- 质量度(相关性/落地页,离线算)
  status        TEXT DEFAULT 'active',
  created_at    TIMESTAMP DEFAULT now()
);

-- 竞价词(广告主买的关键词 + 匹配类型 + 出价)
CREATE TABLE bidword (
  id            BIGSERIAL PRIMARY KEY,
  ad_id         BIGINT REFERENCES ad(ad_id),
  keyword       TEXT,
  match_type    TEXT,                     -- EXACT / PHRASE / BROAD
  bid           DOUBLE PRECISION,         -- 出价(CPC,可被 oCPC 覆盖)
  bid_mode      TEXT DEFAULT 'CPC'        -- CPC / oCPC / oCPM
);
CREATE INDEX ON bidword (keyword);        -- 倒排:keyword → ads

-- 广告向量(query 语义匹配用;可直接复用 item_embedding,或单独烘焙)
CREATE TABLE ad_embedding (
  ad_id     BIGINT PRIMARY KEY REFERENCES ad(ad_id),
  embedding vector(768),
  model     TEXT
);
CREATE INDEX ON ad_embedding USING hnsw (embedding vector_cosine_ops);

-- 计费/曝光日志(广告独有,审计 + 校准 + 结算的源)
CREATE TABLE ad_event (
  id            BIGSERIAL PRIMARY KEY,
  request_id    TEXT,                     -- 一次搜索请求
  query         TEXT,
  user_id       BIGINT,
  ad_id         BIGINT,
  bidword_id    BIGINT,
  position      INT,                      -- 广告位次
  event_type    TEXT,                     -- IMPRESSION / CLICK / CONVERSION
  pctr          DOUBLE PRECISION,         -- 预估(计费可复核)
  ecpm          DOUBLE PRECISION,
  charged_price DOUBLE PRECISION,         -- GSP 实际扣费(次高价)
  ts            TIMESTAMP DEFAULT now()
);
```

**Redis Key 扩展**(经 `RedisKeys` 统一生成):

| Key 模式 | 类型 | 内容 | 用途 |
|---|---|---|---|
| `ad:budget:{advertiserId}:{yyyymmdd}` | String(计数) | 当日已消耗 | pacing 实时扣减/熔断 |
| `ad:pacing:{advertiserId}` | String | 平滑系数(PID 输出) | 出价折扣 |
| `bidword:inv:{keyword}` | ZSet | keyword→adId(score=bid) | 精确/短语匹配召回 |
| `ad:calib:{model}` | String/Hash | 校准函数参数(保序回归分段点) | 在线 pCTR 校准 |

## 4. 模块设计与接口契约

### 4.1 Query 理解层(`recsys-query`,✅ 已落地)

**实际契约**(在 `recsys-common` 的 `com.recsys.common.query`):
```java
interface QueryUnderstandingService {
    StructuredQuery parse(String rawQuery, long userId);   // 永不抛异常,逐字段降级
}
record StructuredQuery(
    String raw,
    String normalized,             // 小写/去标点/压空格,可作伪 query 喂语义召回
    List<TermWeight> terms,        // 分词 + 权重(去停用词/去重/截断;权重恒 1.0,IDF 留 TODO)
    List<CategoryScore> intents,   // 意图类目(降序、过阈值);复用现有 19 个 genre 体系
    List<String> rewrites,         // 改写/同义扩展(配置 recsys.query.synonyms,MVP 最小)
    float[] embedding              // 复用 EmbeddingClient.embedText;不可用时为 null
) { boolean hasEmbedding(); String topIntent(); }
```
- **实现** `QueryUnderstandingServiceImpl` 分五步,**每步独立降级**:归一化 → 分词+权重 → 意图(①词项命中 genre 名 强投票 ②`item.title ILIKE ANY(terms)` 聚合 category 投票 → 归一化过阈值取 TopN)→ 向量化(`ObjectProvider<EmbeddingClient>` 可选)→ 改写。可调参 `recsys.query.*`(stopwords/maxTerms/intentMinScore/synonyms…)。
- **现状**:意图识别纯词法 + 标题投票,**无库/无 embedding 也能工作**;`embedding` 当前恒 null(Gemini 403 / 本地 BGE 未实现),待修后语义那一路自动点亮。
- **已接入召回**(见 §5、§7):编排层有 query 时把 `normalized` → `params["query"]`(SEMANTIC 读)、`intents` → `params["intentCategories"]`(TAG 按 `weight=1+intentBoost·score` 加权),并把 SEMANTIC/TAG 并入启用召回路。实测意图类目精确驱动 TAG(同一物品仅当其类目命中 query 意图才带 TAG 标记)。

### 4.2 广告召回(`recsys-ad` recall 子层,✅)

多路并行,合并去重(同推荐),但路子是 **query→ad**:

| 通道 | 原理 | 数据源 | 对应推荐里的 |
|---|---|---|---|
| `KW_EXACT` | query 精确/短语命中 bidword | `bidword:inv:{keyword}` | (广告独有,关键词倒排) |
| `KW_BROAD` | query 改写词 ∪ 同义命中 | bidword + 改写 | (广告独有) |
| `SEMANTIC_AD` | query-emb → `ad_embedding` ANN | pgvector | 复用 `SEMANTIC`/`VECTOR` 逻辑 |
| `U2A` ✅ | 用户长期向量 → `ad_embedding` ANN(query 无关的个性化定向补充) | `user_embedding` | 复用 `VECTOR` |
| `HOT_AD` | 高 eCPM/高质量广告兜底 | Redis | 复用 `HOT` 思路 |

> **U2A 定向**(✅ 已落地):`AdRepository.u2a` 读用户长期兴趣向量(口径同推荐 `VECTOR` 通道)→ `ad_embedding` 余弦 ANN。query 缺失/`SEMANTIC_AD` 因 embedding 降级为空时,U2A 仍能按用户画像定向出广告;新用户无向量 → 该路空、其他路兜底。`relevance-gate` 把 U2A 的 user↔ad 余弦作为相关性依据(同 `SEMANTIC_AD`,以用户兴趣替代 query 相关性)。配额 `recsys.ad.recall.u2a`(默认 80)。

```java
interface AdRecallService {
    List<AdCandidate> recall(StructuredQuery q, AdRecallContext ctx);
}
record AdCandidate(long adId, long bidwordId, double recallScore, AdChannel channel) {}
```

### 4.3 相关性门槛(relevance-gate,✅)

> **广告独有的硬约束**:不相关的广告宁可不出(空位也比恶心用户强)。

- 算 query↔ad 相关性(query-emb 与 ad-emb 余弦 + 关键词匹配度 + 质量度),低于阈值直接丢弃。
- 放在召回后、竞价前;阈值可配 + 可做分层 A/B。

### 4.4 pCTR/pCVR 预估 + 校准(复用 `recsys-rank`,✅ 加保序回归校准)

- **直接复用** DeepFM/MMoE/DIN:输入额外拼 query 侧特征(query-ad 相关性、match_type、position)。MMoE 的 ctr/cvr 双头天然对应广告 pCTR/pCVR。
- **新增校准层**(广告强约束):
```java
interface Calibrator {
    double calibrate(double rawProb, String model);   // 保序回归 / Platt
}
```
  离线用 `ad_event`(预估 pctr vs 实际 CTR)拟合保序回归,参数写 `ad:calib:{model}`,在线查表线性插值。**未校准的 pCTR 不得进计费**。

### 4.5 竞价排序(eCPM ranking,✅)

```
effQuality = pCTR_calibrated × qualityScore × relevance   // 有效质量(对齐 Google Quality Score 思路:含相关性)
eCPM(AdRank) = pacedBid × effQuality                       // pacedBid = bid × pacing 折扣
最终排序按 eCPM 降序,过滤掉 eCPM < reservePrice 的广告
```
- bid 来自 `bidword.bid`,经 pacing 平滑折扣后为 `pacedBid`;oCPC 模式下被智能出价覆盖(见 §6)。
- `qualityScore` 离线算(落地页 + 历史 CTR),进 `ad.quality_score`。
- **相关性纳入排序分**(`BiddingService`):relevance 折进 effQuality,避免高出价但不相关的 `HOT_AD` 兜底广告盖过关键词/语义相关广告——实测 `horror`/`comedy` 等 query 下相关广告(KW_EXACT)稳居前列,HOT 退回真正的兜底位。
- **reserve price(保留价)**:低于保留价不展示,既保体验也保最低价格。

### 4.6 拍卖与计费(GSP,✅)

> 排序与计费**分离实现**(变现正确性的核心)。

- **GSP(广义次价)**:位置 i 的广告,扣费 = 让它保住该位所需的最小出价:
  `price_i = (eCPM_{i+1} / effQuality_i) + 0.01`,且 ∈ [reserve, 自身 pacedBid](`effQuality_i = pCTR_i × quality_i × relevance_i`,与排序分同源,保证次价正确性)。
- 写 `ad_event(charged_price, ecpm, pctr, pctr_calib, relevance)` 做审计与结算。
- **计费时机**:CPC 模式在**点击**时按该次曝光算好的 `charged_price` 扣广告主预算(`POST /api/ad/click` → `PacingService.charge`),非曝光即扣。
- 进阶:**VCG(✅ 2026-06-23,`VcgAuction`)**——理论激励相容(说真话出价是占优策略),广告付它给其他人造成的福利损失(externality)而非"保位最小出价"。完整 VCG 相对 GSP/List-wise 近似补的关键一环是**位置点击折扣模型** `CTR(i,j)=effQuality_i×θ_j`(θ_1≥θ_2≥…为位次折扣):第 k 位每次曝光总付费 `T_k=Σ_{j=k+1}^{m+1} value_{(j)}·(θ_{j-1}−θ_j)`(移除它 → 下方各广告上移一位的福利损失),换算 CPC `price_k=T_k/(θ_k·effQuality_k)+ε`,clamp `[reserve, 自身出价]`。退化自洽:单广告位 ⇒ 与 GSP 单位次完全一致;θ 全相等 ⇒ 人人付首落选者 eCPM÷自身有效质量。分配仍按 `rankEcpm`(含 EE 探索加成)占位、计费用不含加成的 `value=pacedBid×effQuality`(守 §8 #3 红线)。开关 `recsys.ad.auction.vcg.enabled`(默认关,开启时优先于 List-wise/GSP);`positionDiscounts`/`tailDecay` 配位置折扣。验证 `VcgAuctionTest`(6 例,含单位次退化为 GSP、位置折扣外部性定价、reserve 托底边际)。解释成本高、工业界仍多用 GSP,故默认关、三种拍卖机制并存可切换。

### 4.7 预算控制 / Pacing(`recsys-ad` pacing 子层,✅)

- **实时熔断**:每次计费扣 `ad:budget:{advertiserId}:{today}`,超预算 → 该广告主下线(`status=over_budget`),召回层过滤。
- **平滑(pacing)**:PID 控制器按"理想花费曲线 vs 实际"调 `ad:pacing` 折扣系数,避免上午花光(均匀/按流量曲线)。

### 4.8 混排 / 广告负载(Ad Load,✅ 已落地 2026-06-22)

- **入口** `GET /api/feed?q=&userId=&size=`(`FeedOrchestrator`):query 驱动取自然结果(复用 `RecommendOrchestrator`)+ 竞得广告(复用 `SearchAdsOrchestrator`)→ 频控过滤 → 混排 → 返回带"赞助"标记的信息流(`BlendedFeedResponse`/`FeedEntry`,`ad=true` 即广告)。
- **混排**(`AdMixer`,`recsys-ad`):广告按配置位次插入(`recsys.ad.ad-load.slots`,默认 `[2,6,10]`)、密度上限 `max-ads`(默认 3);**去重**——广告关联 item 已作自然结果出现则跳过;自然结果耗尽即停止(不硬凑广告)。纯函数无副作用。
- **频控**(`FrequencyCapService`,`recsys-ad`):Redis 日维度计数 `ad:freq:{user}:{ad}` / `ad:freq:adv:{user}:{adv}`,混排前丢弃超「同广告 `per-ad`(默认 3)/ 同广告主 `per-advertiser`(默认 8)」的广告,展示后累加(带 TTL)。
- **优雅降级**:Ad Load 关 / 无 query → 纯自然流;广告链路异常 → 退自然流;Redis 不可用 → 不限频。
- **本地实测(2026-06-22)**:`/api/feed?q=war space aliens` → 广告(Aliens,adId 118)精确落在**位次 2**(配置 slot)、标"赞助";连续请求同用户,该广告曝光 **3 次后被频控拦下**(`per-ad=3`),计数器实测精确到 3。

## 5. 一次搜索广告请求时序(对照 `02` §6)

> **现状(已落地)**:`GET /api/search?q=&userId=&size=` 走的是「推荐漏斗 + query 注入」——
> `recsys-query` parse → 把 `normalized`/`intents` 注入 `RecallContext.params` → SEMANTIC(读 query,
> 当前 embedding 降级空)+ TAG(按意图类目加权)驱动召回 → 复用现有 rank/rerank。
> query 请求**跳过结果缓存**(避免不同 query 同 scene 串味)。下面是**目标的广告版时序**(M4–M5 逐步补全)。

```
1. 前端 GET /api/search-ads?q=...&userId=1&slots=3
2. recsys-query:parse → StructuredQuery(terms/intent/rewrites/query-emb)
3. ad-recall:KW_EXACT/BROAD + SEMANTIC_AD + U2A + HOT_AD 并行 → 合并去重 → 候选
4. relevance-gate:query↔ad 相关性 < 阈值的剔除(广告独有硬过滤)
5. pacing 过滤:over_budget / 暂停的广告主剔除
6. rank:复用 DeepFM/MMoE 算 pCTR/pCVR → calibrator 校准
7. 竞价排序:eCPM = bid × pCTR_calib × quality,降序,过滤 < reserve
8. 拍卖:取 top slots,GSP 算每个位置的 charged_price
9. pacing 扣减:ad:budget 累加 charged_price,触顶则该主下线
10. 混排:与自然推荐结果按 Ad Load 规则混排 + 频控
11. 曝光:写 ad_event(IMPRESSION, pctr, ecpm, charged_price),返回
12. 点击→ad_event(CLICK);转化→advertiser 回传 ad_event(CONVERSION)
    → 离线:校准重拟合 / oCPC 调价 / 质量度更新 / 反作弊
```

### M6 — 智能出价 + 进阶变现(🟢 大半落地)
- ✅ **oCPC 智能出价**(2026-06-19):广告主只设 `target_cpa`,平台 `bid = targetCpa × pCVR × k` 自动出价。
  - **在线**(`OcpcBidder`,注入 `BiddingService`):oCPC 广告把目标 CPA 换算成竞价出价(pCVR 复用 mmoe/din 多目标模型的 pCVR head,经 `RankedItem.featureSnapshot["pCVR"]` 透出;无则用先验 `default-pcvr`),随后照常进 eCPM 排序 + GSP 计费(**排序/计费逻辑不变**,oCPC 只动"出价怎么来")。
  - **离线**(`ad-ocpc` 作业,反馈控制):`实际CPA = 花费/转化`,`k_new = clamp(k_old × (目标CPA/实际CPA)^alpha, kMin, kMax)` → Redis `ad:ocpc:{advertiserId}`;无转化则保持 k(不乱调)。
  - **数据**:`02_ad_schema.sql` 的 `ad` 加 `optimization_type/target_cpa`;`seed-ads` 造 ~30% oCPC 广告;`sim-ad-events` 补 CONVERSION 事件喂反馈环。
  - **优雅降级**:oCPC 关闭 / CPC 广告 / 无 target_cpa → 退手动出价;k 缺失 → 1.0;`max-bid` 护栏防出价失控。
- ✅ **延迟转化建模(Delayed Feedback)**(2026-06-22):转化在点击后 T+N 天才到达,任何"按近窗口数转化"的口径都会因近期点击尚未来得及转化而**系统性低估转化、高估 CPA**,直接喂 oCPC 反馈控制就会把 `k` 过度调低、错杀本来达标的广告主。
  - **离线拟合**(`ad-delay` 作业 / `DelayModelJob`):把"已到达转化"的延迟 `d = 转化时刻 − 点击时刻` 建模成**指数分布**,MLE `λ = 1/mean(观测延迟)` → Redis `ad:delay:lambda`。转化**完成曲线** `c(e) = P(延迟 ≤ e) = 1 − e^(−λ·e)`:一次 elapsed=e 的点击若终将转化,到现在已转化的概率即 c(e)。
  - **纠偏**(`ad-ocpc --delay-correct`,默认开):对每个**已到达**转化按其点击的 elapsed 做 **Horvitz–Thompson 加权** `1/c(elapsed)`,把"还在路上"的转化补回来 → 无偏的"终值转化数" → `实际CPA = 花费 / 终值转化`。花费与转化同以"点击落窗口内"为口径,保证 CPA 同分母。
  - **数据**:`sim-ad-events` 把曝光/点击铺在过去 `--days` 天、转化时刻 = 点击时刻 + 指数延迟(均值 `--conv-delay-days`),晚于 now 的转化即"尚未到达"(右删失,下游 `ts <= now()` 只看已观测部分)——于是延迟偏差可复现可量化。**实测**(20000 曝光、平均延迟 4 天):760 转化仅 536 到达(29% 在途);未纠偏广告主 21 实际CPA=3.75(原始 21 转化)、纠偏后 3.11(终值 25.34,逼近目标 3.0);广告主 27 未纠偏 3.64→`k` 错误下调,纠偏后 2.86(低于目标)→`k` 正确上调。
  - **优雅降级**:λ 缺失 / `--delay-correct=false` → 退原始计数(不纠偏,等同旧行为);样本不足 `ad-delay` 不写 Redis(宁可不动也不乱估);`--min-completion` 给权重设上限防极近点击爆炸。**已知局限**:只用已到达样本拟合会因右删失略低估均值延迟(实测 λ 对应 2.8 天 < 真值 4 天)→ 轻微欠纠偏;真实系统应做带删失的 Delayed Feedback Model(Chapelle 2014)联合估计转化概率与延迟。
- 混排 Ad Load 调优、反作弊
- **交付标准**:✅ 广告主只设目标成本即可投放,转化成本由 `ad-ocpc` 反馈控制收敛(`ad-report` 可量化实际 CPA)

### M7 — 生态与精排进阶(✅ 已落地 2026-06-23)

> M6 把"钱怎么收、出价怎么定、成本怎么控、新广告怎么起量、计费怎么防作弊"都打通了。M7 是把广告从
> "逐条最优"推向"**整页/整序列最优 + 创意与质量度精细化**"——收益边际递减但更接近成熟广告系统的形态。
> 每项仍坚持 §9 原则:**排序/计费分离、未校准概率不进计费、每层可降级、复用优先**。

1. ✅ **List-wise 外部性(Externality,2026-06-22)**:逐条 eCPM 降序 + 逐位独立 GSP 忽略广告间相互影响——相邻同类广告互相蚕食 CTR(替代效应)。逐条最优 ≠ 整页最优。
   - **选择**(贪心整页,纯函数 `ListwiseExternality`):第 1 位取 rankEcpm 最高;之后每位选 `adjRank = rankEcpm × extFactor` 最大者,`extFactor = clamp(1 − λ·maxSim(已选), minRetention, 1)`——与已展示广告越像衰减越狠(MMR 思路,相似度复用 `item_embedding` 余弦,`AdEmbeddingSimilarity` 批量预载,缺向量退 0)。于是"出价略低但更多样"的广告可后来居上。
   - **计费**(外部性 GSP):位 i 扣费 = 它在该位竞争时挤掉的次优 `adjRank` ÷ 它自己上下文有效质量 `effQuality × extFactor` + ε,clamp `[reserve, 自身出价]`。经济含义:上下文里表现更差(extFactor 低)的广告需更高出价才能保位,按其更低的上下文质量折算出更高的每次点击价——**可审计可对账**(守 §8 #3)。
   - **复用 + 接入**:`BiddingService.auction` 加 `sim` 入参,开关 `recsys.ad.auction.listwise.enabled`(默认关,`AD_LISTWISE_ENABLED`);关闭或无相似度 → extFactor 恒 1 → 与逐条 eCPM + 原 GSP **完全等价**(零行为变化)。编排层仅在开启时按候选 item 预载向量。
   - **边界(诚实)**:这是 **GSP-with-externality** 教学近似,衡量"广告间内容替代"而非完整 social welfare;完整 VCG 见下条第 4 项(已落地,补上位置点击折扣模型 + 每次点击单位换算)。二者是不同维度的外部性(List-wise 管同质蚕食、VCG 管位置福利),互斥可切换。序列/外部性拍卖的博弈稳定性(防联合出价套利)仍未覆盖。验证:`ListwiseExternalityTest`(5 例,含"冗余高价广告被挤到多样广告之后""扣费落 [reserve,出价]""minRetention 托底")+ 本地 `/api/search-ads` 开/关对比同 query 竞得集合不同(externality 生效)。

2. ✅ **动态创意优化(DCO,2026-06-23)**:一个广告多套创意(标题/图/落地页),固定用一套会浪费高潜创意;但新创意无历史 CTR(冷启动)。把"广告级"EE 下沉到"创意级"——每套创意是一条臂,多臂老虎机择优展示,既利用高 CTR 创意又给新创意探索曝光。
   - **数据**:新表 `ad_creative`(creative_id, ad_id, title, landing_url);`seed-ads --creatives-per-ad`(默认 3)给每个广告造标题变体;`ad_event` 加 `creative_id` 归因展示创意。
   - **离线统计**(`ad-explore-stats` 扩展):按 (广告, 创意) 聚合曝光/点击 → Redis `ad:cstats:{adId}:{creativeId}`="imp,clk"(曝光取 IMPRESSION 的 creative_id,点击按 (request_id, ad_id) 关联回曝光行取创意)。
   - **在线选择**(`CreativeSelector` + 纯函数 `CreativeBandit`):**竞价之后**对每条竞得广告,读其创意级统计跑 **UCB1** `CTR + coef·sqrt(ln(adImpr+e)/impr)` 选一条展示(未曝光创意 +∞ 强制先试)。**只换展示创意、不动排序/计费**;展示创意经 `ad_event.creative_id` 归因,闭环喂回 `ad-explore-stats`。UCB 而非 Thompson Sampling 取其确定性可单测(TS 为等效替代)。
   - **复用 + 降级**:沿用 EE 思路、曝光统计物化、频控/反作弊;净增量只是"creativeId 维度 + 创意表"。开关 `recsys.ad.dco.enabled`(默认关,`AD_DCO_ENABLED`)。无创意/Redis 挂/异常 → 退广告默认创意(`creativeId=0`)。
   - **验证**:`CreativeBanditTest`(6 例,空/单臂、未探索强制先试、充分曝光利用高 CTR、coef 对低曝臂的抬升);本地 300 广告×3 创意、`sim-ad-events` 给创意注入 CTR 因子,DCO 开启后 `/api/search-ads` 为 ad75 选 creative 224(CTR 0.20 > 0.10/0.16)、ad257 选 creative 769(CTR 0.28 > 0.14/0.15),均命中最高 CTR 创意。
   - **后续**:创意爆炸致每臂样本稀疏 → 按创意族共享先验(层级老虎机);创意先过质量度门槛再进探索;真实 DCO 还应让创意进 pCTR 模型(本版创意只决定展示、不进 pCTR)。

3. ✅ **质量度精细化(Quality Score,2026-06-23)**:`ad.quality_score` 原是随机基线(M4 占位),却直接进 eCPM(`eCPM = bid × pCTR × quality × relevance`)——随机值等于给"谁出现 + 收多少钱"注入噪声。
   - **离线**(`ad-quality` 作业 / `QualityScoreJob`):从 `ad_event` 聚合三个可解释信号,纯函数 `QualityScore` 融合:(a) **相关性**——历史 `ad_event.relevance` 均值;(b) **CTR 质量**——经验点击率,贝叶斯收缩向大盘 CTR(低量广告 ≈ 大盘,不被小样本带偏);(c) **CVR / 落地页质量代理**——点击后转化率(收缩向大盘 CVR,转化好≈承接好)。各因子先除大盘均值归一到"围绕 1.0 的乘子"、逐项 clamp,再加权(默认 0.3/0.4/0.3)融合并 clamp 到 `[qMin,qMax]`。**平均广告 quality≈1.0、好广告>1、差广告<1**——把随机噪声换成有方向的信号。写 Redis `ad:quality:{adId}`。
   - **在线**(`QualityScoreService`,注入 `BiddingService`):`refined(adId, fallback)` 查表命中用精细分,否则退广告自带 `quality_score`(新广告无历史 → 基线,平滑过渡)。进 eCPM 的 quality 即此值(它是校准后历史聚合的乘子,非未校准单次概率,不破 §9.3 红线)。
   - **可审计**(守 §8 #3):每广告三因子原始值 + 归一值 + 最终分写 CSV `eval/ad-quality-<ts>.csv` 对账;`SponsoredAd.quality` 也透出进 eCPM 的精细分(与 ecpm 可对账)。Redis 只存最终乘子供在线快查。
   - **与 EE 探索**:质量度是**长期**性能加权(进排序也进计费);`ExplorationService` UCB 是**临时**抬权(只进排序)。二者在 Ad Rank 上各自相乘、正交不抵消。
   - **复用 + 降级**:复用 `ad_event` 日志、`RelevanceGate` 口径;开关 `recsys.ad.quality.enabled`(默认开,`AD_QUALITY_ENABLED`)。无数据/Redis 挂/样本不足 → 退随机基线(不引入噪声)。**已知局限**:教学 `sim-ad-events` 不写 `relevance`,故相关性因子在模拟数据下偏弱(大盘 rel≈0),CTR/CVR 因子承担主要信号;真实流量该因子才充分。验证:`QualityScoreTest`(9 例,收缩两端/归一中性化/融合权重与 clamp)+ 本地 `ad-quality` 给 800 广告打分(分布 568 落 [0.8,1.2]、227 偏低、5 偏高)、`/api/search-ads` 中 ad271 由随机 0.91 → 精细 1.234。

4. ✅ **完整 VCG 位置拍卖(VCG,2026-06-23)**:GSP(逐位次价)只让广告付"保住该位的最小出价",**非激励相容**——广告主有动机谎报出价套利,且付费与"自己给系统造成的真实影响"脱钩。VCG 让广告付它给**其他人**造成的福利损失(externality),从而**说真话出价是占优策略**(truthful)。
   - **位置点击折扣模型**(完整 VCG 相对 List-wise 近似补的关键一环,纯函数 `VcgAuction`):点击率按位次衰减 `CTR(i,j)=effQuality_i×θ_j`,`θ_1≥θ_2≥…≥θ_m>θ_{m+1}=0`(`positionDiscounts` 配,超出按 `tailDecay` 几何外推、强制单调非增)。第 j 位每次曝光价值 = `value_i×θ_j`,`value_i=pacedBid_i×effQuality_i`(不含探索加成的 eCPM)。
   - **分配**:按 `rankEcpm`(含 EE 探索加成)降序占位——与 GSP/List-wise 同口径,探索只影响"谁出现"不进计费(守 §8 #3)。
   - **计费**(VCG 外部性):移除第 k 位 → 其下所有广告各上移一位(第 m+1 名首落选者补进第 m 位)的福利损失 `T_k=Σ_{j=k+1}^{m+1} value_{(j)}·(θ_{j-1}−θ_j)`(reserve 以 `max(首落选者价值, reserve)` 进末项托底,等价 GSP 的 reserve 兜底);换算 CPC `price_k=T_k/(θ_k·effQuality_k)+ε`,clamp `[reserve, 自身出价]`。**可审计可对账**:CPC 与 GSP 同单位,落 `ad_event.charged_price`。
   - **退化自洽**:单广告位(m=1)`price=value_{(2)}/effQuality_1` ⇒ 与 GSP 单位次完全一致;θ 全相等 ⇒ 人人付"首落选者 eCPM ÷ 自身有效质量"。
   - **复用 + 接入**:`BiddingService.auction` 复用已算好的 `scored`(校准/出价/质量/探索),仅多委托 `VcgAuction`。开关 `recsys.ad.auction.vcg.enabled`(默认关,`AD_VCG_ENABLED`);**开启时优先于 List-wise/GSP**(三机制互斥可切换)。关闭 → 行为完全不变。
   - **边界(诚实)**:θ 当前为静态配置(真实系统应从曝光日志按位次估计或随 query/版面变化);未建模"点击率非可分"(位置与广告效应耦合)。验证:`VcgAuctionTest`(6 例,含单位次退化为 GSP、位置折扣外部性定价、θ 全相等、reserve 托底边际、扣费落 [reserve,出价]、θ 单调外推)。

5. ✅ **博弈稳定性分析器(Auction Game Theory,2026-06-23)**:有了机制(GSP/VCG)还不够——计费正确性(§8 #3)要求知道广告主能否靠**谎报出价**或**联合出价**套利。纯函数 `AuctionGame` 用分离-CTR 位置拍卖标准模型(`CTR(i,j)=effQuality_i×θ_j`,同 `VcgAuction`)把"博弈稳定性"从口号变成可度量、可单测的结论。**离线诊断,不改在线 `BiddingService`**。
   - **激励相容(DSIC)** `maxTruthfulRegret`:全员真话时任一广告主单边改报能多赚多少。**VCG ⇒ 0**(说真话是占优策略);**GSP ⇒ >0**——`bestResponse` 精确暴露获利偏离(按"落在某位次区间内价格与自身出价无关",对每个可达位次取代表出价即得全局最优,无需网格)。本地设定 value=[10,8,5]、θ=[1,0.6]:GSP 顶位 A 真话付 8(效用 2),压价落次位付 5(效用 3)→ 多赚 1。
   - **GSP 对称纳什均衡 ⇒ VCG 价**(EOS/Varian 等价定理)`gspSymmetricNashBids`:GSP 虽不真实,但其(最低)对称纳什均衡的每次点击价 = VCG 价。由 VCG 价反推均衡出价(`b_(s)=p^VCG_(s-1)·effQ_(s-1)/effQ_(s)`,顶位/落选者出真值),验证 GSP 跑该出价复现 VCG 分配与价、且无人有获利偏离(确是均衡)。**含义**:即便线上跑 GSP,理性广告主的稳定出价把价格收敛到 VCG——VCG 是正确参考价。
   - **联合出价套利** `findCollusionArbitrage`:在出价乘子网格上搜"成员各自降价但**位次不变**"的偏离,找能压低联盟总付费者。**结论(诚实)**:GSP 与 VCG **都**可被联盟操纵——VCG 只防单边、不防合谋(VCG 非 group-strategyproof:联盟里下位者降价 → 上位者外部性下降 → 上位者付费降)。这正是"序列/外部性拍卖博弈稳定性(防联合出价套利)"的核心风险,分析器使其**可复现、可量化**而非空谈。
   - **边界(诚实)**:等价定理的干净形式要求同质广告效应,故 SNE 验证用等 effQuality 经典设定(异质时位次序可能被打破);合谋检测是**网格搜索**(命中即报,非最优性证明);分析器吃抽象出价/价值,尚未接真实竞价日志(§8 #1 无真实广告数据)。验证:`AuctionGameTest`(8 例,VCG 真实 / GSP 不真实 + 暴露偏离 / SNE 复现 VCG 且为均衡 / GSP+VCG 均被联盟套利 / 单位次 GSP=VCG)。

- **交付标准**:✅ List-wise 外部性整页选择 + 外部性 GSP、✅ DCO 多臂老虎机自动选高 CTR 创意且新创意有探索曝光、✅ 质量度由可解释离线信号(相关性/CTR/CVR)驱动、✅ 完整 VCG 位置拍卖(位置点击折扣 + 外部性定价,激励相容,单位次退化为 GSP)、✅ 博弈稳定性分析器(VCG 真实性 / GSP 非真实性 / GSP-SNE≈VCG / 联合出价套利检测);M7 全部落地。

## 8. 分库分表(ShardingSphere-JDBC,✅ M8 2026-06-23)

> 广告数据按广告主水平扩展:用 ShardingSphere-JDBC 把广告表分到两个 database,主键改用 Snowflake。**写侧(`recsys-advertiser`)+ 在线读侧(`rec-engine`/`recsys-ad`,§8.1)+ 离线侧(`recsys-offline`,§8.2)全链路已接入并实测。**

### 8.0 分布式主键:Snowflake(雪花算法)

> 分库后不能再用单库自增 / IDENTITY —— 各库各自增会撞主键。改用 **Snowflake**:在本地(中间件/应用层)生成**全局唯一、趋势递增**的 64bit 长整型,无需中心协调。

- **64 位结构**(ShardingSphere 默认):`1 符号位 + 41 时间戳(ms) + 10 workerId + 12 序列号`
  ```
  0 | 41 bit 时间戳(自定义纪元起的毫秒) | 10 bit workerId | 12 bit 序列号
  ```
  - 时间戳 41 位:自纪元起约 **69 年**;workerId 10 位:**1024** 个节点;序列号 12 位:同一毫秒每节点最多 **4096** 个(≈ 409 万/秒/节点)。
- **为什么选它**:
  - vs 单库自增/IDENTITY:**去中心化、跨分片不冲突**,不依赖单库;
  - vs UUIDv4:**8 字节**(UUID 16 字节)且**时间有序** —— UUIDv4 随机 → B+ 树乱序插入、页分裂、索引膨胀;
  - vs 号段(Leaf-segment)/Redis INCR:**本地生成、无中心点 RTT、无丢号风险**。
- **时间有序的价值**:主键随时间单调递增 → B+ 树近似顺序追加,写入局部性好,高并发写不退化。
- **两个工程注意**:
  1. **时钟回拨**:NTP 把时钟拨回会生成重复 id;生产用 Leaf-snowflake / Baidu UidGenerator 做回拨保护。本脚手架用 ShardingSphere 内置实现、`worker-id` 固定 1(教学够用;同机多实例须各异,部署按 Pod 序号注入)。
  2. **别拿低位取模分片**:低 12 位是序列号,低 QPS 下每毫秒仅生成 1 个 → 恒为 0 → `id % 2` 恒偶 → 全压一库(本仓踩过,见下方坑 #1)。
- **核心原则:主键 ≠ 分片键**。主键用 Snowflake(全局唯一、与业务解耦、有序);**分片键选业务维度**(本仓 `ad` 按 `advertiser_id`),让"某广告主的广告"等查询单库命中、且分布均匀 —— 拿主键(Snowflake)直接取模分片会因低位坍缩而倾斜。
- **本仓落地**:`keyGenerators: snowflake: type: SNOWFLAKE`,各分片表 `keyGenerateStrategy` 指定主键列;在线写侧(`AdvertiserRepository`)insert 不带主键列 → 中间件注入 → `getGeneratedKeys()` 回传;批量 `seed-ads` 因需确定性 + 回读 `bidword.id`,改显式赋值并经分片键路由(§8.2)。

- **拓扑**:同一 `recsys-postgres` 容器内两个 database —— `ds_0`=`recsys`(原库)、`ds_1`=`recsys_ds1`(新库)。`spring.datasource.url=jdbc:shardingsphere:classpath:sharding.yaml`、`driver-class-name=org.apache.shardingsphere.driver.ShardingSphereDriver`。
- **分片规则**(`recsys-advertiser/src/main/resources/sharding.yaml`):

| 逻辑表 | 分片键 | 算法 | 落点 |
|---|---|---|---|
| `advertiser` | `advertiser_id` | `INLINE ds_${advertiser_id % 2}` | ds_0 / ds_1 |
| `ad` | **`advertiser_id`** | 同上 | 与广告主同库(广告主作用域查询单库命中;分布跟随广告主) |
| `bidword` | `ad_id` | `INLINE ds_${ad_id % 2}` | ds_0 / ds_1 |
| `ad_creative` | `ad_id` | 同上 | ds_0 / ds_1 |
| 其余(`item`/`*_embedding`/`ad_event`…) | — | `!SINGLE defaultDataSource: ds_0` | 全在 ds_0 |

- **主键**:ShardingSphere 的 `SNOWFLAKE` keyGenerator(64bit、时间有序、去中心化);insert 不带主键列,ShardingSphere 注入后经 `Statement.getGeneratedKeys()` 回传(替代单库 IDENTITY——分库后跨库自增会冲突)。
- **两个关键坑(已规避)**:
  1. **别按 Snowflake 低位取模**:Snowflake 低 12 位是序列号,低 QPS 下每毫秒仅生成 1 个 → 恒为 0 → `ad_id % 2` 恒偶 → 全压一库。故 `ad` 改按 `advertiser_id` 分(`advertiser_id` 跨请求分布更均;真实生产可改 hash 或业务键)。
  2. **向量表不分片**:`ad_embedding` 留单表(ds_0)。向量 ANN(hnsw + `<=>` 余弦)必须看到全量,分片会漏召回;且 pgvector 的 `<=>` 等非标准算子不进 ShardingSphere 的分片 SQL 解析路径。
- **跨分片 JOIN 已消除**(ShardingSphere 对 sharded⋈single 需 SQL federation,易慢/受限):`report` 拆两查(广告表广播取广告 → `ad_event` 单表按 `ad_id` 聚合);`findAdRow`/`listAdRows` 把 `ad_embedding` 标志改为单表批查;`advertiser`↔`ad` 同按 `advertiser_id` 分库声明为 `bindingTables`(读侧 `ad⋈advertiser` 状态过滤路由同库、不产生跨库笛卡尔积)。
- **落库前置**:先建 `recsys-offline/sql/04_ad_sharding.sql` —— ① ds_0 拆跨库外键(`ad`↔`advertiser`/`item`、`ad_embedding`↔`ad` 等跨库不成立)+ 去 IDENTITY;② `CREATE DATABASE recsys_ds1`;③ ds_1 建 4 张分片物理表(纯 bigint 主键、无外键)。

### 8.1 读侧接入(rec-engine / `recsys-ad`,✅ 2026-06-23)

> 在线读侧也要看到落在 `ds_1` 的广告。难点:`rec-engine`/`recsys-ad` 满是 pgvector(`<=>`、`vector`、`tsvector`)SQL,**整库走 ShardingSphere 会让解析器在这些非标准算子上报错**。解法:**双数据源**,只把广告表交给 ShardingSphere。

- **双数据源**(`recsys-ad/AdShardingConfig`,被 rec-engine 扫描;sharding.yaml 移到 `recsys-common` 单一来源):
  - **主**(`@Primary`,普通 Postgres):召回 / 排序 / embedding / pgvector ANN / `ad_event` 等绝大多数 SQL —— **零改动**,pgvector 不进 ShardingSphere 解析路径。
  - **次**(`adShardingJdbc`,ShardingSphere):仅 `AdRepository`/`PacingService`/`CreativeSelector`/`AdEventLogger` 读分片广告表。定义了自有 DataSource 会让 Boot 自动配置退避,故主、次都显式声明、主标 `@Primary`(其余 bean 注入零改动)。
- **`recsys-ad` 改造**:`AdRepository` 把"向量 ANN + 详情"拆两步 —— ANN 在主库 `ad_embedding`(不分片、全量)跑出候选 `ad_id`,再用分片库取 `ad⋈advertiser` 详情(绑定表、广播)与 `bidword` 出价(按 `ad_id` 聚合);`kwByDb`/`hot` 同样拆表查、Java 合并;`ocpcParams`/`titles` 直接走分片库(`ad_id` 广播);`PacingService` 的 `advertiser` 预算读、`CreativeSelector` 的 `ad_creative` 读、`AdEventLogger` 点击归因(`ad_event` 主库取计费额 + 分片库按 `ad_id` 取广告主)均改走分片库。
- **本地实测(2026-06-23)**:写侧——API 建广告主 `advertiser_id` Snowflake 按奇偶落 ds_0/ds_1、广告与广告主同库、`get/list/report`/暂停/删除经 ShardingSphere 正确路由 + Redis 倒排同步。读侧——`/api/recommend`(pgvector VECTOR 召回)不受影响;`/api/search-ads` 对 ds_0 与 **ds_1** 的广告均能召回 + 竞价 + GSP 计费(`KW_EXACT` 走 Redis 倒排 + 分片库取详情;`U2A` 走主库 ad_embedding ANN + 分片库回填,且能读回旧种子广告)。
### 8.2 离线侧接入(`recsys-offline`,✅ 2026-06-23)

> 批作业也要写对/读全分片广告表。同构方案:`OfflineAdShardingConfig` 双数据源(主普通库 + 次 ShardingSphere),只 **3 个**作业碰分片表,其余只读 `ad_event`(单表,主库,不变)。

- **边界 / 下一步**:① 现只**分库**未分表(`actualDataNodes` 加 `_${0..1}` 表后缀即分表);② 分库前 `seed-ads` 旧行若残留只在 ds_0,按 `advertiser_id` 路由的查询(如 listAds)会漏看,广播类查询(ANN/keyword)仍可命中,真实历史数据迁移需重分布作业;③ worker-id 现固定 1,同机多实例须各异(部署按 Pod 序号注入);④ offline 多 arg 用空格分隔的 `"-Dspring-boot.run.arguments=--job=seed-ads --clear --ads=200"`(逗号形式不拆分)。

## 10. 关键设计原则(承接 `02`/`04` 的一贯主张)

1. **在线只查 + 轻算,重活离线**:校准参数、质量度、oCPC 系数、pacing 曲线全离线/近线算好写进 Redis,在线查表。
2. **排序与计费分离**:eCPM 决定"谁出现",GSP 决定"收多少钱",两套逻辑独立可测。
3. **未校准概率不进计费**:这是广告区别于推荐的红线。
