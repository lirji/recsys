# 推荐系统 · 关键技术点

> 这份文档是"讲明白原理"的核心。每个技术点 = 它解决什么问题 + 怎么做 + 本项目最佳实践 + 易踩的坑。

## 1. 召回(Recall)—— 多路召回

**解决什么**:从百万物品里快速、广撒网地捞出几百个"可能相关"的候选。追求**快和全**,不追求准。

**本项目用 4 路召回(互补):**

| 召回路 | 原理 | 数据来源 | 解决的问题 |
|---|---|---|---|
| **向量召回** | 用户向量 ↔ 物品向量,找语义最近邻 | pgvector + Gemini embedding | 语义/兴趣相似 |
| **i2i 召回(ItemCF)** | "看了A的人也看B" | 离线算的 `i2i:{itemId}` | 行为协同,效果稳 |
| **热门召回** | 全局/分类热度 Top-N | `recall:hot` ZSet | 兜底、冷启动 |
| **标签召回** | 用户偏好类目/标签 → 取物品 | 用户画像 + item.tags | 可解释、强相关 |

**最佳实践:**
- 每路独立、可并行、可配额(如 vector 取 200、i2i 取 100、hot 取 50)。
- 召回结果**带 channel 标记**,合并去重时保留"来自哪几路"(用于展示理由 + 调试)。
- 任一路挂了不影响整体(热门兜底永远在)。

**坑:**
- 新用户没有 user_embedding / 无历史 → 必须有热门 + 标签兜底(见冷启动)。
- 去重要用 itemId,合并多路时注意 score 不在一个量纲,**别直接相加**(交给排序层统一打分)。

---

## 2. 向量化(Embedding)—— 第三方 + 可降级

**解决什么**:把"电影简介""用户兴趣"这种文本变成向量,才能算相似度。

**做法:**
- 物品侧:`title + category + tags + description` 拼成一段文本 → `EmbeddingClient.embedText()` → 存 `item_embedding`。
- 用户侧:取用户近期正反馈物品的向量,**做加权平均**得到 user_embedding(简单有效,无需训练)。
- 多模态(进阶):电影海报图 → `embedText`/`embedImage` 融合。

**最佳实践:**
- **抽象接口 + 缓存**:`emb:cache:{textHash}` 缓存结果,省 API 额度、提速。
- **维度固定写死配置**(如 768),全库一致;换模型 = 换实现类 + 全量重灌一次。
- **批量化**调用(一次请求多条文本),减少 API 次数。
- **降级链**:Gemini 超额/超时 → 本地 BGE(ONNX,CPU)→ 保证离线灌库不中断。

**坑:**
- 不同模型维度不同(768 / 1024 / 1536),混用会导致检索结果错乱 → 用 `model` 字段标记来源。
- API 有速率限制,灌库时加限流 + 重试退避。

---

## 3. 向量检索 —— pgvector

**解决什么**:在几十万向量里毫秒级找出和"用户向量"最近的 N 个。

**做法:**
```sql
SELECT item_id, 1 - (embedding <=> :userVec) AS sim
FROM item_embedding
ORDER BY embedding <=> :userVec     -- <=> 余弦距离
LIMIT 200;
```
- 建 **HNSW 索引**(近似最近邻,快;精确扫描慢)。
- 距离选 **cosine**(`vector_cosine_ops`),和 embedding 语义匹配最常用。

**坑:**
- 没建索引会全表扫描,数据量一大就慢 → 务必建 HNSW。
- HNSW 是近似检索,召回率和速度可调(`ef_search` 参数),讲原理时可对比 IVFFlat。

## 4. 排序(Rank)—— CTR 预估,离线训练 / 在线服务

**解决什么**:召回的几百个候选,精准排出"用户最可能点击"的顺序。这是推荐系统**最核心、最能体现技术含量**的一层。

**分三阶段进化(从能跑到能讲透):**

| 阶段 | 方法 | 说明 |
|---|---|---|
| v1 | **加权规则打分** | `score = w1*召回分 + w2*热度 + w3*画像匹配`,纯 Java,零模型,先跑通链路 |
| v2 | **逻辑回归 LR** | 引入特征工程,理解"特征→点击概率",可手写或 ONNX |
| v3 | **GBDT (LightGBM)** | 业界主流,Python 训练→ONNX→Java 加载打分 |

**离线训练 / 在线服务标准流程(v3):**
```
行为日志 → 构造样本(label=是否点击, features=用户×物品特征)
       → LightGBM 训练(CPU, 几分钟)→ 评估 AUC
       → 导出 model.onnx → 放到 recsys-rank 资源目录
在线: rank 装配同样的特征 → ONNX Runtime 打分 → 排序
```

**最佳实践:**
- **离线/在线特征必须一致**(同样的特征名、同样的计算逻辑),否则线上效果崩 —— 这叫"特征穿越/不一致",是推荐系统第一大坑。
- 特征快照随结果返回(`featureSnapshot`),便于 debug 和做训练样本。
- 负样本:展示了未点击 = 负样本(impression 日志很重要)。

**坑:**
- **特征不一致**(离线用了在线拿不到的特征)→ 训练好上线却拉胯。
- **数据穿越**(用了"未来"信息当特征)→ 离线 AUC 虚高,上线翻车。

---

## 5. 重排(Re-rank)—— 多样性 + 业务规则

**解决什么**:精排只管"准",但全是同一类型会腻。重排负责**体验和规则**。

**做法:**
- **多样性打散**:同类目连续不超过 N 个(简单滑窗),或 MMR(相关性与多样性权衡)。
- **业务规则**:过滤已看过/已购买、去重、置顶运营内容、过滤下架物品。

**坑:** 打散过度会牺牲相关性 → 控制好打散强度,可配置。

---

## 6. 特征体系 —— 在线/离线特征

**特征三类:**
- 用户特征:活跃度、偏好类目分布、历史 CTR…
- 物品特征:热度、类目、近 N 天 CTR、上架时长…
- 交叉特征:用户对该类目的历史点击率(最有用!)…

**在线/离线:**
- 离线批算 → 写入 Redis(`feat:user:*` / `feat:item:*`)→ 在线直接读。
- 进阶:Flink 实时更新部分特征(如实时点击数),体现"实时性"。

---

## 7. 行为反馈闭环 —— 系统越用越准

```
用户点击/播放 → POST /api/behavior → Kafka
   → 更新在线特征(实时 CTR)
   → 累积训练样本 → 定期重训模型
   → 更新 user_embedding / i2i 倒排
```
**这是推荐系统区别于普通搜索的灵魂**:闭环让系统自我进化。学习项目里哪怕做成"每天跑一次离线作业"也完整体现了原理。

---

## 8. 冷启动(Cold Start)

| 场景 | 策略 |
|---|---|
| **新用户**(无行为) | 热门召回 + 注册时选的兴趣标签 + 人群热门 |
| **新物品**(无行为) | 靠内容向量(embedding)进入向量召回;给曝光探索机会 |
| **新系统**(无数据) | 先灌 MovieLens 历史数据 bootstrap |

> 冷启动是面试高频题,也是这个项目的亮点之一,务必在文档里讲清楚。

---

## 9. 评估指标 —— 怎么证明"推得准"

**离线指标(用历史数据评估):**
- 排序模型:**AUC**(最常用)、LogLoss。
- 召回:**Recall@K**、命中率。
- 整体:**NDCG@K**(考虑排序位置)、MAP。

**在线指标(真实/模拟反馈):**
- **CTR**(点击率)、人均点击数、覆盖率、多样性。

**做法:** 划分训练/测试集(按时间切,避免穿越),离线先看 AUC/Recall@K;前端做个简单"点击模拟"看 CTR 变化。

---

## 10. A/B 实验(生产必备)

- 按 `userId % 100` 分桶,实验配置化(分流比例、策略、指标),A 组规则、B 组模型,对比 CTR。
- 生产脚手架要求:**实验配置可热更**(Nacos)、**埋点带实验组标记**(behavior 日志记录 bucket)、**指标看板**可分组对比。

## 12. 生产化关注点(本脚手架必须预留)

| 关注点 | 做法 |
|---|---|
| 可观测 | Actuator 健康检查 + 结构化日志 + traceId 全链路;预留 Prometheus/Grafana 接口 |
| 降级/兜底 | 召回每路独立可降级、热门永远兜底;embedding 超时降级本地;排序模型加载失败回退规则打分 |
| 样本回流 | impression/click 全量埋点 → Kafka → 落样本表,保证可重训(闭环的数据基础) |
| 特征一致性 | 离线/在线同一套特征定义,杜绝穿越(见 §4 坑)——生产翻车头号原因 |
| 配置中心 | 召回配额、排序策略、AB 分流全部 Nacos 配置化,可热更不停服 |
| 灰度/回滚 | 模型文件版本化(model 字段 + 版本目录),可快速回滚到上一版 |
| 幂等/限流 | 行为上报幂等、embedding API 限流重试退避 |

---

## 11. 一图总结各技术点落在哪层

```
召回   → 多路召回 / 向量化 / pgvector 检索 / 冷启动兜底
排序   → 特征体系 / CTR模型(LR/GBDT) / 离线训练在线服务
重排   → 多样性打散 / 业务规则
闭环   → 行为采集 / 反馈 / 重训
评估   → AUC / Recall@K / NDCG / CTR / A/B
```
