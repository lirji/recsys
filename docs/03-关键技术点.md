# 推荐系统 · 关键技术点

> 这份文档是"讲明白原理"的核心。每个技术点 = 它解决什么问题 + 怎么做 + 本项目最佳实践 + 易踩的坑。
> **每个技术点的深度拆解(含更多实现细节 + 面试要点)见 [`docs/skills/`](skills/README.md)。**

## 1. 召回(Recall)—— 多路召回

**解决什么**:从百万物品里快速、广撒网地捞出几百个"可能相关"的候选。追求**快和全**,不追求准。

**本项目现有 12 路召回(经典 8 路 + 学习/生成型 4 路,互补):** 详见 [skills/01 多路召回全家桶](skills/01-召回-多路召回全家桶.md)。

| 召回路 | channel | 原理 | 数据来源 | 解决的问题 |
|---|---|---|---|---|
| **向量召回** | VECTOR | 用户长期向量 ↔ 物品向量,找语义最近邻 | pgvector + Gemini embedding | 语义/长期兴趣 |
| **i2i(ItemCF)** | I2I | "看了A的人也看B"(物品共现) | 离线 `i2i:{itemId}` | 行为协同,效果稳 |
| **热门召回** | HOT | 全局热度 Top-N(实时 `rt_hot` 优先) | `recall:rt_hot`/`recall:hot` | 兜底、冷启动(始终在线) |
| **标签召回** | TAG | 用户偏好类目 + 实时类目 + query 意图 → 取物品 | 画像 + `rt:user:{id}` | 可解释、强相关 |
| **UserCF(U2U)** | U2U | "和你口味相似的用户喜欢的"(离线物化召回列表) | 离线 `u2u:{userId}` | 用户协同 |
| **Swing i2i** | SWING | 抗热门的物品相似(惩罚兴趣重叠用户对的共现) | 离线 `swing:{itemId}` | 比 ItemCF 更聚焦小众强相关 |
| **语义召回** | SEMANTIC | query / 近期正反馈标题伪 query → embedding → 向量检索 | Gemini + pgvector | 即时意图(区别于 VECTOR 的长期画像) |
| **冷启动探索** | COLD | 跨类目铺开 + UCB 探索(`ColdStartBandit`) | `item` + `cold:cat:*` | 新用户兴趣探索(见 §8) |
| **双塔召回** | TWO_TOWER | DSSM 学习型:item 塔离线烘焙、user 塔在线 ONNX → ANN | `item_tower_embedding` | 学习型召回,补内容向量盲区 |
| **生成式召回** | GENERATIVE | RQ-VAE 语义 ID `(c0,c1,c2)` 前缀检索同簇 | `item_semantic_id` | TIGER 范式可服务版 |
| **词法召回** | LEXICAL | Postgres 全文检索 `ts_rank_cd`(BM25) | `item.title_tsv`(GIN) | 搜索精确匹配(与 SEMANTIC RRF 混合) |
| **TIGER 生成** | TIGER | decoder-only Transformer beam search 自回归生成下一个 item 语义 ID | `tiger.onnx` | 完整生成式召回 |

> **融合关键**:各路先**逐路归一化到 [0,1] 再跨路取 max**(否则热度计数压垮语义/向量余弦分);搜索场景改 **RRF 名次融合**(混合检索)。见 [skills/01 §3](skills/01-召回-多路召回全家桶.md)。

**最佳实践:**
- 每路独立、可并行、可配额(`recsys.recall.quota.*`),任一路挂了不影响整体(热门兜底永远在)。
- 召回结果**带 channel 标记**,合并去重保留全部来源;**主来源按优先级选取**(个性化 I2I/SWING/U2U/VECTOR/SEMANTIC > 偏好 TAG > 探索 COLD > 兜底 HOT),让推荐理由更贴切——否则一个被多路命中的热门片会被标成"热门"。
- **召回路可被分层 A/B 动态启停**:`RecallContext.enabledChannels` 为空=全开,非空=只跑指定路;实验/冷启动据此切换(见 §10、§8)。
- 在线只做"查 + 取 TopN",重活(协同矩阵/相似列表)全离线物化进 Redis。

**坑:**
- 新用户没有 user_embedding / 无正反馈 → I2I/U2U/SWING/VECTOR 都拿不到种子,必须有 COLD/热门/标签兜底(见冷启动)。
- 去重要用 itemId,合并多路时 score 不在一个量纲,**别直接相加**(交给排序层统一打分)。
- 语义召回依赖 Gemini(免费配额 1000/天),失败必须降级返回空、不阻塞链路。

## 2. 向量化(Embedding)—— 第三方 + 可降级

**解决什么**:把"电影简介""用户兴趣"这种文本变成向量,才能算相似度。

**做法:**
- 物品侧:`title + category + tags + description` 拼成一段文本 → `EmbeddingClient.embedText()` → 存 `item_embedding`。
- 用户侧:取用户近期正反馈物品的向量,**做加权平均**得到 user_embedding(简单有效,无需训练)。
- 多模态(进阶):电影海报图 → `embedText`/`embedImage` 融合。

**最佳实践:**
- **抽象接口 + 缓存**:`emb:cache:{textHash}` 缓存结果,省 API 额度、提速。
- **维度固定写死配置**(如 768),全库一致;换模型 = 换实现类 + 全量重灌一次。
- **批量化**调用(一次请求多条文本),减少 API 次数。
- **降级链**:Gemini 超额/超时 → 本地 BGE(ONNX,CPU)→ 保证离线灌库不中断。

**坑:**
- 不同模型维度不同(768 / 1024 / 1536),混用会导致检索结果错乱 → 用 `model` 字段标记来源。
- API 有速率限制,灌库时加限流 + 重试退避。

## 3. 向量检索 —— pgvector

**解决什么**:在几十万向量里毫秒级找出和"用户向量"最近的 N 个。

**做法:**
```sql
SELECT item_id, 1 - (embedding <=> :userVec) AS sim
FROM item_embedding
ORDER BY embedding <=> :userVec     -- <=> 余弦距离
LIMIT 200;
```
- 建 **HNSW 索引**(近似最近邻,快;精确扫描慢)。
- 距离选 **cosine**(`vector_cosine_ops`),和 embedding 语义匹配最常用。

**坑:**
- 没建索引会全表扫描,数据量一大就慢 → 务必建 HNSW。
- HNSW 是近似检索,召回率和速度可调(`ef_search` 参数),讲原理时可对比 IVFFlat。

## 4. 排序(Rank)—— CTR 预估,离线训练 / 在线服务

**解决什么**:召回的几百个候选,精准排出"用户最可能点击"的顺序。这是推荐系统**最核心、最能体现技术含量**的一层。

**分三阶段进化(从能跑到能讲透):**

| 阶段 | 方法 | 说明 |
|---|---|---|
| v1 | **加权规则打分** | `score = w1*召回分 + w2*热度 + w3*画像匹配`,纯 Java,零模型,先跑通链路 |
| v2 | **逻辑回归 LR** | 引入特征工程,理解"特征→点击概率",可手写或 ONNX |
| v3 | **GBDT (LightGBM)** | 业界主流,Python 训练→ONNX→Java 加载打分 |
| v4 | **深度模型全家桶** | DeepFM / DCN v2(显式高阶交叉)/ MMoE / PLE(多任务+ESMM)/ DIN / DIEN / SIM(行为序列→兴趣演化→长序列)——**共 9 策略**,`RANK_STRATEGY` 切换,缺模型回退规则 |

> v4 的 9 个策略 + 双塔粗排、ESMM 消偏、PAL 位置去偏、三大在线离线契约,详见 [skills/02 排序模型全家桶](skills/02-排序-CTR与多任务模型全家桶.md)。

**离线训练 / 在线服务标准流程(v3):**
```
行为日志 → 构造样本(label=是否点击, features=用户×物品特征)
       → LightGBM 训练(CPU, 几分钟)→ 评估 AUC
       → 导出 model.onnx → 放到 recsys-rank 资源目录
在线: rank 装配同样的特征 → ONNX Runtime 打分 → 排序
```

**最佳实践:**
- **离线/在线特征必须一致**(同样的特征名、同样的计算逻辑),否则线上效果崩 —— 这叫"特征穿越/不一致",是推荐系统第一大坑。
- 特征快照随结果返回(`featureSnapshot`),便于 debug 和做训练样本。
- 负样本:展示了未点击 = 负样本(impression 日志很重要)。

**坑:**
- **特征不一致**(离线用了在线拿不到的特征)→ 训练好上线却拉胯。
- **数据穿越**(用了"未来"信息当特征)→ 离线 AUC 虚高,上线翻车。

## 5. 重排(Re-rank)—— 多样性 + 业务规则

**解决什么**:精排只管"准",但全是同一类型会腻。重排负责**体验和规则**。

**本项目做成可插拔策略**(`Reranker` 接口 + `RerankRouter` 按分层 A/B 的 rerank 层选择):
| 策略 | name | 原理 |
|---|---|---|
| **多样性打散** | diversity | 同类目在结果中不超过 `maxSameCategory` 个;不足条数时放宽补齐 |
| **MMR** | mmr | `next = argmax[λ·相关性 − (1−λ)·与已选最大相似]`;相似度有向量取余弦、无向量退化"同类目=1",λ 默认 0.7 |
| **DPP** | dpp | **行列式点过程**(Chen 2018 fast-greedy MAP,Cholesky 增量):用核矩阵行列式(子集"体积")联合建模质量×多样性 |
| **不打散** | none | 严格按相关性截断(对照组) |

**坑:**
- 打散过度会牺牲相关性 → `maxSameCategory`/λ 可配置、可做 A/B。
- MMR 算相似度依赖物品向量,语料未全量灌向量时务必有"同类目"退化路径,否则没向量的物品多样性失效。
- 过滤已看/去重(行为数据已就绪)仍是待接入项。

## 6. 特征体系 —— 在线/离线特征

**特征三类:**
- 用户特征:活跃度、偏好类目分布、历史 CTR…
- 物品特征:热度、类目、近 N 天 CTR、上架时长…
- 交叉特征:用户对该类目的历史点击率(最有用!)…

**在线/离线:**
- 离线批算 → 写入 Redis(`feat:user:*` / `feat:item:*`)→ 在线直接读。
- 进阶:Flink 实时更新部分特征(如实时点击数),体现"实时性"。

## 7. 行为反馈闭环 —— 系统越用越准

```
用户点击/播放 → POST /api/behavior → Kafka
   → 更新在线特征(实时 CTR)
   → 累积训练样本 → 定期重训模型
   → 更新 user_embedding / i2i 倒排
```
**这是推荐系统区别于普通搜索的灵魂**:闭环让系统自我进化。学习项目里哪怕做成"每天跑一次离线作业"也完整体现了原理。

## 8. 冷启动(Cold Start)

| 场景 | 策略 |
|---|---|
| **新用户**(无行为) | `ColdStartDetector` 判冷 → 编排覆盖召回为 `{COLD,HOT,TAG}` + 强多样性重排(`maxSameCategory=1`)做兴趣探索 |
| **兴趣 onboarding** | `POST /api/user/{id}/interests` 写 `profile.categories` → TAG 召回立即生效,快速带出冷区 |
| **新物品**(无行为) | 靠内容向量(embedding)进入向量召回;给曝光探索机会 |
| **新系统**(无数据) | 先灌 MovieLens 历史数据 bootstrap |

**冷启动判定(M3 实现)**:冷 = **正反馈数 < `recsys.cold-start.min-behaviors`(默认 5)且无 user_embedding**。命中时 bucket 打 `cold;` 前缀。

**坑(实测)**:判定**只能数真实互动**(`action IN CLICK/LIKE/PLAY/RATING`),**必须排除 IMPRESSION**——曝光埋点每次推荐都写 IMPRESSION,若计入,用户请求几次后行为数就 ≥ 阈值"假性脱冷",冷启动策略失效。这与 i2i/u2u/swing 取种子的口径一致。

> 冷启动是面试高频题,也是这个项目的亮点之一,务必在文档里讲清楚。
> 已知局限:冷用户即使 onboarding 了类目,TAG 的偏好物品仍常被 COLD/HOT 的高热度跨类目大片在融合分上压过,onboarding 可见效果有限 → 后续可对"冷启动 + 有画像"时给 TAG 加权。

## 9. 评估指标 —— 怎么证明"推得准"

**离线指标(用历史数据评估):**
- 排序模型:**AUC**(最常用)、LogLoss。
- 召回:**Recall@K**、命中率。
- 整体:**NDCG@K**(考虑排序位置)、MAP。

**在线指标(真实/模拟反馈):**
- **CTR**(点击率)、人均点击数、覆盖率、多样性。

**做法:** 划分训练/测试集(按时间切,避免穿越),离线先看 AUC/Recall@K;前端做个简单"点击模拟"看 CTR 变化。

## 10. A/B 实验(生产必备,M3 实现:分层实验)

**分层实验**:recall / rank / rerank 三层各是独立实验,各有自己的 salt 和 variants,**互相正交、可任意组合**(共 2×2×2=8 种组合)。配置见 `recsys.experiment.layers.*`。
- **分桶**:每层 `floorMod(hash(userId|salt), 100)` 按权重命中 variant;**确定性**(同 userId 永远同桶),故结果缓存按 userId 即可,无需按 bucket 拆。`enabled=false` 时每层取第一个 variant 作基线。
- **各层切什么**:recall 层切启用的召回路集合(`channels`);rank 层切排序策略(v1/onnx)+ 融合权重;rerank 层切重排策略(diversity/mmr)+ 参数。
- **埋点**:`ExposureLogger` 每次推荐异步写 IMPRESSION 行,`bucket` 记紧凑分层标签(如 `recall:plus;rank:onnx;rerank:mmr`),冷启动加 `cold;` 前缀 → 后续 `GROUP BY bucket` 算分桶 CTR(点击来自 behavior 真实埋点)。
- 生产脚手架要求:实验配置可热更(Nacos,预留)、埋点带实验组标记(已落 `user_behavior.bucket`)、指标看板可分组对比。

## 12. 生产化关注点(本脚手架必须预留)

| 关注点 | 做法 |
|---|---|
| 可观测 | Actuator 健康检查 + 结构化日志 + traceId 全链路;预留 Prometheus/Grafana 接口 |
| 降级/兜底 | 召回每路独立可降级、热门永远兜底;embedding 超时降级本地;排序模型加载失败回退规则打分 |
| 样本回流 | impression/click 全量埋点 → Kafka → 落样本表,保证可重训(闭环的数据基础) |
| 特征一致性 | 离线/在线同一套特征定义,杜绝穿越(见 §4 坑)——生产翻车头号原因 |
| 配置中心 | 召回配额、排序策略、AB 分流全部 Nacos 配置化,可热更不停服 |
| 灰度/回滚 | 模型文件版本化(model 字段 + 版本目录),可快速回滚到上一版 |
| 幂等/限流 | 行为上报幂等、embedding API 限流重试退避 |
| 分布式主键 | **Snowflake**(64bit:`1 符号 + 41 时间戳 + 10 workerId + 12 序列号`,时间有序、去中心化、8 字节)替代单库自增/IDENTITY(分库后跨库会撞主键);时钟回拨需保护,**别拿低位取模分片**(序列号低位低 QPS 恒 0 → 倾斜)。详见 `docs/05 §8.0`。 |
| 分库分表 | **ShardingSphere-JDBC** 按业务键水平拆分(广告表按 `advertiser_id`/`ad_id` 分到 ds_0/ds_1);**主键 ≠ 分片键**(主键 Snowflake、分片键选业务维度);含 pgvector 的库走**双数据源**——只把分片表交给 ShardingSphere、向量 ANN 与 `<=>` 留主数据源直查(否则解析器报错、且向量 ANN 不可分片)。详见 `docs/05 §8`。 |

## 11. 一图总结各技术点落在哪层

```
召回   → 12 路(VECTOR/I2I/HOT/TAG/U2U/SWING/SEMANTIC/COLD/TWO_TOWER/GENERATIVE/LEXICAL/TIGER) / 向量化 / pgvector / 逐路归一或 RRF
排序   → 特征体系 / CTR模型(LR/GBDT/DeepFM/DCN/MMoE/PLE/DIN/DIEN/SIM,9 策略) / ESMM/PAL / 离线训练在线服务
重排   → 可插拔策略(diversity / mmr / dpp / none)
融合   → (归一召回 + 校准排序 + FTRL + bandit) × channel-boost × 热度去偏
闭环   → 行为采集 / 曝光埋点(带 bucket + position) / 反馈 / 重训
实验   → 分层 A/B(recall×rank×rerank×ad 正交)/ bucket 落库算分桶 CTR + 显著性
评估   → AUC / Recall@K / NDCG / CTR / A/B / 严格无泄漏
广告   → 召回→相关性→定向→竞价(GSP/VCG/List-wise)→GSP 计费→oCPC→延迟反馈纠偏
平台   → 微服务缝(gRPC,默认单体)/ 两级零信任 / 分库分表 / 可观测
```
> 每一行的深度拆解见 [`docs/skills/`](skills/README.md)。
