# 推荐系统 · 架构设计

> 本文是**并行开发的契约**。各模块按这里定义的接口/数据模型独立实现,互不阻塞。

## 1. 推荐系统是怎么工作的(一句话原理)

> 从**海量物品**里,分层漏斗式地筛出**这个用户此刻最可能感兴趣的几个**,展示给他;再把他的真实反馈收回来,让系统越用越准。

经典四段漏斗:

```
百万物品 ──召回──> 几千 ──粗排──> 几百 ──精排──> 几十 ──重排──> 展示给用户
          Recall          Pre-rank        Rank           Re-rank
        (快/广/粗)        (轻量过滤)     (准/重/CTR模型)  (多样性/业务规则)
```

> 学习项目里**粗排可省略**(召回结果不多),先做 **召回 → 精排 → 重排** 三段。

## 2. 系统总体架构

```
                         ┌──────────────────────────────────────────┐
                         │              用户 / 演示前端                │
                         └───────────────────┬──────────────────────┘
                                             │ HTTP
                         ┌───────────────────▼──────────────────────┐
                         │            API Gateway (网关)              │
                         └───────────────────┬──────────────────────┘
                                             │
  ┌──────────────────────────────────────────────────────────────────────────┐
  │                        在线服务链路 (低延迟, 同步)                            │
  │                                                                            │
  │   ┌──────────────┐    ┌───────────┐   ┌──────────┐   ┌──────────┐         │
  │   │ rec-engine   │──> │  recall   │──>│  rank    │──>│  rerank  │──> 结果   │
  │   │ (编排/兜底)   │    │ 多路召回   │   │ 特征+模型 │   │多样性/规则│         │
  │   └──────┬───────┘    └─────┬─────┘   └────┬─────┘   └──────────┘         │
  │          │                  │              │                               │
  └──────────┼──────────────────┼──────────────┼───────────────────────────────┘
             │                  │              │
        ┌────▼──────┐    ┌──────▼──────┐  ┌────▼─────┐   ┌──────────────┐
        │  Redis    │    │ pgvector    │  │  Redis   │   │ embedding-svc │
        │(结果缓存) │    │(向量检索)    │  │(在线特征)│   │(Gemini API)   │
        └───────────┘    └─────────────┘  └──────────┘   └──────┬───────┘
                                                                 │
  ┌──────────────────────────────────────────────────────────────────────────┐
  │                   离线 / 近线链路 (异步, 不影响在线延迟)                       │
  │                                                                            │
  │  行为日志 ─Kafka─> 特征计算 ──> 训练样本 ──> LightGBM训练 ──> 导出ONNX        │
  │  物品入库 ──────> embedding ──> 写入 pgvector                               │
  │  行为日志 ──────> ItemCF/Swing 批算 ──> 写入 Redis(i2i 倒排)                │
  └──────────────────────────────────────────────────────────────────────────┘
```

**两条链路的核心区别(重点理解):**
- **在线链路**:用户请求来了实时算,要求毫秒级,只做"查"和"轻打分"。
- **离线/近线链路**:提前把重活算好(向量、协同过滤倒排、模型),把结果"喂"到在线存储里。

## 3. 模块划分(Maven 多模块单仓)

```
recsys/                          # 父 POM
├── recsys-common/               # 公共:DTO、常量、工具、API 契约接口
├── recsys-gateway/              # 网关
├── recsys-rec-engine/           # 推荐编排服务(对外主入口)
├── recsys-recall/               # 召回服务(向量/CF/热门/标签)
├── recsys-rank/                 # 排序服务(特征装配 + ONNX 打分)
├── recsys-feature/              # 特征服务(在线特征读写 Redis)
├── recsys-embedding/            # 向量化服务(封装 Gemini,可降级)
├── recsys-content/              # 物品/内容服务(物品元数据 CRUD)
├── recsys-user/                 # 用户画像服务(对接你已有画像项目)
├── recsys-behavior/             # 行为采集服务(埋点上报 → Kafka/库)
├── recsys-offline/              # 离线作业(数据导入、CF批算、样本生成)
│   └── train/                   # Python 训练脚本(LightGBM → ONNX)
└── recsys-web/                  # 演示前端
```

> 学习阶段可把 `recall/rank/feature/embedding` 先做成**同一服务里的模块**(单体起步),跑通后再拆成独立微服务。模块边界已划好,拆分零成本。

## 4. 数据模型

### 4.1 PostgreSQL 表

```sql
-- 物品表
CREATE TABLE item (
  item_id      BIGINT PRIMARY KEY,
  title        TEXT,
  category     TEXT,              -- 类型/类目(电影:genres)
  tags         TEXT[],            -- 标签
  description  TEXT,
  popularity   DOUBLE PRECISION DEFAULT 0,  -- 热度(用于热门召回/冷启动)
  created_at   TIMESTAMP DEFAULT now()
);

-- 物品向量表(pgvector)
CREATE EXTENSION IF NOT EXISTS vector;
CREATE TABLE item_embedding (
  item_id   BIGINT PRIMARY KEY REFERENCES item(item_id),
  embedding vector(768),          -- 维度与 EmbeddingClient 配置一致
  model     TEXT                  -- 记录用哪个模型生成,便于换模型
);
CREATE INDEX ON item_embedding USING hnsw (embedding vector_cosine_ops);

-- 用户表 / 用户画像
CREATE TABLE app_user (
  user_id    BIGINT PRIMARY KEY,
  profile    JSONB,               -- 画像:偏好类目、标签权重等
  updated_at TIMESTAMP DEFAULT now()
);

-- 用户向量(由历史行为物品向量聚合而来,用于向量召回)
CREATE TABLE user_embedding (
  user_id   BIGINT PRIMARY KEY,
  embedding vector(768)
);

-- 行为表(点击/播放/评分/收藏...)
CREATE TABLE user_behavior (
  id         BIGSERIAL PRIMARY KEY,
  user_id    BIGINT,
  item_id    BIGINT,
  action     TEXT,                -- impression/click/like/play/rating
  value      DOUBLE PRECISION,    -- 评分值/时长等
  ts         TIMESTAMP DEFAULT now()
);
```

### 4.2 Redis Key 规范

| Key 模式 | 类型 | 内容 | 用途 |
|---|---|---|---|
| `feat:user:{userId}` | Hash | 用户在线特征 | 排序取特征 |
| `feat:item:{itemId}` | Hash | 物品在线特征 | 排序取特征 |
| `recall:hot` | ZSet | 全局热门(score=热度) | 热门召回/冷启动 |
| `i2i:{itemId}` | ZSet | 相似物品(score=相似度) | ItemCF 召回 |
| `cache:rec:{userId}` | String | 推荐结果(短 TTL) | 结果缓存 |
| `emb:cache:{hash}` | String | 文本→向量缓存 | 省 API 调用 |

## 5. 服务间 API 契约(并行开发的关键!)

> 每个 track 只要遵守这里的输入/输出,就能独立开发、独立 mock 联调。

### 5.1 主入口:获取推荐
```
GET /api/recommend?userId={id}&size={n}&scene={feed}

Response:
{
  "userId": 1,
  "scene": "feed",
  "items": [
    { "itemId": 100, "score": 0.92, "recallFrom": ["vector","i2i"], "reason": "和你看过的X相似" }
  ],
  "traceId": "..."
}
```

### 5.2 召回服务(recsys-recall)
```java
interface RecallService {
    // 多路召回合并后的候选集
    List<RecallItem> recall(RecallContext ctx);   // ctx: userId, size, scene
}
record RecallItem(long itemId, double recallScore, String channel) {}
// channel ∈ {vector, i2i, hot, tag}
```

### 5.3 向量化服务(recsys-embedding)
```java
interface EmbeddingClient {
    float[] embedText(String text);
    float[] embedImage(byte[] image);      // 多模态(可选)
    int dimension();                       // 固定维度,如 768
}
// 实现: GeminiEmbeddingClient(默认) / LocalBgeEmbeddingClient(降级)
```

### 5.4 排序服务(recsys-rank)
```java
interface RankService {
    // 输入候选,输出带分数的排序结果
    List<RankedItem> rank(long userId, List<Long> candidateItemIds, String scene);
}
record RankedItem(long itemId, double score, Map<String,Double> featureSnapshot) {}
```

### 5.5 特征服务(recsys-feature)
```java
interface FeatureService {
    Map<String,Double> userFeatures(long userId);
    Map<String,Double> itemFeatures(long itemId);
    // 交叉特征在 rank 内部组合
}
```

### 5.6 行为采集(recsys-behavior)
```
POST /api/behavior
{ "userId":1, "itemId":100, "action":"click", "value":1.0, "ts":... }
→ 写入 Kafka(或直接入库)→ 离线消费
```

## 6. 一次推荐请求的完整时序

```
1. 前端 GET /api/recommend?userId=1&size=10
2. rec-engine 查 cache:rec:1,命中直接返回(可选)
3. recall:并行四路召回
   - vector: 取 user_embedding → pgvector 找最近 N 个物品
   - i2i:   取用户近期物品 → Redis i2i:{itemId} 取相似物品
   - hot:    Redis recall:hot 取热门(兜底/冷启动)
   - tag:    按用户画像偏好类目取物品
   合并去重 → ~几百候选
4. rank:对候选批量取特征(feature)→ ONNX 模型打分 → 排序
5. rerank:类目打散(多样性)+ 业务规则(去重/已看过过滤/置顶)
6. 截断 top size,组装 reason,返回;写结果缓存
7. 前端展示;用户点击 → POST /api/behavior → 进入反馈闭环
```

## 7. 配置与环境

- 维度、模型名、各召回路数量、Redis/PG 连接 → 全部走配置(`application.yml` / Nacos)。
- `GEMINI_API_KEY` 走环境变量,**不进 git**(`.gitignore` 加 `.env`)。
- `docker-compose.yml` 提供:postgres(pgvector 镜像)、redis、(可选)kafka、nacos。
