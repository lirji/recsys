# 10 请求链路与公共参数：一次请求从浏览器到 SpiceDB 都带了什么

> 回答一个高频问题：**"项目发请求时都携带什么公共参数？"** 结论先行：recsys 没有传统意义的
> "公共 query 参数"——**身份是唯一贯穿全链路的公共携带物，且每一跳换一种凭证形态**；租户与角色
> 永远内嵌在凭证里而不是独立参数（"只信 token、拒绝 header 覆盖"的零信任设计）。追踪上下文
> （`traceparent`）由框架在所有跳自动透传。
>
> 相关文档：安全体系总述见 `02-架构设计.md`；细粒度判权接入见 `09-权限接入auth-platform.md`。

## 0. 链路总览

```
浏览器(console SPA)
   │  Authorization: Bearer <终端 token>          ← 唯一公共参数(oidc=Casdoor RS256 / legacy=自签 HS256)
   ▼
网关 recsys-gateway :8080  ──验签──┐
   │  ✂ 剥离 Authorization         │ 零信任重签:终端令牌不落下游
   │  ＋ X-Internal-Auth: <内部令牌>(HMAC,TTL 300s, payload=subject\roles\scene\exp)
   │  ＋ traceparent(W3C,自动透传)
   ▼
下游 servlet 服务(rec-engine/advertiser/behavior/console-api)
   │  gRPC metadata: <同一枚内部令牌>(出站拦截器注入,服务端校验→CALLER_SUBJECT)
   ▼
内部 gRPC 服务(ad-serving/content-service/user-service)

advertiser(细粒度判权,docs/09)
   │  Authorization: Bearer <authz.client.token>(service credential;判权信息在请求体)
   ▼
auth-platform-server :8210
   │  Authorization: Bearer <SpiceDB 预共享密钥>(recsys_dev_key)
   ▼
SpiceDB :8544
```

## 1. 浏览器 → 网关（南北向）

axios 实例的请求拦截器统一注入（`console/src/api/client.ts`），**唯一公共参数就是身份**：

| 携带物 | 内容 | 来源 |
|---|---|---|
| `Authorization: Bearer <token>` | **oidc 模式**：Casdoor access_token（RS256 JWT，约 8-9KB，claim 含 `sub`(全局用户ID)/`owner`(=org,租户)/`groups`(角色来源)/`name`）。**legacy 模式**：网关 `/api/auth/login` 自签 HS256 JWT（`roles` claim） | `resolveToken()` 按 `VITE_AUTH_MODE` 分派（`console/src/auth/oidc.ts`）：oidc 读 oidc-client-ts 的 sessionStorage,legacy 读 localStorage——oidc 下残留的旧 legacy token **绝不误发** |

- 没登录就不带 Authorization（公开只读接口照发：GET `/api/recommend|search|feed|query|console/**` 网关 permitAll）。
- **刻意不带的**：租户/角色不走 header 或 query（全在验签后的 token claim 里，防伪造）；无 Cookie/session（全程 STATELESS）；无 API key。
- **易混淆的业务参数**：控制台顶栏的 `userId`/`scene`（"被推荐的用户"，`useGlobalUser`）按接口以 query 传给 `/api/recommend?userId=...&scene=...`——那是**业务入参**不是登录身份，与 `useAuth`（登录身份）刻意分离。

## 2. 网关 → 下游服务（东西向，零信任重签）

`IdentityPropagationFilter`（`recsys-gateway/.../security/`）在这一跳做"换钥匙"：

| 携带物 | 内容 |
|---|---|
| `X-Internal-Auth: <内部令牌>` | 网关验完终端 JWT 后**重签**的短时令牌（纯 JDK HMAC-SHA256，格式 `b64url(subject\nroles\nscene\nexp).b64url(hmac)`，TTL=`recsys.security.token-ttl-seconds` 默认 300s）。subject 在 oidc 模式=Casdoor `sub`(UUID)、legacy 模式=username——**这也是 SpiceDB 判权主体的来源** |
| ~~`Authorization`~~ | **被剥离**：终端令牌不落下游；客户端伪造的 `X-Internal-Auth` 也被剥掉再重注（即使某下游被直连也无法伪造身份） |
| `traceparent` | W3C Trace Context，micrometer-tracing/OTel 自动传播（P2 可观测）；下游日志 MDC 的 `trace=/user=/scene=` 由此解出 |

下游 servlet 服务由 `InternalTokenAuthFilter`（recsys-platform）解令牌落 SecurityContext，供 `@PreAuthorize` 角色判权与 advertiser 的归属判权（docs/09）取主体。

## 3. 服务间 gRPC（rec-engine → ad-serving / content-service / user-service）

同一枚**内部令牌**放在 gRPC metadata：`InternalAuthGrpcClientInterceptor` 出站注入，服务端
`InternalAuthGrpcServerInterceptor` 校验（`required=true` 时无效即 `UNAUTHENTICATED`），subject 放进
`CALLER_SUBJECT` context 供计费归因/审计。traceparent 同样透传。

## 4. 细粒度判权链路（advertiser → 判权服务 → SpiceDB，docs/09）

| 跳 | 携带物 |
|---|---|
| advertiser → auth-platform-server(:8210) | `Authorization: Bearer <authz.client.token>`（service credential，env `AUTHZ_CLIENT_TOKEN`；dev 未开判权服务安全时为空不带）。判权信息在**请求体**：`{subject(=内部令牌 subject), permission, resource, consistency}` |
| auth-platform-server → SpiceDB(:8544) | `Authorization: Bearer <预共享密钥>`（env `RECSYS_SPICEDB_KEY`，dev 默认 `recsys_dev_key`） |

## 5. 各跳凭证形态速查

| 跳 | 凭证 | 算法/形态 | 有效期 | 谁签发 |
|---|---|---|---|---|
| 浏览器→网关(oidc) | Casdoor access_token | RS256 JWT(JWKS 验签) | 168h(dev) | Casdoor |
| 浏览器→网关(legacy) | 自签终端 JWT | HS256(edge-secret) | 3600s | 网关 `/api/auth/login` |
| 网关→下游 HTTP | 内部令牌(`X-Internal-Auth`) | HMAC-SHA256(internal-secret) | 300s | 网关 IdentityPropagationFilter |
| 服务间 gRPC | 同上内部令牌(metadata) | 同上 | 300s | 出站拦截器 |
| advertiser→判权服务 | service token | Bearer 常量比较 | 静态 | 运维配置 |
| 判权服务→SpiceDB | 预共享密钥 | Bearer | 静态 | 运维配置 |

**设计要点**（为什么这么设计）：
1. **每跳换钥匙**：终端 token 权力大（可对网关发任何请求），不下沉；内部令牌短时、窄用途，泄漏面小。
2. **租户/角色在凭证里**：`owner`/`groups` 由验签保真，接受 header/query 覆盖即等于放弃边界。
3. **公共参数最小化**：除身份与追踪头外无全局注入——需要什么参数由各接口显式声明，避免"隐式全局态"漂移。
