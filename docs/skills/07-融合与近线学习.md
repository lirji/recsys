# 融合与近线学习

> **解决什么**:召回分和排序分量纲不同,要融合成一个最终分;并叠加近线/探索信号(FTRL、bandit)、做校准与去偏。本项目的融合公式比 CLAUDE.md 描述的更丰富。

## 1. 融合公式全展开(`RecommendOrchestrator` step 5)

```
base       = recallWeight·rNorm + rankWeight·rankScore + ftrlTerm + banditTerm      # 加性
finalScore = base · channelBoost · persBoost · popDebias                            # 乘性
```

| 项 | 含义 |
|---|---|
| `rNorm` | 召回分 / 全局最大值(在逐路归一之上再全局归一) |
| `rankScore` | 排序分,**经保序回归校准**(`fusion.calibration`,默认开,读 `rec:calib:{model}`) |
| `ftrlTerm` | `ftrlWeight · pFtrl`,近线 FTRL 分(默认开 weight 0.5,无模型时 0) |
| `banditTerm` | `banditWeight · bandit.score(x)`,R7 contextual bandit 探索加成(默认**关** weight 0.3) |
| `channelBoost` | 逐通道加成,**多命中取 max**;默认 `{TAG:1.5}`,搜索 `{SEMANTIC:2.0,LEXICAL:2.0,TAG:1.8}` |
| `persBoost` | `1 + persW·max(0,cos)`,**仅搜索**的个性化乘子(冷用户无向量 → 无影响) |
| `popDebias` | `1/(1+item_pop_norm)^beta`,热度去偏(默认开 beta 0.5) |

**加性 vs 乘性的设计**:FTRL/bandit 是"学习/探索的边际修正",做加性;channel-boost/热度去偏/个性化是"信号权重的缩放",做乘性。

**搜索 vs 推荐权重**:query 驱动 → `recall-weight=2.0/rank-weight=0.5`(相关性主导);否则 `1.0/1.0`。rank 层实验可覆盖。

## 2. 逐通道加成(channel-boost)

为什么需要:TAG 携带**实时类目偏好**(`rt:user`),但热度上 HOT/CF 会淹没它。乘以 `TAG:1.5` 让"用户当下正在互动的兴趣"浮现。搜索场景抬 SEMANTIC/LEXICAL(相关性主导)。`boostFor` 取 item 命中的所有通道里的最大加成。

## 3. FTRL 近线学习(`FtrlScorer`)

- 加载 Redis `ftrl:weights`,`sigmoid(bias + Σ w[feat])`(特征哈希 `FtrlHashing`),60s 刷新,缺失 → 0。
- 离线/近线 `train-ftrl` 作业:正反馈 + 负采样,FTRL-Proximal 在线学习(纯函数 `FtrlProximal`),写 `ftrl:weights` + `ftrl:state`(`--incremental` warm-start)。
- **FTRL-Proximal 特点**:L1 稀疏 + per-coordinate 学习率,适合高维稀疏 CTR、增量更新——比批训练更快响应新数据。

## 4. Contextual Bandit 探索(R7,`BanditScorer`)

- 加载 Redis `bandit:model`,预计算 `A⁻¹`、`θ̂`,60s 刷新。`x` 从排序 `featureSnapshot` 按模型 `order` 取。
- 两种模式:
  - **LinUCB**:`θ̂ᵀx + α√(xᵀA⁻¹x)`(均值 + 置信上界,探索不确定的方向)。
  - **Thompson**:每请求采样一次 `θ̃`,出 `θ̃ᵀx`(后验采样)。
- 离线 `bandit-stats` 作业:从曝光(上下文)+ 正反馈(reward)经 `FeatureAssembler`(与在线同源)重建岭回归充分统计 `A=λI+Σxxᵀ`、`b=Σr·x`(纯数学核 `LinUcbModel`,Cholesky/Gauss-Jordan 求逆)。
- 默认关;缺 `bandit:model` 退 0;`recsys:tuning` 热开关。

## 5. 校准与热度去偏

- **保序回归校准**(`rec-calibrate` → `rec:calib:{model}`):PAVA 把模型原始分映射到真实点击率单调曲线,让 `rankScore` 可解释、跨模型可比。
- **热度去偏**:`1/(1+pop)^beta` 压制头部热门,提升长尾曝光(读 `featureSnapshot` 的 `item_pop_norm`)。

## 6. 热更新(`DynamicTuningService`)

所有旋钮(recall/rank 权重、各 boost、FTRL/bandit 开关与权重、beta)存 Redis Hash `recsys:tuning`,30s 缓存,**覆盖 yml 默认值**(无需 `@RefreshScope`),Redis 挂 → 退静态默认。比 Nacos 更轻量的运行时调参。

## 7. 踩坑

1. **加性项量纲**:FTRL/bandit 输出要与归一召回/排序分同量级,否则权重难调(用 `tune-fusion` 网格搜索多任务权重,bandit/FTRL 权重靠 A/B)。
2. **bandit 默认关**:开之前确认 `bandit:model` 已由 `bandit-stats` 物化,否则加成恒 0(无害但无效)。
3. **channel-boost 取 max 不是求和**:一个 item 命中多路只取最大加成,避免多路命中被过度放大。

## 8. 面试要点

- **召回分排序分怎么融合**:归一化后加权 + 乘性 boost;为什么归一(量纲对齐)。
- **LinUCB vs Thompson Sampling**:前者置信上界(乐观面对不确定),后者后验采样;都做探索-利用。
- **FTRL 为什么适合 CTR**:L1 稀疏 + per-coordinate 学习率 + 增量更新。
- **保序回归校准**:PAVA 单调映射到真实 CTR,解决模型分不是概率的问题。
- **热更新**:Redis 覆盖层做运行时调参,不重启不改代码。
