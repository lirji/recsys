# 工程化 · 容器编排与可观测

> **解决什么**:让这套多服务系统"一键本地跑通、可容器化、可上 k8s、可观测、可 CI 守门"。这些是脚手架"面向生产"承诺的兑现。

## 1. 本地一键起(`scripts/dev-local.sh`,全链路容器化)

**所有容器编排归拢在 `docker/` 目录**(`docker/docker-compose.yml` 单一入口 + `docker/Dockerfile` + `docker/monitoring/`)。`scripts/dev-local.sh` 是它的一键封装,把「基础设施 + 8 app + 前端」**全部以容器方式**起起来:
- `up`:`docker compose --profile apps --profile console up -d`(缺镜像自动构建)+ 网关健康等待。启动顺序交给 compose `depends_on`+healthcheck(取代旧的手工 staggered)。
- `rebuild <svc>`:改了后端代码 → 重建该服务镜像并重启(容器化下改代码需重建)。
- `frontend`:前端热开发,宿主机跑 Vite → 容器网关 `$GATEWAY_PORT`。
- `build`/`status`/`down`/`restart`/`logs <svc>`/`infra`/`obs` 子命令。
- **固化在 compose、无需手敲的坑**:①各服务 `RECSYS_SECURITY_ENABLED=false`(否则 `/api/advertiser` 等 401);②rec-engine/advertiser/ad-serving 的 `PG_JDBC_URL`+`PG_DS1_JDBC_URL` 指向 `postgres` 容器(ShardingSphere 主/次数据源);③postgres 首启自动建分库 ds_1(`recsys-offline/sql/04b_ds1_bootstrap.sql`,随 schema 一起挂进 initdb),否则上面三个服务的 ds_1 池初始化即挂。
- 端口冲突时覆盖写 `scripts/dev-local.env`(**只改容器对宿主暴露的端口**;容器间用服务名+标准内部端口通信,不受影响)。本机常用:网关 9080、pg 5456、redis 6381、nacos 8858、前端 9095。

## 2. docker compose profiles(`docker/docker-compose.yml`)

从仓库根 `docker compose -f docker/docker-compose.yml ...`,或 `cd docker` 后直接跑。

| profile | 起什么 |
|---|---|
| default | postgres(pgvector/pg16 :5432,首启自动建 schema + 分库 ds_1)+ redis :6379 |
| full | + kafka(KRaft :9092)+ nacos(:8848/:9848) |
| obs | + prometheus :9090 + **tempo**(OTLP :4318)+ alertmanager :9093 + **webhook-logger** :5001 + grafana :3001 |
| console | nginx 托管前端 :8095 + 反代 `/api`→网关(容器网络内直连 `gateway:8080`) |
| **apps** | 全微服务容器化:gateway/rec-engine/ad-serving(8085/9095)/behavior/advertiser/console-api/content(8086/9096)/user(8087/9097)+ nacos —— `NACOS_DISCOVERY=true`、网关默认 lb://、rec-engine 接 gRPC ad/content/user |

对宿主暴露的端口全部可用环境变量覆盖(`GATEWAY_PORT`/`PG_PORT`/`REDIS_PORT`/`NACOS_PORT`/…,见 `docker/.env.example`)。

## 3. 容器化 & k8s

- **一个参数化 Dockerfile(`docker/Dockerfile`)** 服务所有 Spring Boot app:`--build-arg MODULE/PORT`,pom-layer 缓存(21 个 pom),`-pl MODULE -am`,运行时 `eclipse-temurin:21-jre` 非 root(1001)。构建上下文=仓库根(`context: ..`),`.dockerignore` 仍在仓库根。recsys-streaming 除外(需 `--add-opens`)。
- **k8s**(`deploy/k8s/`,kustomize base + dev/prod overlay):三探针,**readiness = `/actuator/health/readiness`**(含 `ModelReadinessHealthIndicator`——模型策略但 ONNX 没加载 → readiness DOWN,k8s 不导流)。rec-engine startupProbe(failureThreshold 30);env `AD_SERVING_MODE=in-process`(kill-switch,直到 money-chain-gate 绿)、`CONTENT/USER_SERVING_MODE=grpc`。

## 4. 可观测(`docker/monitoring/`)

- **Prometheus**:8 个 host target;指标 `recsys_recommend_duration_seconds_*`(P50/P95/P99)、`recsys_ad_duration_seconds_*`、缓存命中率、模型回退率、分桶曝光/点击。
- **告警规则**(`alert-rules.yml`):推荐 SLO 99.5%/P99<200ms 多窗口 burn-rate(14.4×/3×);RecEngineDown、RecommendErrorBudgetBurn Fast/Slow、RankModelFallbackHigh、CacheHitRatioLow、EmptyRecallSpike;广告 SLI(P99<300ms、fill、CTR、eCPM、invalid-click)+ AdServingLatencyP99High、AdFillRatioLow、AdInvalidClickSpike。
- **Alertmanager** → `webhook-logger:5001`(Slack/钉钉/飞书 demo)。
- **Grafana**(:3001):datasource + 2 看板(`recsys-online.json`、`recsys-ads.json`)。
- **Tempo**:OTLP 分布式追踪后端(P2)。
- **console-api BFF 观测**:`SystemHealthService`(并行探 8 app + 2 passive job)、`SystemMetricsService`(PromQL,never throws)、`SystemOverviewService`(静态模块图),供前端总览/诊断/告警页。

## 5. 熔断(Resilience4j)

重外呼加熔断快速失败到降级:
- `GeminiEmbeddingClient`(gemini-embedding)。
- `VectorRecaller` 的 pgvector ANN(pgvector-recall,pg 宕 → VECTOR 路快速返回空、其它路兜底)。
- gRPC 客户端每 RPC 熔断(见 [16](16-微服务拆分与gRPC.md))。
- 配置在 rec-engine `application.yml` 的 `resilience4j.circuitbreaker`。

| 机制 | 粒度 | 存储 |
|---|---|---|

## 8. 踩坑

1. **编排文件都在 `docker/`**——compose 相对路径按 `docker/` 解析(`context: ..`=仓库根、`../recsys-offline/sql`、`./monitoring`);`.dockerignore` 留仓库根(=构建上下文根)。
2. **网关不关 security 本地 401**——`RECSYS_SECURITY_ENABLED=false`(compose 已注入)。
3. **分库 ds_1 未建则分片服务启动即挂**——`sql/04` 的 CREATE DATABASE 是手动步骤,容器由 `sql/04b_ds1_bootstrap.sql`(随 schema 挂进 initdb)首启自动建。
4. **全容器化下改代码不重建镜像不生效**——`dev-local.sh rebuild <svc>`;要即改即跑用宿主机 mvn 模式(见 08)。
5. **端口覆盖只改宿主暴露端口**——容器间用服务名+标准内部端口,改 `*_PORT` 不影响互通。
6. **重训模型后不 clean install**——ONNX 陈旧,`target/classes` 留旧资源。
7. **readiness 探针含模型就绪**——ONNX 没加载会 DOWN,别用 liveness 代替。

## 9. 面试要点

- **readiness vs liveness**:readiness 含 `ModelReadinessHealthIndicator`,模型没就绪不导流;liveness 只看进程活着。
- **money-chain-gate**:计费相关测试单独 gate,必过才允许拆 ad-serving(kill-switch `AD_SERVING_MODE=in-process`)。
- **多窗口 burn-rate 告警**:快窗(14.4×)+ 慢窗(3×)组合,兼顾灵敏与抗抖。
- **三层配置热更**:Nacos(结构化)/ Redis tuning(轻量旋钮)/ Redis exp(实验),都不重启。
- **一个 Dockerfile 参数化所有服务**:MODULE/PORT build-arg,pom-layer 缓存加速。
