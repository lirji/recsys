# 特征体系与在线/离线一致性

> **这是推荐系统上线翻车的头号原因**:在线拿到的特征和离线训练时算的特征不一致,或训练用了"未来"信息。本项目用**三大编码器契约 + as-of 时间点采样**系统性地防这两类问题。

## 1. 特征三类

- **用户特征**:活跃度、偏好类目分布、历史均分、类目数/占比…
- **物品特征**:热度、类目、均分、评分标准差、上架时长…
- **交叉特征**:用户对该类目的历史均分(`catavg:<cat>`)——**最有用**。

## 2. 在线特征存储 —— Redis `feat:*`

- 唯一实现 `RedisFeatureService`:读 `feat:user:{id}` / `feat:item:{id}` Hash。
- **批量读优化**:排序循环里 N 个候选一次 `executePipelined` HGETALL(避免 N+1),出错降级逐 key。缺数据 → 空 map(不崩)。
- 离线 `build-features` 作业物化:`ub RATING ⋈ item` 聚合 → `feat:*`(含 S2 的 std/catcnt/catratio)。
- **注意**:`feat:*` 是**排序稠密特征的唯一来源**(批/近线)。`rt:user:{id}`(实时类目)喂**召回 TAG**、`rt:user:seq:{id}`(实时序列)喂**DIN/DIEN 排序**,都不是 `feat:*`。

## 3. 三大在线/离线契约(与 [02 排序](02-排序-CTR与多任务模型全家桶.md) 同源)

模型的在线打分和离线训练**必须逐位一致**。三个编码器是契约的物理载体:

| 契约 | 类 | 编码 | 训练侧对应 |
|---|---|---|---|
| 稠密 | `FeatureAssembler` | 5 维(`FEATURE_ORDER`)或 8 维(`EXTENDED_ORDER`),按 schema `dense_order` 选 | gen-samples 输出同名列 |
| 稀疏 | `SparseFeatureEncoder` | `[floorMod(userId,B_u), floorMod(itemId,B_i), vocab[cat]]`(可选 +语义 ID → width 6) | rank_schema.json + category_vocab.json |
| 序列 | `SequenceEncoder` | 固定 len、右 padding、`floorMod(itemId,B_i)`,带掩码 | gen-samples-mt 的 as-of 序列 |

分桶取模的模数、category 词表、序列长度 padding 方向**逐位对齐训练器**——差一位,embedding 查错行,模型直接失效。golden test(`RankEncoderContractGoldenTest` 等)守护。

## 4. as-of 时间点采样 —— 杜绝数据穿越

**穿越(leakage)**:训练某条样本时用了该事件**之后**才发生的信息(如"用户总点击数"里已经包含了这次点击),离线 AUC 虚高、上线原形毕露。

本项目 `AsOfFeatureBuilder`(`recsys-offline`)是核心防线:
- 按 `ts` 顺序**流式**处理 ratings,`snapshot*` 只反映**已经 `apply()` 过的事件**。
- 每个样本的特征在**应用该事件之前**做快照(`GenSamplesJob`/`GenSamplesMtJob` 先 snapshot 再 apply)。
- 因此不会有未来/标签信息泄漏,**无需 Redis 的 `feat:*`**。
- 一个已知简化:log-max 归一化的分母用全语料常量(而非 as-of),影响可忽略。

**对照**:`--leaky` 改读全历史 `feat:*` 聚合(旧行为),仅为 A/B 对比"修复泄漏前/后"保留(需先 `build-features`)。

## 5. 在线离线一致性的三重保证

1. **同名同逻辑**:离线 `AsOfFeatureBuilder` 造的特征名 = 在线 `FeatureAssembler` 读的字段名。
2. **同编码**:三大编码器在线/离线共用同一分桶/词表(打进 classpath 的 schema)。
3. **同口径取种子**:召回取种子、冷启动判定、样本正例都用 `action IN (CLICK,LIKE,PLAY,RATING)`,统一排除 IMPRESSION。

## 6. 严格无泄漏评估(strict eval)

不止训练要防穿越,**评估**也要。行为派生作业(`item-cf`/`user-cf`/`swing`/`hot`/`user-embedding`/`bandit-stats`)支持 `--max-ts=<epochSeconds>`,只用切分点前的行为聚合(共享 `BehaviorQuery`,口径同 `AsOfFeatureBuilder`)。`run_strict_eval.sh` 一键:取 splitTs → `--max-ts` 重建无泄漏存储 → eval → 全量重建恢复。实测 pipeline 偏置仅 ±1~2%(见 [09 评估](09-离线评估与严格无泄漏.md))。

## 7. 踩坑

1. **IMPRESSION 混入正例/种子**——曝光埋点每次推荐都写 IMPRESSION,若计入,用户请求几次后行为数就假性达标,冷启动失效、CF 种子污染。**必须只数真实互动**。
2. **在线取不到的特征进了训练**——训练好上线却拿不到该特征,退化。
3. **编码器改了一处没改另一处**——在线加了一维稠密特征但没重训,或改了分桶数没同步 schema,静默错乱。改编码器务必重训 + 重跑 golden test。
4. **rt:user 与 feat 混淆**——实时类目/序列是召回/序列模型的输入,不是排序稠密特征来源。

## 8. 面试要点

- **特征穿越是什么、怎么防**:用未来信息当特征;as-of 快照(先取特征再应用事件)是标准解法。
- **在线离线一致性怎么保证**:同名同逻辑 + 同一套编码器/schema + 同口径取样;golden test 兜底。
- **交叉特征最有用**:用户×类目历史 CTR/均分,`catavg:<cat>`。
- **评估也要防穿越**:`--max-ts` 重建行为派生存储,否则 CF/vector 通道离线数值乐观。
