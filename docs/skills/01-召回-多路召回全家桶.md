# 召回 · 多路召回全家桶

> **解决什么**:从百万物品里快速、广撒网捞出几百个"可能相关"的候选。追求**快 + 全**,不追求准(准交给排序)。
> 本项目实现 **12 路推荐召回**(`recsys-recall`)+ 广告多路召回(`recsys-ad`),并行执行、逐路归一、可优雅降级。

## 1. 原理:为什么要"多路"

单一召回源都有盲区:向量召回懂语义但不懂协同,ItemCF 懂协同但对新物品无力,热门召回永远兜底但毫无个性化。**多路互补 + 融合**才能既广又稳。工业界的共识:召回决定推荐质量的上限,排序只能在召回给的候选里挑。

## 2. 本项目 12 路召回

`RecallChannel` 枚举(`recsys-common`)是唯一事实来源。每个 `*Recaller` 实现 `RecallService` 并声明 `channel()`,由 `MultiChannelRecallService` 统一编排。

| 通道 | 实现类 | 数据来源 | 离线灌数作业 | 降级 |
|---|---|---|---|---|
| **VECTOR** | `VectorRecaller` | `user_embedding` → pgvector ANN `item_embedding` | `user-embedding` / `backfill-embedding` | **唯一带熔断** `@CircuitBreaker(pgvector-recall)` → 空 |
| **I2I** | `I2iRecaller` | 用户近期正反馈 item → Redis `i2i:{itemId}` | `item-cf`(ItemCF+IUF) | 空 |
| **HOT** | `HotRecaller` | `recall:rt_hot`(实时)→ `recall:hot`(离线)→ DB popularity | `hot` + Flink | **永远返回**(DB 兜底层) |
| **TAG** | `TagRecaller` | 画像类目 + 实时类目 `rt:user:{id}` + query 意图类目 | `build-features` + Flink | Redis 挂 → 仅静态画像 |
| **U2U** | `UserCfRecaller` | Redis `u2u:{userId}`(相似用户正反馈物化列表) | `user-cf`(UserCF+IIF) | 空 |
| **SWING** | `SwingRecaller` | Redis `swing:{itemId}`(抗热门 i2i) | `swing` | 空 |
| **SEMANTIC** | `SemanticRecaller` | query/伪 query → embedding → pgvector ANN | 在线 embed | 客户端/embed 失败 → 空(**无熔断**) |
| **COLD** | `ColdStartRecaller` | 各类目 top-k + **UCB 探索**(`ColdStartBandit`) | `cold-bandit-stats` → `cold:cat:*` | bandit 关 → 按排序分交错 |
| **TWO_TOWER** | `TwoTowerRecaller` | user 塔 ONNX(`floorMod(userId,userBuckets)`)→ pgvector ANN `item_tower_embedding` | `import-tower`(train_two_tower) | 模型缺失 → 空 |
| **GENERATIVE** | `SemanticIdRecaller` | 用户近期 item 语义 ID `(c0,c1,c2)` → **最长公共前缀**检索同簇 | `import-semantic-id`(train_rqvae) | 表/种子缺 → 空 |
| **LEXICAL** | `LexicalRecaller` | Postgres 全文检索 `ts_rank_cd`(BM25) over `item.title_tsv` | `idf` 辅助 | **仅 query 场景**,否则空 |
| **TIGER** | `TigerRecaller` | decoder-only Transformer beam search(beam=8)自回归生成下一个 item 的语义 ID → 映射回 item | `train_tiger` → `tiger.onnx` | 模型/历史缺 → 空 |

**取种子的口径统一**:I2I/SWING/GENERATIVE 取 `action IN (CLICK,LIKE,PLAY,RATING)`(排除 IMPRESSION)、`ORDER BY item_id DESC`;TIGER 取 `ts DESC, value>=4` 再反转成 oldest→newest(自回归需时序)。

> **VECTOR vs SEMANTIC**:VECTOR 用**长期画像向量**(历史正反馈聚合),SEMANTIC 用**即时意图**(当前 query 或近期标题伪 query)。两者共用 pgvector,但语义不同。

### 广告召回(`recsys-ad`,详见 [12](12-广告-召回相关性与定向.md))
- **关键词倒排**:`bidword:inv:{keyword}` ZSet(score=bid),query 分词命中。
- **SEMANTIC_AD**:query↔`ad_embedding` 余弦 ANN。
- **U2A**:`user_embedding`→`ad_embedding` 余弦,query 无关的个性化定向(新用户无向量则跳过)。
- 兜底多路 + DB 回退(Redis 空时 `kwByDb`)。

## 3. 融合:归一化取 max vs RRF(关键设计)

`MultiChannelRecallService` 是编排核心。两个模式:

### 默认(userId 推荐)——逐路归一化后跨路取 max
```
每路: score ÷ 本路最大值 → [0,1]
跨路: 同一 item 命中多路 → 取各路归一分的 max
```
**为什么必须先归一化**:热度计数(几千)和余弦相似度(0~1)量纲天差地别。不归一化,HOT/CF 的大数值会碾压 VECTOR/SEMANTIC 的小余弦,语义召回等于白做。

### 搜索场景——RRF(Reciprocal Rank Fusion)
```
贡献 = 1 / (k + rank)   # k=60(recsys.recall.rrf-k),rank 为该 item 在本路的名次
累加多路贡献 → 再归一到 [0,1]
```
搜索请求由编排层注入 `recall-fusion=rrf`,让**词法(LEXICAL/BM25)+ 向量(SEMANTIC)混合检索**按名次融合,而非按分值——名次对不同打分尺度更鲁棒。这就是"混合检索"(hybrid retrieval)。

### 主来源(primary channel)标记
一个 item 被多路命中时,按 **最小优先级**选主来源(供推荐理由):
```
I2I < SWING < U2U < TWO_TOWER < TIGER < GENERATIVE < VECTOR < SEMANTIC < LEXICAL < TAG < COLD < HOT
```
个性化协同优先于兜底热门——否则一个被多路命中的热门片会被标成"热门"。

## 4. 工程:并行 + 超时 + 降级

- **并行执行**:`MultiChannelRecallService` 用有界线程池(`pool-size=12`),每路 `timeout-ms=150`,超时/异常即视为该路返回空。任一路挂不影响整体(HOT 的 DB 兜底层永远有结果)。
- **可启停**:`RecallContext.enabledChannels` 空=全开,非空=只跑指定路。分层 A/B(recall 层 base/plus)与冷启动覆盖 `[COLD,HOT,TAG]` 据此切换。
- **候选规模**:召回目标 `size = max(请求size × 20, 200)`,给排序留足空间。

## 5. 踩坑

1. **不归一化直接相加**——量纲不一致,热度压垮语义。本项目"逐路归一 → 取 max"就是为解决这个。
2. **新用户无种子**:无 `user_embedding`、无正反馈 → I2I/U2U/SWING/VECTOR/TWO_TOWER/GENERATIVE 全拿不到种子,必须 COLD/HOT/TAG 兜底(见 [06 冷启动](06-冷启动.md))。
3. **熔断只加在 VECTOR**:SEMANTIC/TWO_TOWER 的 pgvector 读没包熔断,pg 慢时只能靠 150ms 超时兜底(已知缺口)。
4. **LEXICAL 只在 query 场景生效**,推荐(无 query)场景返回空——别指望它给 userId 推荐兜底。
5. **语义召回依赖 embedding 服务**(Gemini 免费额度 1000/天),失败必须降级返回空、不阻塞链路。

## 6. 面试要点

- **多路召回为什么归一化再融合**:量纲对齐,防止热度淹没语义;跨路取 max 保留最强信号。
- **RRF vs 加权融合**:RRF 按名次(rank)融合,对不同打分尺度鲁棒,是混合检索(词法+向量)的标准做法。
- **学习型召回**:TWO_TOWER(DSSM 双塔,item 塔离线烘焙、user 塔在线 ONNX)、GENERATIVE(RQ-VAE 语义 ID 前缀检索)、TIGER(自回归生成 item id)——代表召回从"检索"走向"生成"。
- **在线/离线分工**:协同矩阵、相似列表、双塔 item 向量全离线物化进 Redis/pgvector,在线只做"查 + 取 TopN"。
- **降级设计**:每路独立可降级,HOT 的 DB 层是永不失败的最终兜底。
