# 实时特征与流式

> **解决什么**:离线批算的特征有延迟(天级)。用户**当下**正在互动的兴趣需要秒级反映。本项目用 Flink 消费行为流,把实时信号写进在线存储供召回/排序即时使用。

## 1. Flink 实时特征作业

`RealtimeFeatureJob`(`recsys-streaming`)运行在**本地嵌入式 MiniCluster**(无需独立 Flink 集群)。消费 Kafka `behavior-events`,processing-time 滑动窗口,产出三类实时信号:

| 输出 | Redis key | 消费方 | 语义 |
|---|---|---|---|
| 实时热门 | `recall:rt_hot` ZSet | `HotRecaller`(优先读,回落 `recall:hot`) | 滑窗内热度 |
| 实时类目偏好 | `rt:user:{id}` Hash(field=类目,value=近期计数,带 TTL) | `TagRecaller`(融合进 TAG 通道) | 用户当下兴趣类目 |
| 实时行为序列 | `rt:user:seq:{id}` ZSet | `DinRankService`/`DienRankService`(R2 序列来源) | 用户最近正反馈序列 |

## 2. TAG 通道怎么融合实时类目

`TagRecaller` 把 `rt:user:{id}` 与静态 `app_user.profile` 类目**取并集**,并按近期计数加权:
```
weight = 1 + boost · count / maxCount
```
这样用户*当下*正在互动的内容,即使 profile 尚未包含,也能在 TAG 中浮现。开关 `recsys.recall.tag.realtime-enabled`(默认开);Redis 不可用则降级为仅静态。

## 3. 实时序列喂 DIN/DIEN(R2)

- 在线 `DinRankService` 优先读 Redis `rt:user:seq:{id}`(实时序列)→ DB 回退 + cache-aside(TTL 3600,`dinSeqRedisEnabled` 默认 true)。
- 这样序列模型能看到用户**最新几次**行为,而非天级滞后的离线序列。
- 离线 `gen-samples-mt` 造 as-of 序列训练,在线 `SequenceEncoder` 编码——序列的在线/离线契约(见 [04](04-特征体系与在线离线一致性.md))。

## 4. 运行方式

- 构建 fat jar:`mvn -pl recsys-streaming -am package`。
- 运行:`bash recsys-streaming/run-streaming.sh`(脚本加了 Flink 需要的 Java 21 `--add-opens` 标志)。
- 需 `docker compose -f docker/docker-compose.yml --profile full up -d`(Kafka)+ behavior 以 `BEHAVIOR_USE_KAFKA=true` 运行。
- `behavior-events` 同时被 rec-engine 的 `SeenItemsConsumer` 消费(物化 `seen:{user}` 已看集)。

## 5. 近线 vs 实时 vs 离线(三条时间线)

| 时效 | 谁 | 例子 |
|---|---|---|
| 秒级(实时) | Flink 流 | `recall:rt_hot` / `rt:user` / `rt:user:seq` |
| 分钟~小时(近线) | 增量作业 | FTRL `train-ftrl`、bandit `bandit-stats`(`--incremental`) |
| 天级(离线) | 批作业 | CF 倒排、user_embedding、模型训练、`feat:*` |

三条线各写各的在线存储,在线链路统一"查表"消费。

## 6. 踩坑

1. **processing-time vs event-time**:本项目用 processing-time 滑窗(简单,教学态);生产乱序数据需 event-time + watermark。
2. **Flink Java 21 需 `--add-opens`**——否则 MiniCluster 起不来(脚本已加)。
3. **实时序列缺失要回退 DB**——Redis 冷启动/挂掉时 DIN 不能没序列,cache-aside 兜底。
4. **rt:user TTL**——实时类目要带 TTL,否则旧兴趣不衰减。

## 7. 面试要点

- **实时特征解决什么**:离线特征有延迟,用户当下兴趣需秒级反映(TAG 实时类目、序列模型实时序列)。
- **HotRecaller 的两级**:实时 `recall:rt_hot` 优先,回落离线 `recall:hot`,再回落 DB。
- **近线学习**:FTRL/bandit 增量更新,介于实时与离线之间。
- **一份行为流多方消费**:Flink(实时特征)+ rec-engine(已看物化),解耦。
