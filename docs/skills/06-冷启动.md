# 冷启动

> **解决什么**:新用户无行为、新物品无反馈、新系统无数据时,推荐链路的大部分召回路(依赖种子)全部失效。冷启动是**面试高频题、本项目亮点**。

## 1. 四类冷启动

| 场景 | 策略 |
|---|---|
| **新用户**(无行为) | `ColdStartDetector` 判冷 → 编排覆盖召回为 `[COLD,HOT,TAG]` + 强多样性重排(`maxSameCategory=1`) |
| **兴趣 onboarding** | `POST /api/user/{id}/interests` 写 `app_user.profile.categories` → TAG 召回立即生效 |
| **新物品**(无行为) | 靠内容向量(embedding)进入 VECTOR 召回;给曝光探索机会 |
| **新系统**(无数据) | 先灌 MovieLens 历史数据 bootstrap |

## 2. 冷启动判定(关键口径)

`ColdStartDetector`:**冷 = 正反馈数 < `recsys.cold-start.min-behaviors`(默认 5)且无 `user_embedding` 行**。
- 正反馈只数 `action IN (CLICK,LIKE,PLAY,RATING)`,**必须排除 IMPRESSION**——曝光埋点每次推荐都写 IMPRESSION,若计入,用户请求几次后行为数就假性达标、"假性脱冷",冷启动策略失效。这与 CF 取种子口径一致。
- `user_embedding` 读走 `derivedJdbc`(派生库)。查询异常 → 保守判为非冷。
- 命中时 bucketTag 打 `cold;` 前缀,便于分桶评估。

## 3. COLD 通道的 UCB 探索(本项目新增)

`ColdStartRecaller` 不只是"跨类目铺开取各类目热度 top-k",还叠加了 **UCB 多臂老虎机探索**(`ColdStartBandit`):
```
score = posRate + coef · √( ln(total + e) / (impr + 1) )
```
- `posRate` = 该类目历史正反馈率(利用),后一项 = 曝光不足的探索加成。
- 统计由离线 `cold-bandit-stats` 作业物化到 Redis `cold:cat:{c}` / `cold:cat:total`。
- bandit 关 → 退化为按排序分交错。

这让冷启动从"静态跨类目铺开"升级为"**探索-利用**动态平衡":曝光不足的类目获得更多冷启动流量,快速定位新用户兴趣。

## 4. 搜索场景的冷启动旁路

冷用户**带 query** 时:`coldOverride = cold && !(queryDriven && search.bypassColdStart)`。即搜索请求可 `bypass-cold-start`——让 query↔item 相关性主导,而非被冷启动的强多样性覆盖。冷用户带 query 也走 query 主导链路。

## 5. 踩坑

1. **IMPRESSION 计入行为数 → 假性脱冷**(见上,最重要)。
2. **onboarding 见效有限**:冷用户即使 onboarding 了类目,TAG 的偏好物品仍常被 COLD/HOT 的高热度跨类目大片在融合分上压过 → 可对"冷启动+有画像"时给 TAG 额外加权(TAG channel-boost 默认 1.5 就是缓解)。
3. **新物品曝光饥饿**:纯内容向量召回给的机会有限,需要 EE/探索兜底(广告侧有 `ExplorationService`,推荐侧靠 COLD bandit)。

## 6. 面试要点

- **冷启动判定为什么排除 IMPRESSION**:曝光≠兴趣,计入会假性脱冷。
- **COLD 通道的 EE**:UCB 平衡"利用高转化类目"与"探索曝光不足类目"。
- **三类冷启动的解法**:用户冷→兴趣引导+跨类目探索;物品冷→内容向量;系统冷→历史数据 bootstrap。
- **搜索旁路**:有 query 时相关性优先,可 bypass 冷启动强多样性。
