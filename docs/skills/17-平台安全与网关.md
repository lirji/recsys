# 平台安全与网关

> **解决什么**:多服务架构下,既要拦住外部未授权请求,又要保证服务间调用可信。本项目实现**两级零信任**(边缘 JWT + 内部 HMAC token),全部收在 `recsys-platform` 共享库,一个开关全局启停。

## 1. 两级零信任模型

```
Browser ──edge JWT(HS256)──► Gateway(边缘鉴权,终结外部 JWT)
                               │ 剥离 Authorization + 伪造的 X-Internal-Auth
                               │ 重新签发内部 HMAC token
                               ▼
                    Downstream 服务(验内部 token → SecurityContext)
```
- **边缘 JWT**:面向浏览器/外部,网关处终结。
- **内部 HMAC token**:东西向(服务间),网关重新签发,下游验签。外部**无法伪造**内部 token(不知 internal-secret),伪造的 `X-Internal-Auth` 头被网关剥离。

## 2. 内部 token(`InternalToken`)

- 纯 JDK HMAC-SHA256 自签:`base64url(subject\nroles\nscene\nexp).base64url(hmac)`。
- 篡改/过期 → 验签返回 null。
- 配置 `AuthProperties`(`recsys.security.*`):`enabled`、`internal-header`(`X-Internal-Auth`)、`internal-secret`、`edge-secret`(HS256,网关)、`token-ttl-seconds=300`、`permit-paths`。

## 3. 网关侧(`recsys-gateway`)

- `AuthController`:`/api/auth/login` 签发登录 JWT(**网关原生,不路由下游**)。
- `GatewaySecurityConfig`:校验边缘 JWT。
- `IdentityPropagationFilter`:重新签发内部 token,**剥离入站 `Authorization` + 伪造 `X-Internal-Auth`**,注入新鲜 token。

## 4. 下游 Servlet 侧(`recsys-platform`)

- `InternalSecurityConfig`:链**永远注册**。`enabled=true` → stateless + permit-paths + `authenticated()` + `InternalTokenAuthFilter`;`enabled=false` → 显式 `permitAll()`(**不会自锁死**)。
- `InternalTokenAuthFilter`:验 token → 填 SecurityContext。
- `MethodSecurityConfig`:`@PreAuthorize` 与同一开关联动(一键关)。
- `GlobalExceptionHandler` + `ApiError`:统一错误契约。
- `RECSYS_SECURITY_ENABLED` 横跨所有服务——**网关本地也要设 false**,否则 `/api/advertiser` 等 401(dev-local.sh 已处理这个坑)。

## 5. 网关路由与治理

- 反应式 Spring Cloud Gateway :8080,**端口参数化**(`REC_ENGINE_PORT:8081`、`BEHAVIOR_PORT:8082`、`ADVERTISER_PORT:8083`、`CONSOLE_PORT:8090`)。
- 路由:rec-engine 收 `/api/{recommend,search,search-ads,feed,query,user,ad,experiment}/**`;behavior/advertiser/console 各自前缀。**ad-serving 刻意不路由**(计费经 rec-engine→gRPC)。
- **discovery profile**:静态路由换 `lb://recsys-<svc>`(Nacos + LoadBalancer)。
- **ratelimit profile**(opt-in,需 Redis):`RequestRateLimiter` Redis 令牌桶,`userKeyResolver` = userId→IP→global 分桶,`GATEWAY_RATE_LIMIT_RPS:50`/`_BURST:100`,超限 429。默认关、无 Redis 也能起。

## 6. gRPC 侧安全(缺口)

- 客户端拦截器 `InternalAuthGrpcClientInterceptor` 已在 rec-engine/advertiser 接线(`GrpcClientHardeningConfig`),会签 HMAC token 进 metadata。
- **但服务端拦截器 `InternalAuthGrpcServerInterceptor` 没有任何服务注册它**(没有 `GlobalServerInterceptorConfigurer`)——客户端签、服务端不验 → 东西向 gRPC 目前靠网络信任。这是 HTTP 路径已闭环、gRPC 路径待补的半成品。

## 7. 踩坑

1. **网关不设 `RECSYS_SECURITY_ENABLED=false` 本地会 401**——网关自身也在鉴权链上。
2. **enabled=false 用显式 permitAll**——而非不注册链,避免误配自锁死。
3. **内部 token TTL 300s**——服务间调用别缓存过期 token。
4. **gRPC 服务端未验签**——生产前必须补 `GlobalServerInterceptorConfigurer` 注册服务端拦截器。

## 8. 面试要点

- **两级零信任**:边缘 JWT(外部)+ 内部 HMAC(东西向),网关重签发、剥离伪造头,外部无法伪造内部身份。
- **为什么内部 token 而非透传 JWT**:最小权限 + 短 TTL + 服务间信任边界清晰;网关是唯一签发点。
- **一个开关全局启停**:`RECSYS_SECURITY_ENABLED`,关时显式 permitAll 不自锁。
- **网关限流**:Redis 令牌桶,userId→IP→global 分级 key,opt-in profile。
- **诚实缺口**:gRPC 服务端鉴权未接线。
