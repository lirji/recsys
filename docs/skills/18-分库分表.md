# 分库分表

> **解决什么**:广告表数据量增长后单库扛不住,要水平拆分。难点是**含 pgvector 的库不能整体交给分片中间件**(向量 ANN 不可分片、非标准算子解析报错)。本项目用 ShardingSphere-JDBC + 双数据源解这个矛盾。

## 1. ShardingSphere-JDBC 分片

- datasource 走 `jdbc:shardingsphere:classpath:sharding.yaml`(`ShardingSphereDriver`),把广告表分到同容器两个 database `ds_0`(原 `recsys`)/`ds_1`(`recsys_ds1`)。
- `sharding.yaml` 放 `recsys-ad-common`(原 recsys-common)资源里,写侧(advertiser)与读侧(recsys-ad)共用。

### 分片键设计
| 表 | 分片键 | 理由 |
|---|---|---|
| `advertiser` | `advertiser_id` | 广告主作用域查询单库命中 |
| `ad` | **`advertiser_id`**(非 ad_id) | 广告与广告主同库(bindingTables 绑定表,JOIN 不跨库) |
| `bidword` | `ad_id` | 竞价词按广告分 |
| `ad_creative` | `ad_id` | 创意按广告分 |
| 其余(item/*_embedding/ad_event…) | 不分片(`!SINGLE`→ds_0) | — |

- `INLINE` 取模算法(`MOD`/`HASH_MOD` 是 auto 算法只配 autoTables)。

## 2. Snowflake 主键(为什么不用自增)

- 分库后跨库 IDENTITY 自增会**撞主键**。改用 ShardingSphere 的 **Snowflake keyGenerator**(8 字节、时间有序、去中心化)。
- insert 不带主键列,`getGeneratedKeys()` 回传(worker-id 固定 1)。
- **关键坑:别按 `ad_id` 等 Snowflake 低位取模分片**——Snowflake 低 12 位是序列号,低 QPS 下恒 0 → 全压一库(倾斜)。故 `ad` 改按 `advertiser_id` 分。

## 3. pgvector 不分片(核心矛盾)

- **`ad_embedding` 刻意不分片**:向量 ANN(hnsw + `<=>`)必须看到**全量**,分片会漏召回;且 pgvector 非标准算子不进 ShardingSphere 解析路径(会报错)。
- 含向量的库走**双数据源**:主(`@Primary` 普通 Postgres,承载召回/排序/embedding/pgvector ANN/`ad_event`,零改动)+ 次(`adShardingJdbc`,ShardingSphere)只读分片广告表。
- `AdRepository` 把"向量 ANN(主库 `ad_embedding` 不分片全量)+ 广告详情(分片库 `ad⋈advertiser` 绑定表 + `bidword` 出价分查)"**拆两步**。

## 4. 四数据源(`AdShardingConfig`)

`recsys-ad` 的 `AdShardingConfig` 定义 4 个数据源(任何扫描到它的服务——rec-engine/ad-serving/offline——都装配):

| 数据源 | Bean | 承载 |
|---|---|---|
| 主 | `@Primary` 普通 PG | 召回/排序/embedding/pgvector ANN/ad_event(零改动) |
| 分片 | `adShardingJdbc`(ShardingSphere) | advertiser/ad/bidword/ad_creative(ds_0/ds_1) |
| 广告事件库 | `adDbJdbc`(`AD_PG_DB`) | ad_event/ad_embedding/ad_servable(DDD 拆库) |
| 派生向量库 | `derivedJdbc`(`DERIVED_PG_DB`) | item_embedding/item_tower_embedding/item_semantic_id/user_embedding |

- 显式 bean 逼 Boot auto-config 让位;`@Primary` 保持共享库的无限定注入不变;拆分是**外科手术式**(只有特定 repo 切数据源)。
- 镜像:`OfflineAdShardingConfig`(offline)+ `AdvertiserDataSourceConfig`(advertiser)。

## 5. 跨分片消除 JOIN

- `report` 拆两查:广告表广播取广告 → `ad_event` 单表按 ad_id 聚合。
- `findAdRow`/`listAdRows` 把 `ad_embedding` 标志拆成单表批查。
- **ShardingSphere 裸聚合坑**:用 `MAX(x)` 而非 `COALESCE(MAX(x),0)`——否则归并失效,每分片各返一行。

## 6. 落库前置 & 全链路

- 先跑 `sql/04_ad_sharding.sql`(建 `recsys_ds1` + 分片骨架 + 拆 ds_0 跨库外键/IDENTITY)。
- **全链路已分片并实测**:写侧(advertiser)+ 在线读侧(rec-engine/ad 双数据源)+ 离线侧(offline 双数据源)。`seed-ads` 写广告表分片落 ds_0/ds_1(实测 advertiser 10/10、ad 100/100);`/api/search-ads` 召回两库广告。
- **placeholder 坑**:`$${PG_PORT::5432}` 只在 launcher 下从 env 解析(隔离的 `mvn spring-boot:run` 会踩)。`PG_JDBC_URL`/`PG_DS1_JDBC_URL` 用单占位符全 URL 规避。
- offline 多 arg 用**空格**分隔(逗号不拆分)。

## 7. 踩坑汇总

1. **Snowflake 低位取模分片 → 全压一库**(序列号低位低 QPS 恒 0)。
2. **pgvector 交给 ShardingSphere → 解析报错 + ANN 漏召回**。用双数据源。
3. **裸聚合归并失效**(`COALESCE(MAX())` 每分片各返一行)。
4. **跨库 JOIN** 要拆成"广播/单表批查"。
5. **placeholder 在隔离 spring-boot:run 不解析**。

## 8. 面试要点

- **分片键 ≠ 主键**:主键用 Snowflake(全局唯一、时间有序),分片键选业务维度(advertiser_id)。
- **绑定表(bindingTables)**:关联表同分片键 → JOIN 不跨库。
- **含向量库为什么双数据源**:pgvector ANN 须看全量、非标准算子不进解析器 → 只把分片表给 ShardingSphere。
- **Snowflake 分片陷阱**:别用低位取模,低 QPS 序列号恒 0 会倾斜。
- **DDD 物理拆库**:ad_event/ad_embedding→AD_PG_DB,派生向量→DERIVED_PG_DB,配合 4 数据源(见 [16](16-微服务拆分与gRPC.md))。
