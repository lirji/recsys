# 广告 · 竞价与计费机制

> **搜索广告的核心**:在通过定向/相关性的候选里,按"预期收入 eCPM"排序,并用**次价拍卖**计费。本项目实现多计费模式(A1)+ GSP + 可选 VCG / List-wise,并有博弈稳定性分析器。

## 1. Ad Rank(排序分怎么算)—— `BiddingService.auction`

```
pctrCalib = clamp01( calibrate(pCTR_raw) )                    # 保序回归校准
effBid    = OCPC: targetCpa·pCVR·k | OCPM: targetCpa | 手动: bid   # 有效出价
pacedBid  = effBid · pacingFactor                              # 预算平滑
quality   = 精细化质量度(缺失退基线)
billFactor= 按 BidType(见下)
ecpm      = pacedBid · billFactor
rankEcpm  = ecpm · boost                                       # EE 探索加成(只进排序)
丢弃 if rankEcpm < reserve
```

## 2. 多计费模式(A1 核心)—— `billFactor`

不同计费单位(CPC/CPM/CPA…)本来量纲不可比。用 `billFactor` 把它们**折算到同一个 eCPM 尺度**:

| BidType | billFactor | 计费单位 | 结算事件 |
|---|---|---|---|
| CPC / OCPC | `pctrCalib · quality · relevance` | 每点击 | click |
| CPM | `quality · relevance` | 每千次曝光 | impression |
| CPA / OCPM | `pctrCalib · pCVR · quality · relevance` | 每转化 | CPA=转化, OCPM=曝光 |

- `eCPM = pacedBid · billFactor` 让跨计费单位**同尺可比**。
- `charged = nextRank / billFactor` 天然按**自身单位**计价(VCG/Listwise 一并正确)。
- `BidType.Unit` 路由结算:编排层按 `SponsoredAd.bidType` 分流——CPM/OCPM 曝光扣、CPC/OCPC 点击扣、CPA 转化扣。

## 3. GSP 次价拍卖(默认)

```
按 rankEcpm 降序排
nextRank = 下一名的 rankEcpm(末位用 reserve)
charged  = nextRank / billFactor_i + ε   ,clamp [reserve, pacedBid_i]
```
- **GSP(Generalized Second Price)**:每个中标广告付的是"能保住其位置的最低价"(下一名的分)÷ 自身 billFactor,而非自己的出价。
- **自身的 EE boost 不进自身计费**(守红线);但注意 O1:GSP 的 `nextRank` 含**下一名的** boost,即邻居的探索加成会略微抬高上位者的扣费(设计取舍)。

## 4. VCG 位置拍卖(可选,`AD_VCG_ENABLED`)

`VcgAuction`:按位置点击折扣 θ_j 建模 CTR(`[1.0,0.7,0.5]`,tailDecay 0.7,单调),付**外部性**:
```
分配: 按 rankEcpm(含 boost)占位
计价: value = pacedBid · effQuality(不含 boost)
T_k   = Σ value_(j)·(θ_{j-1}−θ_j) + marginValue·θ_m       # 你占位给别人造成的损失
price_k = T_k / (θ_k · effQuality_k) + ε
```
- **VCG 激励相容**(truthful):真实出价是最优策略。开启时优先于 List-wise/GSP;单广告位退化为 GSP。

## 5. List-wise 外部性(可选,`AD_LISTWISE_ENABLED`)

`ListwiseExternality`:整页贪心,候选有效质量按与已选广告相似度衰减(MMR 思路):
```
adjRank   = rankEcpm · extFactor
extFactor = clamp(1 − λ·maxSim, minRetention 0.3, 1)  ,λ=0.5
charged   = threshold / (effQuality · extFactor) + ε    # 外部性 GSP
```
相似度由 `AdEmbeddingSimilarity`(复用 `item_embedding` 余弦)提供,缺向量退 0 → extFactor 恒 1 → 与逐条 eCPM+GSP 完全等价。避免同页广告高度雷同。

## 6. 博弈稳定性分析(`AuctionGame`,纯离线)

不改在线,离线诊断激励相容:
- `maxTruthfulRegret`:VCG≈0(真实)、GSP>0(顶位有压价获利偏离)。
- `gspSymmetricNashBids`:GSP 对称纳什均衡复现 VCG 价(EOS/Varian 等价)。
- `findCollusionArbitrage`:证 GSP 和 VCG **均不防合谋**(诚实结论)。

## 7. 工程:批量预取

`BiddingService` 把 `pacingFactors`/`refinedBatch`/`boosts`/`coefficients` 各做**一次 MGET**,零 per-candidate Redis 往返——竞价是热路径,批量预取是关键优化。

## 8. 踩坑

1. **rank/bill 分离**:排序用 rankEcpm(含 boost),计费用不含 boost 的 value/billFactor——boost 只影响谁上,不影响自己扣多少。
2. **billFactor≤0**:退 reserve,不产生负计费。
3. **VCG > Listwise > GSP 的优先级**:开关叠加时 VCG 赢;都关 → 纯 GSP。
4. **charged clamp**:必须 clamp 到 `[reserve, pacedBid]`,否则可能扣超出价或低于底价。

## 9. 面试要点

- **GSP vs VCG**:GSP 付下一名的分(非真实,顶位可压价获利);VCG 付外部性(激励相容,真实出价最优)。为什么工业界多用 GSP:简单、可解释,纳什均衡下近似 VCG。
- **多计费统一到 eCPM**:billFactor 折算,`eCPM=bid·billFactor` 可比,`charged=nextRank/billFactor` 按自身单位计价。
- **oCPC 为什么是"智能"出价**:平台按转化目标自动调出价(见 [14](14-广告-智能出价与转化建模.md))。
- **List-wise 外部性**:整页去雷同,externality GSP 计费。
- **博弈**:VCG 真实、GSP 纳什复现 VCG 价、均不防合谋。
