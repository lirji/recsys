# 推荐系统 · 系统能力

> 本文是 recsys 的**能力清单(Capability Doc)**:这套脚手架"现在能做什么、做到什么程度、边界在哪里"。
> 与 `01-技术栈` / `02-架构设计` / `03-关键技术点` 互补——前三篇讲"是什么/怎么设计",本篇讲"能力交付到哪一步"。
> 状态口径:✅ 已实现且本地实跑验证 · 🟡 已实现但有已知偏置/局限 · ⛔ 暂未实现(见 §9 路线)。

## 0. 一句话能力概述

> 给定 `userId`,在 **MovieLens(9742 部电影 / 10 万条评分)** 数据上,经
> **9 路召回 → 6 种可切换排序策略 → 3 种可插拔重排 → 已看过滤**,
> 在**分层 A/B 实验**框架下产出个性化推荐;
> 冷启动用户走专属链路;离线作业喂养在线存储;在线 Prometheus/Grafana 观测 + 离线多指标评估闭环。

## 1. 召回能力(recall)

主入口 `GET /api/recommend`,多路召回并行执行、合并去重(主来源按 **priority** 取首位,非命中顺序)。

| 通道 | 类型 | 数据来源 | 状态 | 说明 |
|---|---|---|---|---|
| `TWO_TOWER` | 学习型(DSSM 双塔) | item 塔向量烘焙进 `item_tower_embedding`(64维+HNSW),user 塔在线 ONNX 实时算 query→pgvector ANN | 🟡 | 单路 eval 全面领先(@10 Prec0.667/NDCG0.725/HitRate0.948);⚠️ item 向量含测试期训练,绝对值偏乐观 |
| `I2I`(ItemCF) | 协同过滤 | 离线 `item-cf` 写 Redis `i2i:{itemId}` ZSet | ✅ | sim 用 IUF 惩罚活跃用户 + 热门物品;5959 物品出倒排 |
| `SWING` | 协同过滤 | 离线 `swing` 写 `swing:{itemId}` | ✅ | 共现用户对算 Swing;11.2 万用户对 / 3589 物品 |
| `U2U`(UserCF) | 协同过滤 | 离线 `user-cf` 写 `u2u:{userId}` | ✅ | user-user 余弦 + IIF,每用户 TopN 相似用户汇总物品 |
| `VECTOR` | 内容向量 | `user_embedding` → pgvector 最近邻 | 🟡 | 语义召回正确(Toy Story→Jumanji/Balto);⚠️ 仅灌了 1000/9742 条向量,覆盖受限 |
| `SEMANTIC` | 内容向量 | query 或近期正反馈标题伪 query → Embedding → pgvector | ⛔/🟡 | 代码就绪、可降级;⚠️ **Gemini key 已失效(403),当前恒空降级** |
| `TAG` | 画像 + 实时 | 静态画像偏好类目 ∪ **实时类目偏好 `rt:user:{id}`**(Flink 滑窗) | ✅ | onboarding 类目立即生效;实时类目按近期计数加权排前,实时独有类目也纳入(补画像滞后) |
| `COLD` | 冷启动 | 窗口函数跨类目铺开(`ROW_NUMBER PARTITION BY category`) | ✅ | 仅冷启动链路启用 |
| `HOT` | 兜底 | 离线 `hot` 写 `recall:hot`;实时 `recall:rt_hot` 优先 | ✅ | 永远在线的最终兜底,三级降级 rt_hot→离线 hot→DB popularity |

- **优先级**(主来源标记):`I2I 0 / SWING 1 / U2U 2 / TWO_TOWER 3 / VECTOR 4 / SEMANTIC 5 / TAG 6 / COLD 7 / HOT 8`(数字小优先)。
- **多来源合并**:`RecallItem.channels` 记录某物品被哪些路召回(实测 user1 出现 `['VECTOR','HOT']` / `['I2I','HOT']` 等)。
- **按实验/冷启动启停**:`RecallContext.enabledChannels` 控制本次启用哪些路(分层 A/B 的 `base` 不含 TWO_TOWER,`plus` 含)。

## 2. 排序能力(rank)

`RankRouter` 按 `recsys.rank.strategy`(`RANK_STRATEGY` 环境变量)路由;**任何模型未就绪/返回空都优雅回退规则排序**,并打 `recsys.rank{requested,served,reason}` 埋点。

| 策略 | 模型 | 输入 | 输出 | 状态 | 离线指标 |
|---|---|---|---|---|---|
| `v1` | 规则 | feat:user/item + 交叉特征 | 单分 | ✅ | eval@10 NDCG0.297 / HitRate0.795(基线) |
| `onnx` | LightGBM→ONNX | dense[5] | CTR | ✅ | 时间切分 AUC0.6660(as-of);eval@10 NDCG0.392 |
| `deepfm` | DeepFM(PyTorch) | dense[5] + sparse[3](uid/iid/cat embedding) | CTR | ✅ | 随机切分 AUC0.82;**eval@10 NDCG 最高 0.446** |
| `mmoe` | MMoE 多目标 + ESMM | dense[5] + sparse[3] | **ctr + cvr 双头** | ✅ | valid CTR0.794/CTCVR0.817;eval@10 NDCG0.384/HitRate0.923 |
| `din` | DIN 行为序列 + MMoE 头 | dense[5] + sparse[3] + seq[20] + seq_len | **ctr + cvr 双头** | 🟡 | valid CTR0.821/**CTCVR0.830(最高)**;eval@20 HitRate**0.983**/多样性/新颖最高;⚠️在线序列含测试期评分(轻度乐观) |

**关键能力点**:
- **在线/离线特征严格一致**:稠密特征共用 `FeatureAssembler`(5 特征),稀疏 id 共用 `SparseFeatureEncoder` 的分桶+vocab,序列共用 `SequenceEncoder`(定长20/右padding) —— 杜绝特征穿越。
- **多目标可调融合**:mmoe/din 在线 `score = pCTR·(cvrBias + cvrWeight·pCVR)`,`recsys.rank.multi-task` 调参**无需重训**。
- **诚实结论**:eval 的 ground-truth(rating≥4)恰等于 deepfm 单标签,故 deepfm NDCG/精度占优是结构性的;多目标模型以此换来 HitRate@20 / 覆盖 / 多样 / 新颖更优——是权衡而非退化。AUC↑ ≠ topK NDCG↑。

## 3. 重排能力(rerank)

`RerankRouter` 按实验选 strategy,回退 diversity。

| 策略 | 能力 | 状态 |
|---|---|---|
| `diversity` | 类目打散(maxSameCategory 限制) | ✅ |
| `mmr` | MMR 最大边际相关(读 `item_embedding` 算余弦,lambda 默认0.7,无向量退化同类目=1) | ✅ |
| `none`(noop) | 不重排 | ✅ |
- 冷启动链路**强制 diversity + maxSameCategory=1**。

## 4. 实验与冷启动能力

### 4.1 分层 A/B(✅)
- 三层独立分桶:`recall × rank × rerank`,每层 `salt + variants[name/weight/params]`。
- `floorMod(hash(userId|salt),100)` 按权重选 variant——**确定性**、可正交组合(实测 8 种组合正交落桶)。
- 曝光经 `ExposureLogger` 异步写 `user_behavior`(`action=IMPRESSION` 带 `bucket=bucketTag`),离线/在线 CTR 均按此聚合。

### 4.2 冷启动(✅)
- `ColdStartDetector`:正反馈数 < `min-behaviors`(默认5)且无 `user_embedding` 判冷;⚠️**只数 CLICK/LIKE/PLAY/RATING,排除 IMPRESSION**(否则系统自写曝光会让用户假性脱冷)。
- 命中冷启动:覆盖 recall 层={COLD,HOT,TAG}、强制 diversity、bucket 前缀 `cold`。
- **兴趣 onboarding**:`POST/GET /api/user/{id}/interests` 写画像 → TAG 召回立即生效。
- ✅ **融合层 channel-boost**(`recsys.fusion.channel-boost`):TAG 默认 1.5 倍,让兴趣信号(画像 + 实时 rt:user)不被 HOT/CF 高热度在全局归一化中压过;多路命中取最大 boost。实测 user1 灌实时 Crime 偏好后,boost 关→TAG 物品 #5/2 条入 top20;boost 开(1.5)→Crime 物品升至 **#1**、TAG 物品 6 条入 top20。

## 5. 已看过滤(✅)

- `SeenItemsFilter` 在编排层(召回后/排序前)剔除用户已正反馈物品(`action IN CLICK/LIKE/PLAY/RATING`,**排除 IMPRESSION**)。
- **fail-open**:查询异常返回空集=不过滤,不拖垮主链路;过滤后召回池空则回落原始召回(宁可少量重复也不返回空)。
- 开关 `recsys.filter.seen-enabled`(默认 true)。实测:过滤关 top20 内 75% 是已看,过滤开交集为 0。

## 6. 离线作业能力(recsys-offline)

`JobRunner` 按 `--job=<name>` 分发(单 Spring Boot app,非多 main)。引导顺序:

| 作业 | 能力 | 产出 |
|---|---|---|
| `import-items` | 导入 MovieLens 物品 | `item` 表(9742) |
| `import-behavior` | 导入 ratings.csv(10万) | `user_behavior`(RATING) |
| `backfill-embedding` | 灌物品向量(Gemini,L2归一化,断点续灌) | `item_embedding` 🟡仅1000条 |
| `user-embedding` | 用户正反馈向量聚合 | `user_embedding`(598用户) |
| `item-cf` / `user-cf` / `swing` | 协同过滤批算 | Redis i2i/u2u/swing |
| `hot` | SQL 加权聚合热榜 | `recall:hot` |
| `build-features` | 物化在线特征 | Redis `feat:item`/`feat:user` |
| `gen-samples` | 单目标训练样本(**默认 as-of 无穿越**,`--leaky` 对照) | `train/samples.csv`(14.5万行) |
| `gen-samples-mt` | 多目标 + 行为序列样本 | `train/samples_mt.csv`(30.2万行,click/like 双标签) |
| `import-tower` | 双塔 item 向量入库(自建表) | `item_tower_embedding` |

## 7. 实时特征能力(recsys-streaming,🟡 本地 MiniCluster)

- Flink 1.20 嵌入式 MiniCluster(无独立集群),消费 Kafka `behavior-events`,处理时间滑动窗口产出两类特征:
  - **实时热度** ZSet `recall:rt_hot`(在线 HotRecaller 优先读,三级降级);
  - **用户实时类目偏好** Hash `rt:user:{id}`。
- 实测:8 次 CLICK→rt_hot 生效、4 次 LIKE→权重2、`rt:user` 出 Adventure/Comedy。
- ✅ **在线已消费**:`TagRecaller` 把 `rt:user:{id}` 叠加进 TAG 召回(实时类目按近期计数加权,见 §1);开关 `recsys.recall.tag.realtime-enabled`(默认 true)、强度 `realtime-boost`(默认 1.0)。Redis 不可用时优雅降级为纯静态画像。

## 8. 评估与观测能力

### 8.1 离线评估(✅,recsys-offline)
- `eval`:时间分位点切 ground-truth(ts>splitTs 且 value≥4 为测试正例),复用**线上同一** recall→rank 链路,算 **Precision/Recall/NDCG/MAP/MRR/HitRate/Coverage/Diversity/Novelty @K**。
  - 支持 `--k=10,20,50`(多档)、`--recall-only`(逐路对比)、`--rank-strategy=v1|onnx|deepfm|mmoe|din`、`--max-users`(抽样)、`--threads`(并行,实测 ≈8.7x 加速)。
  - 🟡 CF/向量/双塔用全量(含测试期)建,指标偏**乐观**(javadoc 已标注)。
- `ab-report`:按 bucket 聚合 IMPRESSION(分母)/CLICK·LIKE·PLAY(分子)→ 在线分桶 CTR(T+1 精算)。
- `run_eval_compare.sh`:一次跑 5 种排序策略同口径对比。

### 8.2 在线观测(✅,Prometheus + Grafana)
- 各服务 `/actuator/prometheus` 暴露 Micrometer 指标,`docker compose --profile obs up -d` 起观测栈(Grafana :3001)。

| 指标 | 含义 |
|---|---|
| `recsys_recommend_duration_seconds`(直方图) | 端到端延迟 P50/P95/**P99**,tag rank/cold/outcome |
| `recsys_recommend_cache_total{result}` | 结果缓存命中率 |
| `recsys_recommend_empty_total` / `_seen_cleared_total` | 空召回 / 已看清空召回池异常计数 |
| `recsys_exposure_total{recall,rank,rerank,cold}` | 分桶曝光数(CTR 分母) |
| `recsys_click_total{...}` | 分桶点击数(CTR 分子,服务端回查 `expo:` 归因,客户端不传 bucket 也能算) |
| `recsys_rank_total{requested,served,reason}` | **模型回退率** = `served=rule` 占 `requested=onnx\|deepfm`,reason 区分 not_ready/empty |

- 看板"Recsys 在线观测"含 12 面板(延迟/QPS/缓存命中/分桶CTR/模型回退率/策略命中分布/JVM)。
