# 推荐系统 · 系统能力

> 本文是 recsys 的**能力清单(Capability Doc)**:这套脚手架"现在能做什么、做到什么程度、边界在哪里"。
> 与 `01-技术栈` / `02-架构设计` / `03-关键技术点` 互补——前三篇讲"是什么/怎么设计",本篇讲"能力交付到哪一步"。
> 状态口径:✅ 已实现且本地实跑验证 · 🟡 已实现但有已知偏置/局限 · ⛔ 暂未实现(见 §9 路线)。

## 0. 一句话能力概述

> 给定 `userId`,在 **MovieLens(9742 部电影 / 10 万条评分)** 数据上,经
> **12 路召回 → 6 种可切换排序策略 → 3 种可插拔重排 → 已看过滤**,
> 在**分层 A/B 实验**框架下产出个性化推荐;
> 冷启动用户走专属链路;离线作业喂养在线存储;在线 Prometheus/Grafana 观测 + 离线多指标评估闭环。
> **另支持 query 驱动入口**(`GET /api/search`):经 Query 理解层(`recsys-query`)解析意图,驱动 SEMANTIC/TAG 召回——搜索/搜索广告的第一步(见 §11 与 `docs/05`)。

## 1. 召回能力(recall)

主入口 `GET /api/recommend`,多路召回并行执行、合并去重(主来源按 **priority** 取首位,非命中顺序)。

| 通道 | 类型 | 数据来源 | 状态 | 说明 |
|---|---|---|---|---|
| `TWO_TOWER` | 学习型(DSSM 双塔) | item 塔向量烘焙进 `item_tower_embedding`(64维+HNSW),user 塔在线 ONNX 实时算 query→pgvector ANN | 🟡 | 单路 eval 全面领先(@10 Prec0.667/NDCG0.725/HitRate0.948);⚠️ item 向量含测试期训练,绝对值偏乐观 |
| `GENERATIVE` | 学习型(RQ-VAE 语义 ID) | 离线 `train_rqvae.py` 把 item 向量量化成 codeword 元组 `(c0,c1,c2)` → `item_semantic_id`;在线按用户近期 item 的语义 ID **前缀检索同簇** | 🟡 2026-06-19 | TIGER 范式的可服务版(前缀检索替代自回归生成);需先建 `03_semantic_id.sql` + 跑 `train_rqvae`→`import-semantic-id`;缺表则空 |
| `TIGER` | 学习型(生成式) | 离线 `train_tiger.py` 训 decoder-only Transformer;在线读历史 item 语义 ID → **自回归 beam search 生成下一个 item 的语义 ID** → 映射回 item | 🟡 2026-06-22 | **完整 TIGER**(真自回归生成,非前缀检索);user 塔 ONNX 缺失则空;模型小/数据有限,生成 ID 经前缀兜底映射 |
| `I2I`(ItemCF) | 协同过滤 | 离线 `item-cf` 写 Redis `i2i:{itemId}` ZSet | ✅ | sim 用 IUF 惩罚活跃用户 + 热门物品;5959 物品出倒排 |
| `SWING` | 协同过滤 | 离线 `swing` 写 `swing:{itemId}` | ✅ | 共现用户对算 Swing;11.2 万用户对 / 3589 物品 |
| `U2U`(UserCF) | 协同过滤 | 离线 `user-cf` 写 `u2u:{userId}` | ✅ | user-user 余弦 + IIF,每用户 TopN 相似用户汇总物品 |
| `VECTOR` | 内容向量 | `user_embedding` → pgvector 最近邻 | 🟡 | 语义召回正确(Toy Story→Jumanji/Balto);⚠️ 仅灌了 1000/9742 条向量,覆盖受限 |
| `SEMANTIC` | 内容向量 | query 或近期正反馈标题伪 query → Embedding → pgvector | ⛔/🟡 | 代码就绪、可降级;⚠️ **Gemini key 已失效(403),当前恒空降级** |
| `LEXICAL` | 词法(BM25) | query → Postgres 全文检索 `ts_rank_cd`(item 标题+类目 tsvector + GIN) | ✅ 2026-06-22 | 搜索专用,与 SEMANTIC 向量路经 **RRF 混合检索**;仅 query 驱动时启用,缺列降级空。**本地实测**:"war space aliens"→Star Wars 三部曲 + Alien |
| `TAG` | 画像 + 实时 | 静态画像偏好类目 ∪ **实时类目偏好 `rt:user:{id}`**(Flink 滑窗) | ✅ | onboarding 类目立即生效;实时类目按近期计数加权排前,实时独有类目也纳入(补画像滞后) |
| `COLD` | 冷启动 | 窗口函数跨类目铺开(`ROW_NUMBER PARTITION BY category`) | ✅ | 仅冷启动链路启用 |
| `HOT` | 兜底 | 离线 `hot` 写 `recall:hot`;实时 `recall:rt_hot` 优先 | ✅ | 永远在线的最终兜底,三级降级 rt_hot→离线 hot→DB popularity |

- **优先级**(主来源标记):`I2I 0 / SWING 1 / U2U 2 / TWO_TOWER 3 / TIGER 4 / GENERATIVE 5 / VECTOR 6 / SEMANTIC 7 / LEXICAL 8 / TAG 9 / COLD 10 / HOT 11`(数字小优先)。
- **多来源合并**:`RecallItem.channels` 记录某物品被哪些路召回(实测 user1 出现 `['VECTOR','HOT']` / `['I2I','HOT']` 等)。
- **按实验/冷启动启停**:`RecallContext.enabledChannels` 控制本次启用哪些路(分层 A/B 的 `base` 不含 TWO_TOWER,`plus` 含)。

## 2. 排序能力(rank)

`RankRouter` 按 `recsys.rank.strategy`(`RANK_STRATEGY` 环境变量)路由;**任何模型未就绪/返回空都优雅回退规则排序**,并打 `recsys.rank{requested,served,reason}` 埋点。

| 策略 | 模型 | 输入 | 输出 | 状态 | 离线指标 |
|---|---|---|---|---|---|
| `v1` | 规则 | feat:user/item + 交叉特征 | 单分 | ✅ | eval@10 NDCG0.297 / HitRate0.795(基线) |
| `onnx` | LightGBM→ONNX | dense[5] | CTR | ✅ | 时间切分 AUC0.6660(as-of);eval@10 NDCG0.392 |
| `deepfm` | DeepFM(PyTorch) | dense[5] + sparse[3](uid/iid/cat embedding) | CTR | ✅ | 随机切分 AUC0.82;**eval@10 NDCG 最高 0.446** |
| `mmoe` | MMoE 多目标 + ESMM | dense[5] + sparse[3] | **ctr + cvr 双头** | ✅ | valid CTR0.794/CTCVR0.817;eval@10 NDCG0.384/HitRate0.923 |
| `din` | DIN 行为序列 + MMoE 头 | dense[5] + sparse[3] + seq[20] + seq_len | **ctr + cvr 双头** | 🟡 | valid CTR0.821/**CTCVR0.830(最高)**;eval@20 HitRate**0.983**/多样性/新颖最高;⚠️在线序列含测试期评分(轻度乐观) |

**关键能力点**:
- **在线/离线特征严格一致**:稠密特征共用 `FeatureAssembler`(5 特征),稀疏 id 共用 `SparseFeatureEncoder` 的分桶+vocab,序列共用 `SequenceEncoder`(定长20/右padding) —— 杜绝特征穿越。
- **多目标可调融合**:mmoe/din 在线 `score = pCTR·(cvrBias + cvrWeight·pCVR)`,`recsys.rank.multi-task` 调参**无需重训**。
- **诚实结论**:eval 的 ground-truth(rating≥4)恰等于 deepfm 单标签,故 deepfm NDCG/精度占优是结构性的;多目标模型以此换来 HitRate@20 / 覆盖 / 多样 / 新颖更优——是权衡而非退化。AUC↑ ≠ topK NDCG↑。

## 3. 重排能力(rerank)

`RerankRouter` 按实验选 strategy,回退 diversity。

| 策略 | 能力 | 状态 |
|---|---|---|
| `diversity` | 类目打散(maxSameCategory 限制) | ✅ |
| `mmr` | MMR 最大边际相关(读 `item_embedding` 算余弦,lambda 默认0.7,无向量退化同类目=1) | ✅ |
| `none`(noop) | 不重排 | ✅ |
- 冷启动链路**强制 diversity + maxSameCategory=1**。

## 4. 实验与冷启动能力

### 4.1 分层 A/B(✅)
- 三层独立分桶:`recall × rank × rerank`,每层 `salt + variants[name/weight/params]`。
- `floorMod(hash(userId|salt),100)` 按权重选 variant——**确定性**、可正交组合(实测 8 种组合正交落桶)。
- 曝光经 `ExposureLogger` 异步写 `user_behavior`(`action=IMPRESSION` 带 `bucket=bucketTag`),离线/在线 CTR 均按此聚合。

### 4.2 冷启动(✅)
- `ColdStartDetector`:正反馈数 < `min-behaviors`(默认5)且无 `user_embedding` 判冷;⚠️**只数 CLICK/LIKE/PLAY/RATING,排除 IMPRESSION**(否则系统自写曝光会让用户假性脱冷)。
- 命中冷启动:覆盖 recall 层={COLD,HOT,TAG}、强制 diversity、bucket 前缀 `cold`。
- **兴趣 onboarding**:`POST/GET /api/user/{id}/interests` 写画像 → TAG 召回立即生效。
- ✅ **融合层 channel-boost**(`recsys.fusion.channel-boost`):TAG 默认 1.5 倍,让兴趣信号(画像 + 实时 rt:user)不被 HOT/CF 高热度在全局归一化中压过;多路命中取最大 boost。实测 user1 灌实时 Crime 偏好后,boost 关→TAG 物品 #5/2 条入 top20;boost 开(1.5)→Crime 物品升至 **#1**、TAG 物品 6 条入 top20。

## 5. 已看过滤(✅)

- `SeenItemsFilter` 在编排层(召回后/排序前)剔除用户已正反馈物品(`action IN CLICK/LIKE/PLAY/RATING`,**排除 IMPRESSION**)。
- **fail-open**:查询异常返回空集=不过滤,不拖垮主链路;过滤后召回池空则回落原始召回(宁可少量重复也不返回空)。
- 开关 `recsys.filter.seen-enabled`(默认 true)。实测:过滤关 top20 内 75% 是已看,过滤开交集为 0。

## 7. 实时特征能力(recsys-streaming,🟡 本地 MiniCluster)

- Flink 1.20 嵌入式 MiniCluster(无独立集群),消费 Kafka `behavior-events`,处理时间滑动窗口产出两类特征:
  - **实时热度** ZSet `recall:rt_hot`(在线 HotRecaller 优先读,三级降级);
  - **用户实时类目偏好** Hash `rt:user:{id}`。
- 实测:8 次 CLICK→rt_hot 生效、4 次 LIKE→权重2、`rt:user` 出 Adventure/Comedy。
- ✅ **在线已消费**:`TagRecaller` 把 `rt:user:{id}` 叠加进 TAG 召回(实时类目按近期计数加权,见 §1);开关 `recsys.recall.tag.realtime-enabled`(默认 true)、强度 `realtime-boost`(默认 1.0)。Redis 不可用时优雅降级为纯静态画像。

## 8. 评估与观测能力

### 8.1 离线评估(✅,recsys-offline)
- `eval`:时间分位点切 ground-truth(ts>splitTs 且 value≥4 为测试正例),复用**线上同一** recall→rank 链路,算 **Precision/Recall/NDCG/MAP/MRR/HitRate/Coverage/Diversity/Novelty @K**。
  - 支持 `--k=10,20,50`(多档)、`--recall-only`(逐路对比)、`--rank-strategy=v1|onnx|deepfm|mmoe|din`、`--max-users`(抽样)、`--threads`(并行,实测 ≈8.7x 加速)。
  - 🟡 CF/向量/双塔用全量(含测试期)建,指标偏**乐观**(javadoc 已标注)。
- `ab-report`:按 bucket 聚合 IMPRESSION(分母)/CLICK·LIKE·PLAY(分子)→ 在线分桶 CTR(T+1 精算)。
- `run_eval_compare.sh`:一次跑 5 种排序策略同口径对比。

### 8.2 在线观测(✅,Prometheus + Grafana)
- 各服务 `/actuator/prometheus` 暴露 Micrometer 指标,`docker compose --profile obs up -d` 起观测栈(Grafana :3001)。

| 指标 | 含义 |
|---|---|
| `recsys_recommend_duration_seconds`(直方图) | 端到端延迟 P50/P95/**P99**,tag rank/cold/outcome |
| `recsys_recommend_cache_total{result}` | 结果缓存命中率 |
| `recsys_recommend_empty_total` / `_seen_cleared_total` | 空召回 / 已看清空召回池异常计数 |
| `recsys_exposure_total{recall,rank,rerank,cold}` | 分桶曝光数(CTR 分母) |
| `recsys_click_total{...}` | 分桶点击数(CTR 分子,服务端回查 `expo:` 归因,客户端不传 bucket 也能算) |
| `recsys_rank_total{requested,served,reason}` | **模型回退率** = `served=rule` 占 `requested=onnx\|deepfm`,reason 区分 not_ready/empty |

- 看板"Recsys 在线观测"含 12 面板(延迟/QPS/缓存命中/分桶CTR/模型回退率/策略命中分布/JVM)。

## 11. Query 理解 / 搜索入口能力(`recsys-query`,🟡)

> 搜索 / 搜索广告的第一步(M4),把推荐的「userId 驱动」补上「query 驱动」。完整扩展蓝图见 `docs/05-搜索广告扩展设计`。

- **契约**(`recsys-common.query`):`QueryUnderstandingService.parse(rawQuery, userId)` → `StructuredQuery(raw, normalized, terms, intents, rewrites, embedding)`。
- **入口**:`GET /api/search?q=&userId=&size=`(scene=search);`GET /api/recommend` 加可选 `q`;`GET /api/query/parse?q=` 单独调试。
- **能力**(`QueryUnderstandingServiceImpl`,五步各自降级):

| 步骤 | 做法 | 状态 |
|---|---|---|
| 归一化 | 小写 / 去标点 / 压空格 | ✅ |
| 分词 + 权重 | 去停用词 / 去重 / 截断;权重恒 1.0 | 🟡 权重无 IDF |
| 意图识别 | ① 词项命中 genre 名(强投票)② `item.title ILIKE ANY(terms)` 聚合 category 投票 → 归一化过阈值 TopN | ✅ 纯词法,复用 19 genre 体系 |
| 向量化 | `ObjectProvider<EmbeddingClient>` 可选 | ✅ Gemini(默认)或**本地 BGE**(`provider=local`,离线 ONNX);均缺时降级 null |
| 改写 | `recsys.query.synonyms` 同义扩展 | 🟡 MVP 最小 |
| **LLM 增强** | `ObjectProvider<LlmClient>` 可选,一次调用同时产出**拼写纠错** + **意图分类** + **改写扩展** | ✅ 2026-06-19 |

- **LLM query 理解**(`recsys.query.llm.*` + `recsys.llm.*`,✅ 2026-06-19):新增 `LlmClient` 契约(`recsys-common.llm`,与 `EmbeddingClient` 同属「外部模型客户端」家族)+ `GeminiChatClient` 实现(`recsys-embedding`,调 Gemini `generateContent`,Redis 缓存 `llm:cache:{hash}` 省额度)。在分词**前**用 LLM 做一次结构化理解(强制 JSON 输出):
  - **拼写纠错**:`corrected` 作为有效 query 替换 `normalized`(并据此分词 / 向量化),改善"horrer moviez"这类口语/错拼的语义召回;
  - **意图分类**:LLM 直接产出类目,且**严格对齐本库 genre 词表**(词表外类目丢弃),比"标题 ILIKE 投票"更鲁棒,按序赋递减分(1.0/0.9/…)优先于词法意图;
  - **改写扩展**:`expansions` 并入 `rewrites`(上限 `max-expansions`)。
  - **优雅降级**:`recsys.llm.enabled=false`(默认)/ 无 `GEMINI_API_KEY` / 调用失败 / 返回非合法 JSON → 整步跳过,回落纯词法链路,行为与接入前**完全一致**(`LlmClient.isReady()` 把关)。开关:`LLM_ENABLED=true`。

- **接入召回**:编排层有 query 时,`normalized` → `params["query"]`(SEMANTIC 读)、`intents` → `params["intentCategories"]`(TAG 按 `weight=1+intentBoost·score` 加权,`recsys.recall.tag.intent-boost` 默认 2.0),并把 SEMANTIC/TAG 并入启用召回路;query 请求跳过结果缓存。
- **混合检索(hybrid search,✅ 2026-06-22)**:搜索把**词法**(`LEXICAL`,Postgres 全文检索 BM25/`ts_rank_cd`,精确命中片名/类目词面)与**向量语义**(`SEMANTIC`,近义/语义相关)两路并跑,经 **RRF(Reciprocal Rank Fusion,`贡献=1/(k+rank)`,`recsys.recall.rrf-k` 默认 60)** 按名次融合 —— 对量纲迥异的检索器(BM25 分 vs 余弦)比分数归一化更鲁棒。query 驱动请求由编排层注入 `recall-fusion=rrf`,`MultiChannelRecallService` 据此切 RRF 合并(默认 userId 推荐仍走"按路归一化取 max",零影响)。`LEXICAL` 仅 query 场景启用、缺 `title_tsv` 列降级空;Gemini 403 时 SEMANTIC 空,词法路独力支撑搜索(实测 "war space aliens"→Star Wars 三部曲 + Alien)。
- **搜索专用融合**(`recsys.search.*`,✅ 2026-06-18):query 驱动请求覆盖默认 fusion —— 召回分(=query↔item 相关性)权重抬到 2.0、排序分(个性化 CTR)压到 0.5,`channel-boost` 抬升 `SEMANTIC: 2.0 / TAG: 1.8`,且 `bypass-cold-start` 让冷用户带 query 时也走 query 主导链路(不被冷启动的 COLD 探索 + 强多样性重排冲淡)。前提是召回分先**按路归一化**(`MultiChannelRecallService` 内每路归一化到 [0,1] 再跨路取 max,否则热度计数压垮余弦分)。
- **搜索温和个性化**(`recsys.search.personalization-weight`,默认 0.5,✅ 2026-06-19):融合分 ×(1+权重·max(0,cos(BGE user_embedding, item_embedding))),在 query 相关性主导下按用户长期口味微调次序("准 + 懂你")。实测 user 12(喜剧/动作)搜 "war movie":同样相关下 `This Means War`(亲和 0.72)被顶到 `War Horse`(0.66)之上;冷用户无 user_embedding → 自动纯相关性(优雅降级)。`PersonalizationScorer` 一次 pgvector 查询拿全候选亲和度。
- **实测**(BGE + 搜索融合后):`outer space war with aliens and spaceships` → SEMANTIC 主导出 Aliens / Battlestar Galactica / Babylon 5 / Battle: LA;`scary horror movie with ghosts` → Scary Movie / The Haunting / Ghost / Poltergeist;老用户与冷用户(999999)结果一致由 query 驱动。意图类目仍精确驱动 TAG(同一物品仅当 category ∈ query 意图时带 `TAG`)。
- **边界**:LLM 关闭时意图为词法+标题投票(非学习模型)、无拼写纠错;面向英文 MovieLens 分词;LLM 增强依赖 `GEMINI_API_KEY`(403 时自动降级纯词法);搜索个性化(用户画像融合进搜索排序)已由 `personalization-weight` 部分覆盖。

## 12. 搜索广告能力(`recsys-ad`,🟡 M4–M5)

> 把推荐的「userId 驱动」升级为「query 驱动的赞助广告」:在 `recall→rank→rerank` 之上叠加广告链路。完整设计见 `docs/05`。入口 `GET /api/search-ads?q=&userId=&slots=`。

## 13. 位置偏差去偏(PAL,排序模型,✅ 2026-06-19)

> 训练样本里"点击"受展示位次影响(同样相关的物品排得越靠前点得越多),直接拿观测点击训 pCTR 会把位置效应学进相关性里。**PAL(Position-aware Learning)** 把它拆开。

- **做法**:训练时 `pCTR = sigmoid(相关性 logit + 位置偏置 b(position))`,位置偏置塔 `b(position)` 专门吸收位置效应;**导出 ONNX 时 `position=None`,该塔不进计算图** —— 线上推理仍是原来的输入(MMoE 双输入 / DIN 四输入),`MmoeRankService`/`DinRankService`/`SequenceEncoder`/`SparseFeatureEncoder` **一律不改**。这是 PAL 相对"位置当普通特征 + 预测时置 0"的关键优势:在线零改动、零契约风险。
- **落地范围**:多目标模型 `train_mmoe.py` / `train_din.py`(CTR head 加位置偏置;CVR 不受位置影响)。LightGBM/DeepFM(单标签、走 `samples.csv`)未接,保持原样。
- **数据**:`gen-samples-mt` 多输出一列 `position`。本仓历史样本由评分派生、**无真实曝光位次**,故 `position` 默认恒 `0`(=未知,偏置塔对其掩码为 0 → PAL **休眠**,精度等同未去偏);真实系统应从曝光日志取位次。`--position-proxy` 用**热度名次**做教学代理(热门≈靠前)来观察机制(代理与热度特征相关,仅教学)。
- **契约**:`position ∈ [0, max_position]`(默认 10),`mmoe_schema.json`/`din_schema.json` 记 `max_position` + `position_debiased=true`。`position==0` 不施偏置,保证未知位次样本不被错误"去偏"。
- **边界**:无真实曝光位次时为休眠基建(机制正确、等真实点击日志接入即生效);代理仅教学;DeepFM/LGBM 未接 PAL。

## 14. 生成式召回(RQ-VAE 语义 ID,`GENERATIVE` 通道,🟡 2026-06-19)

> **TIGER**(Transformer Index for GEnerative Recommenders)范式的**可服务**落地版:用语义 ID 把"内容/行为向量"变成"由粗到细的离散码",共享前缀=语义相近,从而把召回变成"按前缀找同簇"。

- **离线**(`train/train_rqvae.py`,PyTorch):**RQ-VAE**(残差量化自编码器)读 item 向量(默认 `item_tower.csv` 的 64 维双塔向量,`--input` 可换内容 embedding),encoder→潜向量→**3 层残差量化**(每层一个 K=256 的 codebook,straight-through + commitment/codebook 损失)→ 每个 item 得语义 ID `(c0,c1,c2)`,写 `train/item_semantic_id.csv`。
- **灌库**(`import-semantic-id` 作业):CSV → `item_semantic_id(item_id,c0,c1,c2,model)`(自建表 + 前缀索引;需先建 `03_semantic_id.sql` 或由作业自建)。
- **在线**(`SemanticIdRecaller`,`GENERATIVE` 通道):取用户近期正反馈 item 作种子 → 查其语义 ID → 检索与任一种子**共享前缀**的其它 item,按**最长公共前缀深度**打分(3=`c0c1c2` 全同 > 2=`c0c1` > 1=`c0`)归一化到 (0,1]。纯 JDBC,契合本仓"离线烘焙 + 在线查表"范式。
- **两档实现**:`GENERATIVE`(本节,前缀检索)是可服务版;`TIGER`(下文,§14.1)是**完整自回归生成**。前者零在线模型、永远可用;后者真生成、更接近论文,二者可并存。
- **优雅降级**:`item_semantic_id` 表缺失 / 用户无种子 / 种子无语义 ID → 通道返回空,其它召回路兜底。默认在分层 A/B 的 `plus` recall 变体里启用。
- **边界**:用 `item_tower.csv`(行为向量)做量化,语义 ID 实为"行为相似簇"(换 `--input` 为内容 embedding 即得"内容语义簇");RQ-VAE 在含测试期的双塔向量上训练,绝对指标偏乐观(同 TWO_TOWER 注意事项)。

### 14.1 完整 TIGER 自回归生成(`TIGER` 通道,🟡 2026-06-22)

> 论文版 TIGER:不再"前缀检索",而是**用 decoder-only Transformer 读用户历史、自回归生成下一个 item 的语义 ID**。

- **离线**(`train/train_tiger.py`):把每个历史 item 映射成 3 个语义 ID 码 token(`token = base + level·K + code`),序列 = `[历史码 token… , BOS, t0, t1, t2]`;decoder-only Transformer(`d=64/2 层/4 头`)teacher-forcing + causal mask,只在 BOS/t0/t1 三位对 t0/t1/t2 算交叉熵。训练对(历史→下一个正反馈 item)取自 `samples_mt.csv` 的 `label_like=1` 行(seq_items 作历史、item_id 作目标)。导出 `tiger.onnx`(`tokens[N,T]→logits[N,T,V]`,动态 T)+ `tiger_schema.json` 放入 **recsys-recall** 资源。
- **在线**(`TigerRecaller`):用户近期正反馈 item(oldest→newest)→ 查语义 ID → 拼前缀 `[…, BOS]` → **逐级 beam search**(L=3):每级把所有 beam 批量喂 ONNX、取末位 logits、限制到该级码 token 区间、log-softmax 取 top-B → 扩展 → 全局保留 top-B → 得 B 个生成的 `(c0,c1,c2)` → 映射回 item(精确,无则按 `c0c1`/`c0` 前缀兜底)。3 级 = 3 次(批量)ONNX 调用。
- **在线/离线契约**:token 化 / `max_hist` / `L` / `K` 必须一致,由 `tiger_schema.json` 携带(在线读同一份),类比 `SequenceEncoder`。训练用 causal mask;导出 `causal=False`(只喂前缀,末位 attend 全前缀 = 等价 causal 末位),ONNX 不含 mask 算子。
- **优雅降级**:`tiger.onnx`/schema 缺失 → 通道空;无历史/无语义 ID → 空;生成的语义 ID 无对应 item → 前缀兜底;均不拖垮其它路。重训后 `mvn -pl recsys-recall clean install` 重打包。
- **本地实测闭环(2026-06-22)**:`train_rqvae`(9105 语义 ID)→`train_tiger`(47971 训练对,token 准确率 0.22→0.31,远高于随机 1/255)→ 打包重启 → `/api/recommend`(plus 桶 user 3/42):beam search 每请求生成 8~22 个候选,`recallFrom` 出现 `TIGER`(user3 ×4、user42 ×1),0 报错。**踩坑**:`nn.TransformerEncoder` 导出 ONNX 会烘焙固定 batch/seq 的 reshape(`dynamo=False` 也救不了),变长 beam 解码必崩;改为**手写多头注意力(reshape 用动态 `x.shape`)**后,onnxruntime 在 `N×T` 任意形状([1,1]/[8,50]/[50,10])均通过。
- **边界**:本仓 item 少(~9k)、RQ-VAE 码碰撞较多、Transformer 小且训练对有限,生成命中率有限(故保留前缀兜底);`GENERATIVE` 前缀版仍是更稳的兜底。这是**教学/脚手架级**的完整 TIGER,非生产调优。

## 15. 曝光日志闭环(real impression-log loop,✅ 2026-06-19)

> **根因改进**:此前训练样本由 MovieLens 评分派生 —— 负样本靠热度采样近似、无真实展示位次,导致 PAL 休眠、校准/eval 偏乐观。本节把链路闭成"**服务时记真实曝光 → 从曝光日志造样本**",一次性给排序模型喂上**真实负样本 + 真实位次**。

- **捕获半**(在线):`user_behavior` 加 `position` 列;`ExposureLogger` 落 IMPRESSION 时写真实展示位次(1 基)+ 已有的 `bucket`。线上每次 `/api/recommend` 都在积累带位次的曝光日志。
- **消费半**(离线 `gen-samples-impr`):从 IMPRESSION 日志构造与 `gen-samples-mt` **同表头/同契约**的 `samples_mt.csv`(trainers 零改动消费)——
  - **真实负样本** = 曝光未点击;**真实位次** = `position` 列直接喂 PAL;
  - 标签由**归因**得到:曝光后 `--window-hours` 内该 (user,item) 有 CLICK/PLAY/RATING → `label_click=1`,有 LIKE 或 RATING≥4 → `label_like=1`;
  - 稠密特征/序列走 as-of(把 RATING 流式并入 `AsOfFeatureBuilder`,在曝光 ts 之前 snapshot,无穿越)。
- **教学反馈模拟器**(`sim-rec-events`,仿 `sim-ad-events`):本仓评分多早于曝光、缺真实点击,故造一批**带位置偏置**的曝光+点击:`P(click)=examine(position)×relevance`,`examine=decay^(pos-1)`、`relevance` 由物品热度驱动。非真实流量,仅为让闭环 + PAL 可演示。
- **本地实测闭环(2026-06-19)**:`sim-rec-events`(8000 会话×10 位)→ 8 万曝光,CTR 按位次单调衰减 **0.246(pos1)→0.056(pos10)**(decay=0.85)→ `gen-samples-impr` 产出 **80839 样本(10554 正 / 70285 真实负)**、位次→CTR 曲线完整保留 → 重训 MMoE:**PAL 由休眠转激活(有位次样本 80839/80839)**,relevance 头 AUC 0.44→**0.56**(学到热度→点击),且**导出 ONNX 仍是 `dense+sparse` 双输入(位置塔在导出时丢弃,线上推理契约不变)**。
- **边界**:模拟器为教学(真实流量请用线上曝光日志);冷启动期曝光稀疏时样本量有限;归因窗口/口径(click=任意正反馈)可按业务细化。
